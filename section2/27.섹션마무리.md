# 27. 섹션 마무리

## 핵심 요약

- 이번 섹션에서는 목표를 추가하고 삭제할 수 있는 첫 React Native 앱을 완성했다.
- React Native에서는 `div`, `h1`, `input` 같은 HTML 요소 대신 내장 핵심 컴포넌트를 사용한다.
- UI와 커스텀 컴포넌트는 `View`, `Text`, `TextInput`, `Button`, `Pressable`, `Modal`, `Image` 같은 컴포넌트를 조합해 만든다.
- 스타일링은 CSS 파일이 아니라 JavaScript 객체와 `StyleSheet`로 작성한다.
- React Native 스타일은 CSS와 비슷하지만 CSS는 아니며, 상속이 없고 플랫폼별 차이가 생길 수 있다.
- 상태, props, 이벤트 처리 같은 React의 핵심 개념은 웹 React와 거의 동일하게 동작한다.
- 목록은 간단한 스크롤에는 `ScrollView`, 긴 동적 목록에는 성능이 더 좋은 `FlatList`를 사용한다.

## 학습 목표

- 섹션 2에서 만든 목표 관리 앱의 전체 흐름을 복습한다.
- React Native UI 구축의 핵심 재료가 내장 컴포넌트라는 점을 정리한다.
- React Native 스타일링이 CSS와 어떻게 비슷하고 다른지 이해한다.
- React의 상태 관리와 이벤트 처리 패턴이 React Native에서도 그대로 사용된다는 점을 확인한다.
- `ScrollView`와 `FlatList`의 차이를 다시 정리한다.

## 주요 내용

### 첫 React Native 앱 완성

이번 섹션에서는 강의 목표를 관리하는 간단한 앱을 만들었다. 사용자는 목표를 입력하고, 목록에 추가하고, 추가된 항목을 눌러 삭제할 수 있다.

앱 자체는 작지만 React Native 앱을 만들 때 필요한 핵심 개념을 두루 다뤘다.

```text
입력 UI 만들기
-> 상태로 입력값 관리하기
-> 목표 목록 상태 만들기
-> 목록 렌더링하기
-> 항목 삭제하기
-> 모달과 이미지로 UI 다듬기
```

### 핵심 컴포넌트로 UI 만들기

React Native에서는 브라우저 DOM이 없기 때문에 HTML 요소를 사용할 수 없다.

```jsx
// React Native에서는 사용할 수 없음
<div>
  <h1>Hello</h1>
</div>
```

대신 React Native가 제공하는 핵심 컴포넌트를 사용한다.

```jsx
<View>
  <Text>Hello</Text>
</View>
```

이번 섹션에서 자주 사용한 컴포넌트는 다음과 같다.

| 컴포넌트 | 역할 |
| --- | --- |
| `View` | 레이아웃과 컨테이너 구성 |
| `Text` | 텍스트 출력 |
| `TextInput` | 사용자 텍스트 입력 |
| `Button` | 기본 버튼 |
| `Pressable` | 터치 가능한 영역과 터치 피드백 |
| `ScrollView` | 스크롤 가능한 컨텐츠 |
| `FlatList` | 성능 최적화된 목록 |
| `Modal` | 화면 위에 뜨는 오버레이 |
| `Image` | 이미지 표시 |

커스텀 컴포넌트도 결국 이런 핵심 컴포넌트들을 조합해서 만든다.

### StyleSheet와 React Native 스타일링

React Native에는 CSS 파일이 없다. 대신 JavaScript 객체로 스타일을 작성하고, 필요하면 `StyleSheet.create()`로 스타일 객체를 만든다.

```jsx
const styles = StyleSheet.create({
  goalItem: {
    margin: 8,
    padding: 8,
    borderRadius: 6,
    backgroundColor: '#5e0acc',
  },
});
```

스타일 프로퍼티 이름은 CSS와 비슷하다. `margin`, `padding`, `backgroundColor`, `borderRadius`처럼 익숙한 이름을 많이 사용한다.

하지만 이것은 CSS가 아니라 JavaScript 객체다. 그래서 다음 차이를 기억해야 한다.

- `background-color`가 아니라 `backgroundColor`처럼 카멜 케이스를 쓴다.
- 스타일 상속이 없다.
- 부모 `View`에 `color`를 줘도 자식 `Text`의 글자색이 자동으로 바뀌지 않는다.
- React Native가 스타일 지시를 각 플랫폼의 네이티브 스타일 규칙으로 변환한다.

### 플랫폼별 차이

React Native는 하나의 코드로 iOS와 Android 앱을 만들 수 있게 해주지만, 내부적으로는 두 플랫폼의 네이티브 UI 요소를 사용한다.

그래서 같은 스타일이라도 플랫폼에 따라 결과가 달라질 수 있다. 섹션 중간에는 `Text`에 `borderRadius`를 직접 적용했을 때 Android에서는 둥근 모서리가 보이지만 iOS에서는 제대로 적용되지 않는 차이를 확인했다.

이런 경우에는 `Text`를 `View`로 감싸고, 둥근 모서리 같은 컨테이너 스타일을 `View`에 적용하는 식으로 해결할 수 있다.

### React의 핵심 개념은 그대로 사용

React Native 앱도 결국 React 앱이다. 그래서 상태와 props, 이벤트 처리 방식은 웹 React와 매우 비슷하다.

이번 섹션에서 사용한 React 개념은 다음과 같다.

- `useState`로 입력값과 목표 목록 관리
- props로 부모에서 자식에게 데이터 전달
- 함수 prop으로 자식 이벤트를 부모 상태 변경에 연결
- 배열 상태를 업데이트할 때 이전 상태 기반 업데이트 함수 사용
- 컴포넌트를 `GoalInput`, `GoalItem`으로 분리

다만 모바일 앱에서는 클릭이라는 표현보다 누르기나 터치에 가까우므로, 이벤트 이름이 `onClick`이 아니라 `onPress`다.

```jsx
<Button title="Add Goal" onPress={addGoalHandler} />
```

### 목록 렌더링

처음에는 목표 목록을 직접 `map()`으로 렌더링하고, 스크롤을 위해 `ScrollView`를 사용했다.

`ScrollView`는 내용 전체를 한 번에 렌더링한다. 기사처럼 길이가 어느 정도 정해진 컨텐츠에는 적합하지만, 항목이 아주 많아질 수 있는 동적 목록에는 성능 문제가 생길 수 있다.

그래서 목표 목록은 `FlatList`로 변경했다.

```jsx
<FlatList
  data={courseGoals}
  renderItem={(itemData) => {
    return <GoalItem text={itemData.item.text} />;
  }}
  keyExtractor={(item) => item.id}
/>
```

`FlatList`는 화면에 필요한 항목 중심으로 렌더링하므로 긴 목록에 더 적합하다.

### React Native는 다른 플랫폼을 위한 React 앱

이번 섹션의 결론은 React Native가 완전히 낯선 새 세계라기보다, 네이티브 모바일 플랫폼을 대상으로 하는 React 앱이라는 점이다.

상태 관리, 이벤트 처리, 컴포넌트 분리 방식은 익숙한 React와 같다. 다만 UI를 만들 때 HTML 대신 React Native 핵심 컴포넌트를 사용하고, CSS 대신 React Native 스타일 객체를 사용한다.

이 차이에 익숙해지는 데에는 연습이 필요하다. 앞으로 더 많은 앱을 만들면서 핵심 컴포넌트, 스타일링, 플랫폼 차이에 점점 더 자연스럽게 익숙해질 수 있다.

## 정리

이번 섹션에서는 React Native의 기본기를 목표 관리 앱 하나에 압축해서 연습했다. 핵심 컴포넌트로 UI를 만들고, `StyleSheet`로 스타일링하고, 상태와 이벤트로 앱을 동작하게 만들었다.

또한 `ScrollView`와 `FlatList`, `Pressable`, `Modal`, `Image`, `StatusBar`까지 사용하면서 실제 앱에서 자주 마주치는 구성 요소를 경험했다. 앞으로의 강의에서는 이 기본기를 바탕으로 더 복잡한 앱과 React Native 기능을 다루게 된다.
