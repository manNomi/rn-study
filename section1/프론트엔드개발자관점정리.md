# 섹션 1. 프론트엔드 개발자 관점 정리

## 한 줄 요약

React Native는 React의 컴포넌트, 상태, JSX 사고방식을 그대로 사용하되, 렌더링 대상이 브라우저 DOM이 아니라 iOS와 Android의 네이티브 UI라는 점을 이해하는 섹션이다.

## 이미 알고 있다고 보는 것

- React 컴포넌트와 JSX
- props와 state
- `useState` 같은 기본 Hook
- 브라우저 기반 React 앱의 렌더링 흐름
- npm, npx, package.json, 의존성 설치
- 개발 서버를 띄우고 변경 사항을 확인하는 워크플로
- 번들러가 소스 코드를 개발 환경에 맞게 처리한다는 개념

## 새로 잡아야 하는 관점

웹 프론트엔드에서는 React가 만든 결과물을 `react-dom`이 브라우저 DOM에 반영한다.

```text
React + react-dom
-> 브라우저 DOM
-> 웹 앱
```

React Native에서는 React가 만든 결과물을 React Native가 iOS와 Android의 네이티브 UI로 연결한다.

```text
React + React Native
-> iOS / Android 네이티브 UI
-> 모바일 앱
```

따라서 React Native를 배운다는 것은 React를 처음부터 다시 배우는 일이 아니다. React는 그대로 가져가되, 브라우저가 아닌 모바일 네이티브 플랫폼에서 UI를 만들고 실행하는 방식을 익히는 일에 가깝다.

## 핵심 개념

### React Native의 역할

React Native는 React 코드로 모바일 앱을 만들 수 있게 해주는 프레임워크다. JSX로 작성한 React Native 컴포넌트는 각 플랫폼의 네이티브 UI 요소로 변환된다.

예를 들어 웹에서는 `<input>`이 브라우저의 입력 요소가 되지만, React Native에서는 `<TextInput>`이 Android의 입력 위젯이나 iOS의 입력 UI로 연결된다.

### JavaScript 로직은 그대로 실행된다

React Native가 모든 코드를 네이티브 코드로 컴파일하는 것은 아니다.

컴포넌트와 UI 표현은 네이티브 요소로 연결되지만, 상태 관리나 조건문 같은 JavaScript 로직은 앱 안의 JavaScript 실행 환경에서 동작한다.

프론트엔드 개발자 입장에서는 다음처럼 이해하면 편하다.

```text
UI 컴포넌트
-> 네이티브 UI로 연결

JavaScript 로직
-> 앱 내부 JavaScript 런타임에서 실행
```

### Expo를 사용하는 이유

Expo는 React Native 앱을 더 쉽게 만들고 실행하게 해주는 도구와 생태계다.

React Native CLI만 사용할 수도 있지만, Expo를 쓰면 다음 작업이 쉬워진다.

- 프로젝트 생성
- 개발 서버 실행
- 실제 기기에서 미리보기
- Android Emulator / iOS Simulator 실행
- 카메라, 위치, 알림 같은 네이티브 기능 접근
- 앱 설정 관리

프론트엔드 개발자가 Vite나 Next.js CLI로 프로젝트를 빠르게 시작하는 것과 비슷하게, Expo는 React Native 개발의 시작 비용을 낮춰준다.

## 개발 환경 흐름

### 프로젝트 생성

강의에서는 Expo 기반 프로젝트를 만든다.

```bash
npx create-expo-app --template blank
```

`--template blank`를 사용하면 TypeScript나 추가 구조가 붙은 복잡한 템플릿 대신, 가장 기본적인 JavaScript 프로젝트에서 시작할 수 있다.

### 프로젝트 구조

Expo 프로젝트에서 자주 보게 되는 파일과 폴더는 다음과 같다.

| 항목 | 역할 |
| --- | --- |
| `App.js` | 앱의 루트 컴포넌트 |
| `app.json` | Expo 앱 설정 파일 |
| `assets` | 이미지, 아이콘 같은 정적 자산 |
| `package.json` | 의존성, 실행 스크립트 |
| `node_modules` | 설치된 패키지 |

웹 React 프로젝트와 비교하면 `App.js`가 진입점 역할을 하고, `app.json`이 모바일 앱 설정을 담당한다는 점이 새롭다.

### 앱 실행과 미리보기

Expo 프로젝트는 다음 명령어로 개발 서버를 실행한다.

```bash
npm start
```

웹에서는 보통 브라우저에서 `localhost`를 열어 확인하지만, React Native에서는 다음 방식으로 확인한다.

- 실제 iPhone 또는 Android 기기에서 Expo Go로 QR 코드 스캔
- Android Studio의 Android Emulator 사용
- macOS + Xcode의 iOS Simulator 사용

즉 프론트엔드 개발자가 새로 익혀야 하는 것은 코드 작성보다도 "브라우저 대신 기기나 시뮬레이터에서 확인하는 흐름"이다.

## 프론트엔드 개발자가 특히 기억할 차이

### 브라우저가 기본 실행 환경이 아니다

React Native 앱은 브라우저에서 돌아가는 웹 앱이 아니다. DOM도 없고 CSS도 없으며, 모바일 플랫폼의 네이티브 UI 위에서 동작한다.

### react-dom이 없다

웹에서는 `react-dom`이 렌더링 대상이다. React Native에서는 이 역할을 React Native가 맡는다.

```text
웹 React: React + react-dom
앱 React: React + react-native
```

### 개발 확인 방식이 다르다

웹 프론트엔드에서는 브라우저 DevTools와 localhost가 중심이다. React Native에서는 Expo Go, Emulator, Simulator가 중심이 된다.

### iOS 테스트에는 macOS가 필요하다

iOS Simulator는 Xcode가 필요하고, Xcode는 macOS에서만 사용할 수 있다. Windows나 Linux에서는 Android 테스트가 중심이 된다.

## 섹션 1에서 얻어야 할 결론

React Native는 React를 완전히 대체하는 다른 기술이 아니라, React를 모바일 네이티브 앱에 적용하는 렌더링 환경이다.

프론트엔드 개발자가 이 섹션에서 가져가야 할 핵심은 다음이다.

```text
React 지식은 그대로 쓴다.
하지만 DOM, CSS, 브라우저 중심 사고는 내려놓아야 한다.
Expo를 통해 모바일 앱 개발 환경을 열고,
실제 기기나 시뮬레이터에서 앱을 확인하는 흐름에 익숙해져야 한다.
```
