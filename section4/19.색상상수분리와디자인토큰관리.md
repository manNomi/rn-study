# 19. 색상 상수 분리와 디자인 토큰 관리

## 핵심 요약

- 앱 여러 곳에서 같은 색상을 반복해서 하드 코딩하면 유지보수가 어려워진다.
- React Native에는 CSS 변수나 CSS custom properties가 없기 때문에 JavaScript 상수 파일로 공통 값을 관리한다.
- `constants/colors.js` 같은 파일을 만들고 색상 값을 객체로 모아두면 앱 전역에서 재사용할 수 있다.
- `primary500`, `primary600`, `primary800`, `accent500`처럼 색상 이름과 단계 값을 함께 쓰면 색상 체계를 이해하기 쉽다.
- 컴포넌트에서는 하드 코딩된 헥스 코드 대신 `Colors.primary500`처럼 상수를 참조한다.
- 색상 변경이 필요할 때 한 파일만 수정하면 관련 컴포넌트 전체에 반영된다.

## 학습 목표

- React Native에서 공통 색상 값을 관리하는 방법을 익힌다.
- CSS 변수 대신 JavaScript 상수 객체를 사용하는 이유를 이해한다.
- 색상 이름을 의미와 단계로 나누어 관리하는 패턴을 배운다.
- `PrimaryButton`, `StartGameScreen`, `Title`, `App`에서 색상 상수를 재사용하는 흐름을 이해한다.
- 프론트엔드 개발에서 디자인 토큰을 분리하는 감각을 React Native에도 적용한다.

## 주요 내용

### 하드 코딩된 색상의 문제

이전 강의까지 여러 컴포넌트에서 같은 색상 코드를 직접 입력했다.

```jsx
backgroundColor: '#4e0329'
```

```jsx
color: '#ddb52f'
```

작은 앱에서는 큰 문제가 없어 보일 수 있지만, 앱이 커지면 같은 색상이 여러 파일에 흩어진다.

```text
PrimaryButton
-> 버튼 배경색
-> 버튼 ripple 색상

StartGameScreen
-> 입력 영역 배경색
-> 입력 필드 강조색

Title
-> 제목 텍스트 색상
-> 제목 경계선 색상

App
-> LinearGradient 색상
```

이 상태에서 브랜드 색상이나 테마 색상을 바꾸려면 관련 파일을 모두 찾아 수정해야 한다.

```text
문제점
-> 같은 색상 코드가 여러 파일에 중복됨
-> 색상 변경 시 수정 범위가 넓어짐
-> 비슷한 색상 코드가 섞이면 어떤 색이 기준인지 알기 어려움
```

웹에서 CSS 변수를 사용하는 이유와 같은 문제다.

### React Native에는 CSS 변수가 없다

웹에서는 보통 CSS 변수로 공통 색상을 관리한다.

```css
:root {
  --color-primary-500: #4e0329;
  --color-accent-500: #ddb52f;
}
```

그리고 필요한 곳에서 참조한다.

```css
.button {
  background-color: var(--color-primary-500);
}
```

하지만 React Native는 CSS 파일을 사용하지 않는다.

따라서 CSS 변수도 사용할 수 없다.

대신 일반 JavaScript 모듈을 이용해 공통 값을 관리한다.

```text
웹
-> CSS variables
-> design tokens in CSS

React Native
-> JavaScript constants
-> design tokens as JS objects
```

### constants 폴더 만들기

공통으로 재사용할 값을 모으기 위해 `constants` 폴더를 만든다.

```text
constants/
  colors.js
```

`constants`라는 이름은 필수는 아니다.

```text
가능한 이름
-> constants
-> util
-> utils
-> theme
-> tokens
```

이번 강의에서는 재사용할 상수 값을 관리하는 목적이므로 `constants` 폴더를 사용한다.

### colors.js 파일 만들기

`colors.js`에는 앱에서 사용할 색상 객체를 정의한다.

```jsx
const Colors = {
  primary500: '#72063c',
};

export default Colors;
```

이 객체는 특별한 React Native API가 아니라 일반 JavaScript 객체다.

```text
Colors
-> 앱 전역에서 참조할 색상 모음
```

기본값으로 export하면 다른 파일에서 쉽게 import할 수 있다.

```jsx
import Colors from '../constants/colors';
```

### 색상 이름에 단계 붙이기

색상 이름을 `primary`, `accent`처럼 의미 중심으로 붙이면 이해하기 쉽다.

하지만 같은 계열의 색상도 여러 단계가 필요할 수 있다.

```text
primary500
-> 기본 primary 색상

primary600
-> 조금 더 진한 primary 색상

primary700
-> 더 어두운 primary 색상

primary800
-> 매우 어두운 primary 색상

accent500
-> 기본 강조 색상
```

숫자는 반드시 `500`, `600`이어야 하는 것은 아니다.

다만 간격을 두고 숫자를 붙이면 나중에 중간 색상을 추가하기 좋다.

```text
primary500
primary600
primary700
primary800
```

이런 방식은 Tailwind CSS나 여러 디자인 시스템에서 색상 단계로 자주 사용하는 패턴과 비슷하다.

### Colors 객체 예시

이번 앱에서 사용할 색상 상수는 다음처럼 구성할 수 있다.

```jsx
const Colors = {
  primary500: '#72063c',
  primary600: '#640233',
  primary700: '#4e0329',
  primary800: '#3b021f',
  accent500: '#ddb52f',
};

export default Colors;
```

여기서 `primary` 계열은 자주색 계열이고, `accent500`은 노란색 강조 색상이다.

```text
primary
-> 앱의 주요 배경과 버튼에 사용

accent
-> 제목, 입력 밑줄, 강조 요소에 사용
```

색상을 의미 단위로 묶으면 컴포넌트 입장에서 어떤 색을 사용해야 하는지 더 명확해진다.

### PrimaryButton에서 색상 상수 사용하기

기존 `PrimaryButton`에는 배경색과 Android ripple 색상이 하드 코딩되어 있었다.

```jsx
android_ripple={{ color: '#640233' }}
```

```jsx
buttonInnerContainer: {
  backgroundColor: '#72063c',
}
```

이제 `Colors`를 import한다.

```jsx
import Colors from '../constants/colors';
```

그리고 하드 코딩된 값을 상수로 바꾼다.

```jsx
<Pressable
  android_ripple={{ color: Colors.primary600 }}
  style={({ pressed }) =>
    pressed
      ? [styles.buttonInnerContainer, styles.pressed]
      : styles.buttonInnerContainer
  }
>
  <Text style={styles.buttonText}>{children}</Text>
</Pressable>
```

```jsx
const styles = StyleSheet.create({
  buttonInnerContainer: {
    backgroundColor: Colors.primary500,
    paddingVertical: 8,
    paddingHorizontal: 16,
    elevation: 2,
  },
});
```

이제 버튼 색상은 `colors.js`의 값을 따른다.

### StartGameScreen에서 색상 상수 사용하기

`StartGameScreen`에서도 입력 영역 배경색과 입력 필드 강조색을 상수로 바꿀 수 있다.

```jsx
import Colors from '../constants/colors';
```

기존 하드 코딩 값을 다음처럼 교체한다.

```jsx
inputContainer: {
  backgroundColor: Colors.primary800,
}
```

입력 필드의 아래 경계선과 글자 색상은 강조 색상으로 맞춘다.

```jsx
numberInput: {
  borderBottomColor: Colors.accent500,
  color: Colors.accent500,
}
```

이렇게 하면 `StartGameScreen`은 실제 헥스 코드를 몰라도 된다.

```text
StartGameScreen
-> Colors.primary800 사용
-> Colors.accent500 사용
```

컴포넌트는 색상의 실제 값보다 의미에 의존하게 된다.

### Title 컴포넌트에서 색상 상수 사용하기

`Title` 컴포넌트에서도 같은 노란색을 사용한다.

기존에는 텍스트 색상과 경계선 색상을 직접 지정했다.

```jsx
color: '#ddb52f',
borderColor: '#ddb52f',
```

이제 `Colors`를 import한다.

```jsx
import Colors from '../constants/colors';
```

그리고 상수로 교체한다.

```jsx
const styles = StyleSheet.create({
  title: {
    fontSize: 24,
    fontWeight: 'bold',
    color: Colors.accent500,
    textAlign: 'center',
    borderWidth: 2,
    borderColor: Colors.accent500,
    padding: 12,
  },
});
```

이제 `StartGameScreen`과 `Title`은 같은 강조 색상 토큰을 공유한다.

### App.js에서 색상 상수 사용하기

`App.js`의 `LinearGradient`에도 하드 코딩된 색상이 있었다.

```jsx
<LinearGradient
  colors={['#4e0329', '#ddb52f']}
  style={styles.rootScreen}
>
```

여기도 `Colors`를 import해서 교체한다.

```jsx
import Colors from './constants/colors';
```

```jsx
<LinearGradient
  colors={[Colors.primary700, Colors.accent500]}
  style={styles.rootScreen}
>
```

앱 전체 배경 그러데이션도 이제 색상 상수에 의존한다.

```text
Colors.primary700
-> 어두운 자주색 배경

Colors.accent500
-> 노란색 강조 배경
```

### 변경 후 화면은 그대로다

이번 변경은 UI를 새로 꾸미는 작업이 아니다.

기존에 보이던 화면과 거의 동일하게 보여야 정상이다.

```text
변경 전
-> 헥스 코드를 각 파일에 직접 작성

변경 후
-> 같은 헥스 코드를 Colors 객체에서 참조
```

화면의 모양은 유지하면서 코드 구조만 좋아지는 리팩터링이다.

```text
사용자 경험
-> 동일

개발자 경험
-> 좋아짐
```

### 색상 변경이 쉬워진다

이제 버튼, 제목, 입력 필드, 배경에서 쓰이는 색상을 바꾸려면 `colors.js`만 수정하면 된다.

```jsx
const Colors = {
  primary500: '#72063c',
  primary600: '#640233',
  primary700: '#4e0329',
  primary800: '#3b021f',
  accent500: '#ddb52f',
};
```

예를 들어 `accent500`을 바꾸면 이를 참조하는 모든 곳이 함께 바뀐다.

```text
Title 텍스트 색상
Title 경계선 색상
TextInput 아래 경계선
TextInput 글자 색상
LinearGradient 두 번째 색상
```

이런 구조는 앱이 커질수록 더 중요해진다.

### 프론트엔드 개발자 관점에서의 핵심 차이

프론트엔드 웹 개발자라면 이 흐름은 디자인 토큰을 분리하는 작업으로 이해하면 된다.

웹에서는 보통 다음과 같이 관리한다.

```css
:root {
  --color-primary-500: #72063c;
  --color-primary-600: #640233;
  --color-accent-500: #ddb52f;
}
```

React Native에서는 CSS가 없으므로 JavaScript 객체가 그 역할을 한다.

```jsx
const Colors = {
  primary500: '#72063c',
  primary600: '#640233',
  accent500: '#ddb52f',
};
```

개념적으로는 같다.

```text
하드 코딩 제거
-> 공통 토큰으로 추출

스타일 일관성 확보
-> 여러 컴포넌트에서 같은 값 참조

테마 변경 비용 감소
-> 한 파일만 수정
```

다만 React Native에서는 스타일이 JavaScript 객체이기 때문에, 색상 토큰도 JavaScript 모듈로 관리하는 것이 자연스럽다.

### 폴더 이름에 대한 감각

강의에서는 `constants` 폴더를 사용했다.

```text
constants/colors.js
```

하지만 실제 프로젝트에서는 팀 컨벤션에 따라 다른 이름을 사용할 수도 있다.

```text
theme/colors.js
shared/constants/colors.js
tokens/colors.js
styles/colors.js
```

중요한 것은 폴더 이름 자체가 아니라, 여러 곳에서 재사용되는 값을 한 곳에 모아두는 구조다.

```text
반복되는 값
-> 상수화

디자인 의미가 있는 값
-> 토큰화

여러 컴포넌트에서 공유
-> 공통 모듈로 분리
```

## 정리

이번 강의에서는 앱 곳곳에 하드 코딩되어 있던 색상 값을 `constants/colors.js` 파일로 분리했다. React Native에는 CSS 변수가 없기 때문에, 일반 JavaScript 객체를 사용해 앱 전역에서 재사용할 색상 값을 관리한다.

`Colors.primary500`, `Colors.primary600`, `Colors.primary800`, `Colors.accent500`처럼 의미와 단계가 드러나는 이름을 사용하면 컴포넌트가 색상의 실제 값이 아니라 역할에 의존하게 된다. 그 결과 색상 변경은 한 파일에서 처리할 수 있고, 앱 전체 스타일의 일관성도 유지하기 쉬워진다.

프론트엔드 개발자 관점에서는 CSS 변수나 디자인 토큰을 JavaScript 모듈로 옮긴 것이라고 이해하면 된다.
