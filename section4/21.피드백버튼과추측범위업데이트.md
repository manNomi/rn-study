# 21. 피드백 버튼과 추측 범위 업데이트

## 핵심 요약

- 게임 화면에는 앱의 추측값이 정답보다 높은지 낮은지 알려주는 피드백 버튼이 필요하다.
- `PrimaryButton`을 재사용해 `+`, `-` 버튼을 만들고, `nextGuessHandler`에 방향 정보를 전달한다.
- 다음 추측값은 기존 추측값보다 크거나 작아야 하므로 가능한 숫자 범위를 동적으로 좁혀야 한다.
- 범위를 관리하기 위해 `minBoundary`, `maxBoundary` 값을 사용한다.
- 잘못된 힌트를 주면 가능한 범위가 모순되어 난수 생성 함수가 무한 재귀에 빠질 수 있다.
- `Alert`를 사용해 사용자의 거짓 힌트를 막고, 잘못된 입력일 때는 새 추측값을 계산하지 않는다.
- `GameScreen`에서 정답과 비교하려면 `App.js`에서 `userNumber`를 prop으로 전달해야 한다.

## 학습 목표

- `PrimaryButton`을 게임 화면에서 재사용해 피드백 버튼을 만드는 방법을 익힌다.
- `bind`를 사용해 이벤트 핸들러에 미리 인수를 전달하는 방식을 이해한다.
- 높은 숫자와 낮은 숫자 피드백에 따라 난수 생성 범위를 좁히는 로직을 배운다.
- 컴포넌트 바깥 변수와 상태의 차이를 이해한다.
- 잘못된 사용자 피드백이 왜 무한 루프를 만들 수 있는지 파악한다.
- `Alert`로 잘못된 게임 진행을 막는 방법을 익힌다.

## 주요 내용

### 피드백 버튼이 필요한 이유

현재 `GameScreen`은 앱이 추측한 숫자를 보여준다.

다음 단계는 사용자가 이 추측값에 대해 피드백을 주는 것이다.

```text
앱의 추측값이 너무 낮다
-> 더 큰 숫자를 추측해야 함
-> + 버튼

앱의 추측값이 너무 높다
-> 더 작은 숫자를 추측해야 함
-> - 버튼
```

이 피드백을 기반으로 앱은 다음 추측값을 계산한다.

### PrimaryButton 재사용하기

이미 만든 `PrimaryButton`을 게임 화면에서도 사용할 수 있다.

```jsx
import PrimaryButton from '../components/ui/PrimaryButton';
```

처음에는 아이콘 대신 `+`, `-` 텍스트를 버튼 내용으로 사용한다.

```jsx
<PrimaryButton>+</PrimaryButton>
<PrimaryButton>-</PrimaryButton>
```

나중에는 이 텍스트를 실제 아이콘으로 바꿀 수 있다.

이번 단계에서는 동작을 먼저 구현한다.

### 버튼을 가로로 배치하기

`+`, `-` 버튼은 텍스트 아래에 나란히 놓는 것이 자연스럽다.

이를 위해 두 버튼을 `View`로 감싼다.

```jsx
<View>
  <Text>Higher or lower?</Text>
  <View>
    <PrimaryButton>+</PrimaryButton>
    <PrimaryButton>-</PrimaryButton>
  </View>
</View>
```

바깥 `View`는 안내 텍스트와 버튼 묶음을 세로로 배치한다.

안쪽 `View`는 두 버튼을 가로로 배치하는 컨테이너가 된다.

```text
바깥 View
-> Higher or lower?
-> 버튼 묶음

안쪽 View
-> + 버튼
-> - 버튼
```

이후 스타일에서 `flexDirection: 'row'`를 적용하면 두 버튼이 옆으로 놓인다.

### nextGuessHandler 만들기

버튼을 누르면 다음 추측값을 계산해야 한다.

이때 버튼마다 거의 같은 로직을 사용하므로, 함수 하나에 방향 정보만 넘겨 처리한다.

```jsx
function nextGuessHandler(direction) {
  // direction: 'lower' 또는 'greater'
}
```

`direction` 값은 두 가지다.

```text
lower
-> 현재 추측값보다 더 작은 숫자를 생성

greater
-> 현재 추측값보다 더 큰 숫자를 생성
```

하나의 함수로 처리하면 중복 코드를 줄일 수 있다.

### onPress에 인수 전달하기

`PrimaryButton`의 `onPress`에는 함수 포인터를 전달한다.

하지만 `nextGuessHandler`는 `direction` 인수가 필요하다.

이럴 때 `bind`를 사용해 나중에 호출될 함수의 인수를 미리 지정할 수 있다.

```jsx
<PrimaryButton onPress={nextGuessHandler.bind(this, 'greater')}>
  +
</PrimaryButton>

<PrimaryButton onPress={nextGuessHandler.bind(this, 'lower')}>
  -
</PrimaryButton>
```

`bind`의 첫 번째 인수는 함수 내부의 `this`를 정한다.

여기서는 `this`를 사용하지 않기 때문에 중요하지 않다.

두 번째 인수부터는 실제 함수가 호출될 때 전달될 매개변수가 된다.

```text
nextGuessHandler.bind(this, 'greater')
-> 버튼이 눌리면 nextGuessHandler('greater') 실행

nextGuessHandler.bind(this, 'lower')
-> 버튼이 눌리면 nextGuessHandler('lower') 실행
```

### 추측 범위는 동적으로 바뀐다

처음 가능한 범위는 1부터 99까지다.

```text
초기 범위
-> minBoundary = 1
-> maxBoundary = 100
```

하지만 사용자의 피드백에 따라 이 범위는 계속 좁아진다.

예를 들어 현재 추측값이 60이고 사용자가 “더 낮다”라고 알려주면, 정답은 60보다 작다.

```text
현재 추측값: 60
피드백: lower

새 최대 경계
-> 60
```

반대로 현재 추측값이 30이고 사용자가 “더 높다”라고 알려주면, 정답은 30보다 크다.

```text
현재 추측값: 30
피드백: greater

새 최소 경계
-> 31
```

이렇게 최소 경계와 최대 경계를 계속 업데이트해야 한다.

### 컴포넌트 바깥에서 경계값 관리하기

경계값은 컴포넌트가 다시 렌더링되어도 유지되어야 한다.

강의에서는 컴포넌트 함수 바깥에 변수를 둔다.

```jsx
let minBoundary = 1;
let maxBoundary = 100;
```

이 변수들은 `GameScreen` 함수가 다시 실행되어도 초기화되지 않는다.

```text
컴포넌트 내부 지역 변수
-> 렌더링마다 다시 계산될 수 있음

컴포넌트 외부 변수
-> 렌더링과 무관하게 값 유지
```

다만 실제 앱에서는 게임을 새로 시작할 때 이 값들을 다시 초기화해야 한다는 점도 염두에 두어야 한다.

이번 단계에서는 다음 추측 로직을 구현하는 데 집중한다.

### lower 피드백 처리

사용자가 `-` 버튼을 누르면 현재 추측값이 너무 크다는 의미다.

그러면 다음 추측값은 현재 추측값보다 작아야 한다.

```jsx
if (direction === 'lower') {
  maxBoundary = currentGuess;
}
```

`generateRandomBetween` 함수에서 `max`는 상한 경계처럼 사용된다.

따라서 `maxBoundary`를 `currentGuess`로 두면 현재 추측값보다 작은 범위에서 난수가 생성된다.

```text
currentGuess = 60
maxBoundary = 60

생성 가능
-> 1부터 59 사이
```

같은 숫자를 다시 추측하지 않도록 `currentGuess`를 exclude 값으로도 넘긴다.

### greater 피드백 처리

사용자가 `+` 버튼을 누르면 현재 추측값이 너무 낮다는 의미다.

그러면 다음 추측값은 현재 추측값보다 커야 한다.

```jsx
else {
  minBoundary = currentGuess + 1;
}
```

최소 경계는 생성 가능한 값에 포함되기 때문에 `currentGuess + 1`로 설정한다.

```text
currentGuess = 30
minBoundary = 31

생성 가능
-> 31부터 maxBoundary - 1 사이
```

이렇게 하면 앱이 같은 숫자를 다시 추측하지 않고 더 큰 범위에서 새 숫자를 만든다.

### 새 추측값 계산하기

경계값을 업데이트한 뒤에는 새 난수를 계산한다.

```jsx
const newRndNumber = generateRandomBetween(
  minBoundary,
  maxBoundary,
  currentGuess
);
```

그 다음 상태를 업데이트한다.

```jsx
setCurrentGuess(newRndNumber);
```

상태가 바뀌면 `GameScreen`이 다시 렌더링되고, `NumberContainer`에 표시되는 숫자도 바뀐다.

```text
버튼 클릭
-> direction 전달
-> min/max boundary 업데이트
-> 새 난수 생성
-> currentGuess 상태 업데이트
-> 화면 숫자 변경
```

### nextGuessHandler 예시

핵심 로직은 다음처럼 정리할 수 있다.

```jsx
function nextGuessHandler(direction) {
  if (direction === 'lower') {
    maxBoundary = currentGuess;
  } else {
    minBoundary = currentGuess + 1;
  }

  const newRndNumber = generateRandomBetween(
    minBoundary,
    maxBoundary,
    currentGuess
  );

  setCurrentGuess(newRndNumber);
}
```

이제 `+`를 누르면 더 큰 숫자가, `-`를 누르면 더 작은 숫자가 추측된다.

하지만 아직 큰 문제가 남아 있다.

### 잘못된 힌트가 만들 수 있는 문제

사용자가 정직하게 힌트를 주면 게임은 잘 동작한다.

하지만 사용자가 틀린 힌트를 주면 범위가 모순될 수 있다.

예를 들어 정답이 31인데 앱이 60을 추측했다고 하자.

정답은 60보다 작으므로 사용자는 `-`를 눌러야 한다.

그런데 사용자가 `+`를 누르면 앱은 정답이 60보다 크다고 믿게 된다.

```text
실제 정답: 31
현재 추측: 60
사용자 피드백: greater

앱이 믿는 범위
-> 61 이상

문제
-> 실제 정답 31은 범위 밖으로 밀려남
```

이런 잘못된 힌트가 반복되면 최소 경계와 최대 경계가 같아지거나 역전될 수 있다.

그 결과 `generateRandomBetween`이 유효한 숫자를 찾지 못하고 계속 자기 자신을 호출한다.

```text
minBoundary = 48
maxBoundary = 48

가능한 숫자 없음
-> exclude와 충돌
-> 재귀 반복
-> 최대 호출 스택 초과
```

이것이 앱이 멈추는 원인이다.

### console.log로 범위 확인하기

문제가 발생했을 때는 경계값을 로그로 확인할 수 있다.

```jsx
console.log(minBoundary, maxBoundary);
```

버튼을 누를 때마다 현재 최소/최대 경계를 확인하면, 어느 순간 범위가 너무 좁아지거나 모순되는 것을 볼 수 있다.

```text
정상 범위
-> 1, 60
-> 31, 60
-> 31, 45

문제 범위
-> 48, 48
```

디버깅할 때는 이렇게 상태나 외부 변수 값을 출력해 흐름을 추적하는 것이 유용하다.

### 잘못된 피드백 막기

무한 루프를 막으려면 사용자가 거짓 힌트를 주는 상황을 먼저 차단해야 한다.

현재 추측값과 실제 정답인 `userNumber`를 비교한다.

```jsx
if (
  (direction === 'lower' && currentGuess < userNumber) ||
  (direction === 'greater' && currentGuess > userNumber)
) {
  // 잘못된 힌트
}
```

첫 번째 조건은 다음 상황을 잡는다.

```text
currentGuess < userNumber
-> 현재 추측값은 이미 정답보다 작음

direction === 'lower'
-> 그런데 더 작은 값을 추측하라고 함

결론
-> 거짓 힌트
```

두 번째 조건은 반대 상황이다.

```text
currentGuess > userNumber
-> 현재 추측값은 이미 정답보다 큼

direction === 'greater'
-> 그런데 더 큰 값을 추측하라고 함

결론
-> 거짓 힌트
```

이 조건에 걸리면 새 난수를 계산하지 않고 함수를 종료한다.

### Alert로 사용자에게 알려주기

잘못된 힌트를 조용히 무시하면 사용자는 무슨 일이 일어났는지 알기 어렵다.

따라서 `Alert`를 사용해 경고를 띄운다.

```jsx
import { Alert } from 'react-native';
```

```jsx
Alert.alert(
  "Don't lie!",
  'You know that this is wrong...',
  [{ text: 'Sorry!', style: 'cancel' }]
);

return;
```

경고를 띄운 뒤 `return`으로 함수를 종료한다.

그래야 아래의 난수 생성 로직이 실행되지 않는다.

```text
잘못된 힌트
-> Alert 표시
-> return
-> min/max boundary 변경 안 함
-> 새 추측값 생성 안 함
```

### userNumber 전달을 잊지 않아야 한다

`nextGuessHandler`에서 `currentGuess`와 정답을 비교하려면 `GameScreen`이 `userNumber`를 알아야 한다.

이를 위해 `App.js`에서 `GameScreen`에 prop을 전달해야 한다.

```jsx
if (userNumber) {
  screen = <GameScreen userNumber={userNumber} />;
}
```

그리고 `GameScreen`에서는 객체 구조 분해로 받는다.

```jsx
function GameScreen({ userNumber }) {
  // userNumber 사용
}
```

이 prop 전달을 빠뜨리면 `userNumber`가 `undefined`가 되어 비교 로직이 제대로 동작하지 않는다.

### 최종 nextGuessHandler 흐름

전체 흐름은 다음처럼 정리할 수 있다.

```jsx
function nextGuessHandler(direction) {
  if (
    (direction === 'lower' && currentGuess < userNumber) ||
    (direction === 'greater' && currentGuess > userNumber)
  ) {
    Alert.alert(
      "Don't lie!",
      'You know that this is wrong...',
      [{ text: 'Sorry!', style: 'cancel' }]
    );
    return;
  }

  if (direction === 'lower') {
    maxBoundary = currentGuess;
  } else {
    minBoundary = currentGuess + 1;
  }

  const newRndNumber = generateRandomBetween(
    minBoundary,
    maxBoundary,
    currentGuess
  );

  setCurrentGuess(newRndNumber);
}
```

이제 사용자가 올바른 피드백을 주면 다음 추측값이 계산되고, 잘못된 피드백을 주면 경고가 표시된다.

### 프론트엔드 개발자 관점에서의 핵심 차이

이번 강의의 대부분은 React Native 전용이라기보다 React 상태와 이벤트 처리의 일반적인 응용이다.

```text
버튼 이벤트
-> onPress로 처리

이벤트 핸들러에 인수 전달
-> bind 또는 래퍼 함수 사용

상태 업데이트
-> setCurrentGuess

부모 데이터 전달
-> props
```

웹 React라면 다음처럼 작성할 수 있다.

```jsx
<button onClick={() => nextGuessHandler('greater')}>+</button>
```

React Native에서는 `PrimaryButton`과 `onPress`를 사용한다.

```jsx
<PrimaryButton onPress={nextGuessHandler.bind(this, 'greater')}>
  +
</PrimaryButton>
```

핵심 개념은 같다.

다만 React Native에서는 사용자에게 경고를 띄울 때 브라우저 `alert()`보다 `Alert.alert()` 같은 네이티브 API를 사용하는 것이 자연스럽다.

```text
웹
-> window.alert()
-> custom modal

React Native
-> Alert.alert()
-> 플랫폼 기본 경고창
```

## 정리

이번 강의에서는 `+`, `-` 피드백 버튼을 추가하고, 사용자의 힌트에 따라 앱의 다음 추측값을 계산하는 로직을 구현했다. `direction` 값을 기준으로 `minBoundary`와 `maxBoundary`를 좁히고, 그 범위 안에서 새 난수를 만들어 `currentGuess` 상태를 업데이트한다.

또한 잘못된 힌트가 들어오면 가능한 숫자 범위가 모순되어 `generateRandomBetween`이 무한 재귀에 빠질 수 있다는 문제를 확인했다. 이를 막기 위해 `currentGuess`와 `userNumber`를 비교하고, 잘못된 피드백이면 `Alert.alert()`로 경고를 띄운 뒤 함수를 종료한다.

마지막으로 `GameScreen`이 정답을 비교할 수 있도록 `App.js`에서 `userNumber`를 prop으로 전달해야 한다는 점도 중요하다.
