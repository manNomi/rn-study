# 16. 게임 화면 기본 구조와 SafeAreaView

## 핵심 요약

- `GameScreen`은 앱이 사용자의 숫자를 추측하는 실제 게임 진행 화면이다.
- 게임 화면에는 상대방의 추측 숫자, 더 높은지 낮은지 알려주는 컨트롤, 라운드 기록 영역이 필요하다.
- 여러 UI 영역을 묶기 위해 `View`를 사용하고, 화면에 텍스트를 표시할 때는 반드시 `Text`를 사용한다.
- `View` 안에 문자열을 직접 넣으면 React Native에서 오류가 발생한다.
- 화면 전체를 차지하도록 최상위 컨테이너에 `flex: 1`을 적용한다.
- 기기 상단 노치와 상태 표시줄 영역을 피하려면 `SafeAreaView`를 사용할 수 있다.

## 학습 목표

- `GameScreen`의 기본 UI 영역을 설계한다.
- 게임 진행 화면에 필요한 요소를 화면 구조로 분리하는 방법을 익힌다.
- `View`와 `Text`의 역할 차이를 다시 확인한다.
- 플레이스홀더 텍스트를 임시로 둘 때 React Native의 raw text 제한을 주의한다.
- `StyleSheet`로 게임 화면 전체 컨테이너 스타일을 설정한다.
- `SafeAreaView`가 필요한 이유를 이해한다.

## 주요 내용

### 게임 화면에서 필요한 것

이전 강의에서는 유효한 숫자를 입력하면 `StartGameScreen`에서 `GameScreen`으로 전환되도록 만들었다.

이제 `GameScreen`을 실제 게임 화면으로 채워야 한다.

이 화면의 핵심 동작은 다음과 같다.

```text
앱이 숫자를 추측한다.
사용자는 추측한 숫자가 너무 높은지 낮은지 알려준다.
앱은 피드백을 바탕으로 새로운 숫자를 다시 추측한다.
모든 추측 라운드를 기록한다.
```

따라서 게임 화면에는 다음 UI 영역이 필요하다.

```text
제목 영역
-> Opponent's Guess

추측 숫자 표시 영역
-> 앱이 추측한 숫자 출력

피드백 컨트롤 영역
-> Higher or lower?
-> + 버튼
-> - 버튼

라운드 기록 영역
-> 지금까지의 추측값 목록
```

이번 강의에서는 이 구조의 뼈대를 먼저 만든다.

### GameScreen에 View와 Text 가져오기

여러 UI 요소를 그룹화하려면 `View`가 필요하다.

화면에 텍스트를 보여주려면 `Text`가 필요하다.

```jsx
import { Text, View } from 'react-native';
```

React Native에서는 `div`, `h1`, `p` 같은 HTML 태그를 사용할 수 없다.

대신 `View`로 레이아웃 컨테이너를 만들고, `Text`로 문자열을 렌더링한다.

### 기본 화면 구조 만들기

`GameScreen`의 기본 구조는 다음처럼 시작할 수 있다.

```jsx
function GameScreen() {
  return (
    <View>
      <Text>Opponent's Guess</Text>

      {/* GUESS */}

      <View>
        <Text>Higher or lower?</Text>
        {/* + */}
        {/* - */}
      </View>

      <View>
        {/* 라운드 기록 */}
      </View>
    </View>
  );
}
```

여기서 `GUESS`, `+`, `-`, 라운드 기록은 아직 실제 컴포넌트가 아니라 앞으로 채울 자리다.

처음에는 화면을 빠르게 설계하기 위해 플레이스홀더를 둘 수 있다.

하지만 React Native에서는 `View` 안에 일반 문자열을 그대로 두면 안 된다.

### View 안에 raw text를 넣으면 안 된다

React Native에서는 다음 코드가 오류를 일으킨다.

```jsx
<View>
  GUESS
</View>
```

`View`는 레이아웃 컨테이너이고, 문자열을 직접 렌더링하는 컴포넌트가 아니다.

문자열을 화면에 표시하려면 반드시 `Text`로 감싸야 한다.

```jsx
<View>
  <Text>GUESS</Text>
</View>
```

하지만 이번 단계에서 `GUESS`는 실제로 보여줄 텍스트가 아니라 자리 표시일 뿐이다.

그럴 때는 JSX 주석으로 남겨둘 수 있다.

```jsx
{/* GUESS */}
```

이렇게 하면 화면에는 아무것도 렌더링하지 않으면서 나중에 채워야 할 위치를 표시할 수 있다.

### 피드백 컨트롤 영역

앱이 추측한 숫자에 대해 사용자는 더 큰 숫자인지 더 작은 숫자인지 알려줘야 한다.

이를 위한 영역도 별도의 `View`로 묶는다.

```jsx
<View>
  <Text>Higher or lower?</Text>
  {/* + */}
  {/* - */}
</View>
```

나중에는 `+`와 `-` 플레이스홀더를 실제 버튼으로 바꿀 것이다.

이 버튼들은 앱에게 다음 정보를 준다.

```text
+
-> 앱이 추측한 숫자가 너무 낮다.
-> 더 높은 숫자를 추측해야 한다.

-
-> 앱이 추측한 숫자가 너무 높다.
-> 더 낮은 숫자를 추측해야 한다.
```

이번 단계에서는 아직 버튼 로직을 만들지 않고, UI 영역만 잡아둔다.

### 라운드 기록 영역

게임이 진행되면 앱이 여러 번 숫자를 추측하게 된다.

각 추측을 기록하면 사용자는 게임 진행 상황을 볼 수 있다.

이를 위해 별도의 라운드 기록 영역을 만든다.

```jsx
<View>
  {/* 라운드 기록 */}
</View>
```

나중에는 이 영역에 추측값 목록을 렌더링한다.

추측이 많아질 수 있으므로, 추후에는 목록과 스크롤 처리도 필요해질 수 있다.

### 화면 전체 컨테이너 스타일 만들기

현재 구조만 만들면 콘텐츠가 화면 왼쪽 위에 몰려 보일 수 있다.

화면 전체를 감싸는 최상위 `View`에 스타일을 적용한다.

```jsx
import { StyleSheet, Text, View } from 'react-native';
```

```jsx
const styles = StyleSheet.create({
  screen: {
    flex: 1,
    padding: 12,
  },
});
```

`flex: 1`은 이 화면 컨테이너가 사용 가능한 전체 공간을 차지하게 한다.

`padding: 12`는 화면 가장자리와 콘텐츠 사이에 기본 간격을 준다.

```text
flex: 1
-> 화면 전체 공간 사용

padding: 12
-> 모든 방향에 안쪽 여백 추가
```

이 스타일을 최상위 `View`에 적용한다.

```jsx
function GameScreen() {
  return (
    <View style={styles.screen}>
      <Text>Opponent's Guess</Text>

      {/* GUESS */}

      <View>
        <Text>Higher or lower?</Text>
        {/* + */}
        {/* - */}
      </View>

      <View>
        {/* 라운드 기록 */}
      </View>
    </View>
  );
}
```

### padding만으로는 충분하지 않을 수 있다

`padding: 12`를 적용해도 콘텐츠가 여전히 상단 모서리에 너무 가까워 보일 수 있다.

값을 `40`처럼 크게 올리면 텍스트가 안쪽으로 이동하는 것을 확인할 수 있다.

```jsx
screen: {
  flex: 1,
  padding: 40,
}
```

하지만 단순히 큰 padding을 주는 방식은 기기마다 안정적인 해결책이 아닐 수 있다.

특히 iPhone의 노치, 상태 표시줄, Android 상단 영역처럼 기기별 안전 영역을 고려해야 한다.

```text
문제
-> 콘텐츠가 상단 노치나 상태 표시줄에 너무 가까움

임시 해결
-> paddingTop 또는 marginTop 크게 주기

더 나은 방향
-> SafeAreaView 사용
```

### SafeAreaView가 필요한 이유

모바일 기기는 화면의 실제 사용 가능 영역이 기기마다 다르다.

예를 들어 다음 요소들이 콘텐츠와 겹칠 수 있다.

```text
상태 표시줄
노치
카메라 영역
둥근 모서리
홈 인디케이터 주변 영역
```

이런 영역을 피해서 콘텐츠를 배치할 때 `SafeAreaView`를 사용할 수 있다.

`SafeAreaView`는 화면의 안전 영역 안에 콘텐츠가 들어오도록 도와주는 컴포넌트다.

```jsx
import { SafeAreaView } from 'react-native';
```

강의는 이 지점에서 `SafeAreaView`를 소개하며, 다음 단계에서 이를 활용해 게임 화면의 상단 간격 문제를 더 적절하게 다루게 된다.

### 현재 단계의 GameScreen 예시

이번 단계에서 만들 수 있는 `GameScreen`의 형태는 다음과 같다.

```jsx
import { StyleSheet, Text, View } from 'react-native';

function GameScreen() {
  return (
    <View style={styles.screen}>
      <Text>Opponent's Guess</Text>

      {/* GUESS */}

      <View>
        <Text>Higher or lower?</Text>
        {/* + */}
        {/* - */}
      </View>

      <View>
        {/* 라운드 기록 */}
      </View>
    </View>
  );
}

export default GameScreen;

const styles = StyleSheet.create({
  screen: {
    flex: 1,
    padding: 12,
  },
});
```

아직 실제 게임 로직은 없다.

하지만 화면의 큰 영역은 잡혔다.

```text
제목
추측 숫자 자리
높다/낮다 피드백 컨트롤 자리
라운드 기록 자리
```

### 프론트엔드 개발자 관점에서의 핵심 차이

웹에서는 화면 상단 간격을 보통 CSS로 처리한다.

```css
.screen {
  padding: 40px 12px 12px;
}
```

React Native에서도 `padding`을 사용할 수 있지만, 모바일 기기의 노치와 상태 표시줄을 고려해야 한다.

```text
웹
-> 브라우저 viewport와 CSS 박스 모델 중심

React Native
-> 기기별 안전 영역 고려 필요
-> SafeAreaView 활용 가능
```

또 하나의 차이는 텍스트 렌더링이다.

웹에서는 `div` 안에 문자열을 직접 넣을 수 있다.

```html
<div>GUESS</div>
```

React Native에서는 문자열은 반드시 `Text` 안에 있어야 한다.

```jsx
<Text>GUESS</Text>
```

플레이스홀더로만 남길 내용이라면 JSX 주석을 사용하는 편이 안전하다.

```jsx
{/* GUESS */}
```

## 정리

이번 강의에서는 `GameScreen`의 기본 UI 구조를 만들었다. 화면에는 상대방의 추측 제목, 추측 숫자 표시 자리, 높다/낮다 피드백 컨트롤 영역, 라운드 기록 영역이 필요하며, 이를 여러 `View`로 그룹화했다.

또한 `View` 안에 일반 문자열을 직접 넣으면 오류가 발생하므로, 실제 텍스트는 `Text`로 감싸고 임시 자리 표시는 JSX 주석으로 처리해야 한다는 점을 다시 확인했다. 마지막으로 화면 상단 노치와 상태 표시줄 같은 기기 안전 영역을 다루기 위해 `SafeAreaView`가 필요하다는 흐름까지 살펴봤다.
