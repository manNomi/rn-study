# 33. FlatList 컨테이너와 라운드 수 전달

## 핵심 요약

- `FlatList`의 마지막 항목이 화면 하단에 걸리는 문제는 부모 컨테이너의 높이 제약을 명확히 주어 해결한다.
- `FlatList`는 자체적으로 스크롤을 제공하지만, 어느 영역 안에서 스크롤해야 하는지는 부모 레이아웃이 결정한다.
- `FlatList`를 감싸는 `View`에 `flex: 1`을 주면 남은 공간 안에서 목록이 안정적으로 스크롤된다.
- `padding`을 추가하면 목록 항목이 화면 경계에 너무 붙지 않아 가독성이 좋아진다.
- 게임이 끝났을 때 `guessRounds.length`를 `onGameOver`에 전달하면 총 라운드 수를 `App` 컴포넌트에 보고할 수 있다.
- `App`은 전달받은 라운드 수를 상태로 저장하고, `GameOverScreen`에 `roundsNumber` prop으로 넘긴다.
- 이로써 게임 오버 화면에서 장치가 몇 라운드 만에 숫자를 맞혔는지 표시할 수 있다.

## 학습 목표

- `FlatList`의 스크롤 영역을 부모 컨테이너로 제어하는 방법을 이해한다.
- `flex: 1`이 목록 화면에서 왜 중요한지 복습한다.
- 게임 진행 화면의 내부 상태를 상위 컴포넌트에 보고하는 패턴을 익힌다.
- `onGameOver(guessRounds.length)`처럼 이벤트 콜백에 데이터를 함께 전달하는 방식을 배운다.
- `GameScreen`, `App`, `GameOverScreen` 사이의 데이터 흐름을 정리한다.

## 주요 내용

### FlatList 하단 항목이 잘리는 문제

이전 단계에서 `GuessLogItem`을 도입하면서 로그 항목이 보기 좋아졌다.

하지만 라운드가 많아져 목록이 길어지면 마지막 항목이 화면 아래쪽에 걸려서 보기 어려운 문제가 생긴다.

```text
현상
-> 스크롤은 가능함
-> 하지만 마지막 항목이 화면 하단 경계에 붙거나 일부만 보임
-> 로그를 읽기 불편함
```

문제의 핵심은 `FlatList`가 어느 높이 안에서 스크롤해야 하는지 부모 레이아웃이 충분히 제한하지 않았다는 점이다.

`FlatList`는 스크롤 가능한 목록 컴포넌트이지만, 모바일 화면에서 안정적으로 동작하려면 주변 컨테이너가 공간을 명확히 잡아줘야 한다.

### 부모 컨테이너로 높이 제한하기

해결 방법은 `FlatList`를 감싸는 `View`에 스타일을 추가하는 것이다.

```jsx
<View style={styles.listContainer}>
  <FlatList
    data={guessRounds}
    keyExtractor={(item) => item.toString()}
    renderItem={(itemData) => (
      <GuessLogItem
        roundNumber={guessRoundsListLength - itemData.index}
        guess={itemData.item}
      />
    )}
  />
</View>
```

그리고 `listContainer` 스타일을 정의한다.

```jsx
const styles = StyleSheet.create({
  listContainer: {
    flex: 1,
    padding: 16,
  },
});
```

여기서 핵심은 `flex: 1`이다.

```text
flex: 1
-> 부모 안에서 가능한 남은 공간을 차지
-> FlatList가 그 공간 안에서 스크롤
-> 화면 바깥으로 무한히 밀려나는 문제를 줄임
```

`padding: 16`은 목록이 화면 경계에 너무 붙지 않도록 여백을 준다.

### 왜 FlatList 주변 View가 필요한가

`FlatList` 자체에 스타일을 줄 수도 있지만, 이 상황에서는 주변 컨테이너가 목록의 영역을 제어하는 역할을 한다.

```text
GameScreen 전체
-> Title
-> NumberContainer
-> Card
-> listContainer
   -> FlatList
```

위 구조에서 `Title`, `NumberContainer`, `Card`가 이미 일정 공간을 차지한다.

그 아래 남은 공간을 `listContainer`가 가져가고, `FlatList`는 그 안에서 스크롤한다.

이렇게 하면 목록이 길어져도 화면 전체 레이아웃이 무너지지 않는다.

### 웹 프론트엔드와의 차이

웹에서는 보통 문서 흐름상 콘텐츠가 길어지면 브라우저가 페이지 전체 스크롤을 제공한다.

React Native에서는 이런 동작을 기대하면 안 된다.

```text
웹
-> body/document가 자연스럽게 스크롤될 수 있음

React Native
-> 스크롤이 필요한 영역을 FlatList나 ScrollView로 명시
-> 그 영역의 높이는 부모 레이아웃으로 제어
```

즉 React Native에서는 스크롤 컴포넌트와 레이아웃 컨테이너가 함께 설계되어야 한다.

### 총 라운드 수를 GameOverScreen에 표시하기

이제 게임 오버 화면에 실제 라운드 수를 표시해야 한다.

이미 `GameOverScreen`은 `roundsNumber` prop을 받도록 만들어져 있다.

```jsx
function GameOverScreen({ roundsNumber, userNumber, onStartNewGame }) {
  // ...
}
```

`App` 컴포넌트도 라운드 수를 저장하는 상태를 가지고 있다.

```jsx
const [guessRounds, setGuessRounds] = useState(0);
```

여기서 `guessRounds`는 `GameScreen` 안의 배열 상태와 이름이 비슷하지만 의미가 다르다.

```text
GameScreen의 guessRounds
-> 각 라운드의 추측값 배열
-> 예: [31, 45, 63]

App의 guessRounds
-> 게임 종료 후 총 라운드 수
-> 예: 6
```

이름이 같으면 헷갈릴 수 있으므로 실무에서는 `roundsCount`, `totalRounds` 같은 이름도 고려할 만하다.

강의 흐름에서는 기존 이름을 그대로 사용한다.

### gameOverHandler가 라운드 수를 받도록 변경

`App` 컴포넌트에는 게임 종료 시 호출되는 `gameOverHandler`가 있다.

기존에는 게임 종료 상태만 바꿨다.

```jsx
function gameOverHandler() {
  setGameIsOver(true);
}
```

이제 라운드 수를 매개변수로 받아 상태에 저장한다.

```jsx
function gameOverHandler(numberOfRounds) {
  setGameIsOver(true);
  setGuessRounds(numberOfRounds);
}
```

이 함수는 `GameScreen`에 `onGameOver` prop으로 전달된다.

```jsx
<GameScreen
  userNumber={userNumber}
  onGameOver={gameOverHandler}
/>
```

### GameScreen에서 guessRounds.length 전달하기

`GameScreen`은 실제 라운드 로그 배열을 가지고 있다.

```jsx
const [guessRounds, setGuessRounds] = useState([initialGuess]);
```

게임이 끝났을 때 총 라운드 수는 이 배열의 길이다.

```jsx
guessRounds.length
```

따라서 `useEffect`에서 장치의 추측값이 사용자 숫자와 같아졌을 때 `onGameOver`를 호출하면서 라운드 수를 함께 전달한다.

```jsx
useEffect(() => {
  if (currentGuess === userNumber) {
    onGameOver(guessRounds.length);
  }
}, [currentGuess, userNumber, onGameOver, guessRounds]);
```

강의 흐름의 핵심은 다음과 같다.

```text
GameScreen
-> guessRounds.length 계산
-> onGameOver(라운드 수) 호출

App
-> gameOverHandler(numberOfRounds)
-> setGuessRounds(numberOfRounds)

GameOverScreen
-> roundsNumber prop으로 표시
```

### GameOverScreen으로 라운드 수 넘기기

`App` 컴포넌트는 게임이 끝났을 때 `GameOverScreen`을 렌더링한다.

```jsx
if (gameIsOver && userNumber) {
  screen = (
    <GameOverScreen
      roundsNumber={guessRounds}
      userNumber={userNumber}
      onStartNewGame={startNewGameHandler}
    />
  );
}
```

이제 `GameOverScreen`의 문장 안에 실제 라운드 수가 표시된다.

```jsx
<Text style={styles.summaryText}>
  Your phone needed{' '}
  <Text style={styles.highlight}>{roundsNumber}</Text>
  {' '}rounds to guess the number{' '}
  <Text style={styles.highlight}>{userNumber}</Text>.
</Text>
```

이렇게 하면 게임이 6번 만에 끝났다면 `6 rounds`로 표시되고, 8번 만에 끝났다면 `8 rounds`로 표시된다.

### 전체 데이터 흐름

이번 단계의 흐름은 다음처럼 정리할 수 있다.

```text
1. GameScreen에서 장치가 추측할 때마다 guessRounds 배열 업데이트

2. currentGuess === userNumber가 되면 게임 종료

3. GameScreen이 onGameOver(guessRounds.length) 호출

4. App의 gameOverHandler가 numberOfRounds를 받음

5. App이 총 라운드 수를 상태에 저장

6. App이 GameOverScreen에 roundsNumber로 전달

7. GameOverScreen이 최종 결과 문장에 표시
```

React 관점에서 보면 하위 컴포넌트가 상위 컴포넌트의 상태를 직접 수정하지 않고, 콜백 prop을 호출해 필요한 데이터를 보고하는 구조다.

```text
하위 -> 상위 데이터 전달
-> props로 받은 함수 호출
-> 함수 인수로 데이터 전달
```

웹 React에서도 폼 입력값을 부모에게 전달하거나, 카드 클릭 이벤트를 상위 리스트에 알릴 때 자주 쓰는 패턴이다.

## 실무 관점에서의 주의점

### 이름 충돌 줄이기

강의에서는 `GameScreen`의 라운드 배열과 `App`의 라운드 수 상태가 모두 `guessRounds`라는 이름으로 등장한다.

작은 예제에서는 괜찮지만, 실무에서는 다음처럼 의도를 더 드러내는 이름이 좋다.

```jsx
// GameScreen
const [guessRounds, setGuessRounds] = useState([initialGuess]);

// App
const [roundsCount, setRoundsCount] = useState(0);
```

또는:

```jsx
const [totalRounds, setTotalRounds] = useState(0);
```

상태 이름이 데이터의 형태를 드러내면 디버깅이 훨씬 쉬워진다.

### useEffect 의존성

`useEffect` 안에서 `guessRounds.length`를 사용한다면 `guessRounds`를 의존성에 포함하는 것이 자연스럽다.

```jsx
useEffect(() => {
  if (currentGuess === userNumber) {
    onGameOver(guessRounds.length);
  }
}, [currentGuess, userNumber, onGameOver, guessRounds]);
```

다만 배열 상태가 변경될 때마다 effect가 재실행될 수 있으므로, 실제 앱에서는 effect의 역할과 의존성을 조심해서 관리해야 한다.

이 예제에서는 게임 종료 조건 확인이 목적이므로 `currentGuess`가 바뀌는 흐름과 함께 동작한다.

### FlatList 컨테이너는 레이아웃의 일부다

`FlatList`는 목록 렌더링 컴포넌트이지만, 혼자서 모든 레이아웃 문제를 해결하지 않는다.

```text
FlatList
-> 무엇을 어떻게 렌더링할지 담당

부모 View
-> 목록이 차지할 영역을 결정
```

목록이 화면 안에서 어디까지 차지해야 하는지, 다른 UI와 어떻게 나눠 가져야 하는지는 여전히 Flexbox와 부모 컨테이너의 책임이다.

## 정리

이번 단계에서는 두 가지 마무리 작업을 했다.

첫째, `FlatList`를 감싸는 `listContainer`에 `flex: 1`과 `padding`을 추가해 목록이 화면 경계를 벗어나지 않고 안정적으로 스크롤되도록 만들었다.

둘째, 게임이 끝났을 때 `GameScreen`이 `guessRounds.length`를 `onGameOver` 콜백에 전달하고, `App`이 이를 상태로 저장해 `GameOverScreen`에 넘기도록 했다.

이로써 게임 진행 화면의 로그 목록과 게임 오버 화면의 결과 요약이 연결되었고, 사용자는 장치가 몇 라운드 만에 숫자를 맞혔는지 확인할 수 있게 되었다.
