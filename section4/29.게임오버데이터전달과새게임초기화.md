# 29. 게임 오버 데이터 전달과 새 게임 초기화

## 핵심 요약

- `GameOverScreen`에 표시되는 라운드 수와 정답 숫자는 `App`에서 props로 전달한다.
- `GameOverScreen`은 `roundsNumber`, `userNumber`, `onStartNewGame`을 받아 화면에 표시하고 버튼에 연결한다.
- `App`에서는 `guessRounds` 상태를 추가해 라운드 수를 관리할 준비를 한다.
- 새 게임 시작 버튼을 누르면 `userNumber`를 `null`로 되돌려 시작 화면으로 돌아간다.
- 새 게임이 시작될 때 `guessRounds`도 `0`으로 초기화한다.
- `GameScreen`의 `minBoundary`, `maxBoundary`는 컴포넌트 외부 변수이므로 새 게임 시작 시점에 재설정해야 한다.
- `GameScreen`이 처음 마운트될 때 실행되는 `useEffect`를 사용해 경계값을 초기화할 수 있다.
- 이번 단계에서는 게임 재시작 흐름은 완성하지만, 라운드 수 증가 로직은 다음 단계에서 구현한다.

## 학습 목표

- 부모 컴포넌트에서 화면 컴포넌트로 결과 데이터를 props로 전달하는 방법을 익힌다.
- `GameOverScreen`에서 props를 객체 구조 분해로 받아 사용하는 방식을 이해한다.
- 새 게임 시작 시 여러 상태를 초기화하는 흐름을 배운다.
- 조건부 렌더링이 상태 초기화와 어떻게 연결되는지 이해한다.
- 컴포넌트 외부 변수의 장점과 초기화 필요성을 함께 이해한다.
- `useEffect`의 빈 의존성 배열을 사용해 컴포넌트 마운트 시점 로직을 실행하는 방법을 익힌다.

## 주요 내용

### GameOverScreen에 필요한 실제 데이터

이전 단계에서는 게임 오버 화면에 플레이스홀더를 넣었다.

```text
Your phone needed X rounds to guess the number Y.
```

이제 `X`와 `Y`를 실제 값으로 교체해야 한다.

```text
X
-> 장치가 몇 번의 라운드 만에 정답을 맞혔는지

Y
-> 사용자가 선택한 정답 숫자
```

또 `Start New Game` 버튼도 실제로 새 게임을 시작하도록 만들어야 한다.

```text
Start New Game 버튼
-> 게임 상태 초기화
-> 시작 화면으로 돌아가기
```

### GameOverScreen은 데이터를 직접 관리하지 않는다

`GameOverScreen`은 `App`에서 렌더링되는 화면이다.

게임 상태의 중심은 `App`이 관리하고 있으므로, `GameOverScreen`은 필요한 데이터를 props로 받는 것이 자연스럽다.

```text
App
-> userNumber 상태 보유
-> guessRounds 상태 보유 예정
-> 새 게임 시작 함수 보유

GameOverScreen
-> 전달받은 값을 표시
-> 전달받은 함수를 버튼에 연결
```

이 화면은 결과를 보여주는 역할에 집중하고, 상태 관리는 부모에 맡긴다.

### GameOverScreen에서 props 받기

`GameOverScreen`은 세 가지 값을 받는다.

```jsx
function GameOverScreen({
  roundsNumber,
  userNumber,
  onStartNewGame,
}) {
  // ...
}
```

각 prop의 역할은 다음과 같다.

```text
roundsNumber
-> 정답을 맞히기까지 걸린 라운드 수

userNumber
-> 사용자가 선택한 숫자

onStartNewGame
-> 새 게임 시작 버튼을 눌렀을 때 실행할 함수
```

객체 구조 분해를 사용하면 `props.roundsNumber`처럼 접근하지 않아도 된다.

### 결과 텍스트에 실제 값 넣기

기존 플레이스홀더 `X`, `Y`를 props 값으로 교체한다.

```jsx
<Text style={styles.summaryText}>
  Your phone needed{' '}
  <Text style={styles.highlight}>{roundsNumber}</Text>
  {' '}rounds to guess the number{' '}
  <Text style={styles.highlight}>{userNumber}</Text>.
</Text>
```

라운드 수와 정답 숫자는 계속 강조 스타일을 유지한다.

```text
roundsNumber
-> highlight 스타일

userNumber
-> highlight 스타일
```

데이터만 실제 값으로 바뀌고, UI 구조는 이전 단계와 같다.

### 새 게임 버튼에 함수 연결하기

`PrimaryButton`에 `onPress` prop을 추가한다.

```jsx
<PrimaryButton onPress={onStartNewGame}>
  Start New Game
</PrimaryButton>
```

`GameOverScreen`은 새 게임을 어떻게 시작해야 하는지 직접 알 필요가 없다.

버튼이 눌렸을 때 부모가 전달한 함수를 호출하기만 하면 된다.

```text
GameOverScreen
-> 버튼 클릭 감지
-> onStartNewGame 호출

App
-> 실제 상태 초기화 처리
```

이것도 React에서 자주 쓰는 “부모가 상태와 함수를 소유하고 자식은 호출만 하는” 패턴이다.

### App에서 GameOverScreen에 props 전달하기

`App`에서 `GameOverScreen`을 렌더링할 때 필요한 값을 전달한다.

```jsx
if (gameIsOver && userNumber) {
  screen = (
    <GameOverScreen
      roundsNumber={guessRounds}
      userNumber={userNumber}
      onStartNewGame={startNewGameHandler}
    />
  );
}
```

이제 `GameOverScreen`은 `App`이 가진 상태와 함수를 사용할 수 있다.

```text
guessRounds
-> roundsNumber로 전달

userNumber
-> userNumber로 전달

startNewGameHandler
-> onStartNewGame으로 전달
```

### guessRounds 상태 추가하기

라운드 수를 표시하려면 `App`에서 라운드 수 상태를 준비해야 한다.

```jsx
const [guessRounds, setGuessRounds] = useState(0);
```

초기값은 `0`이다.

게임이 아직 시작되지 않았거나 새 게임이 시작되기 전에는 시도 횟수가 없기 때문이다.

```text
guessRounds = 0
-> 아직 라운드 없음
```

이번 단계에서는 이 상태를 만들고 전달만 한다.

실제 라운드 수를 증가시키는 로직은 다음 강의에서 구현한다.

### startNewGameHandler 만들기

새 게임 시작 버튼을 눌렀을 때 실행할 함수를 `App`에 만든다.

```jsx
function startNewGameHandler() {
  setUserNumber(null);
  setGuessRounds(0);
}
```

새 게임을 시작하려면 최소 두 가지를 초기화해야 한다.

```text
userNumber
-> null로 초기화
-> 선택된 숫자가 없는 상태로 되돌림

guessRounds
-> 0으로 초기화
-> 새 게임의 라운드 수를 다시 시작
```

`gameIsOver`는 이미 게임 오버 화면에 있는 시점에서 `true`이므로 꼭 다시 설정할 필요는 없다.

필요하다면 명시적으로 유지할 수도 있지만, 이번 흐름에서는 `userNumber`를 `null`로 만드는 것만으로 화면 전환이 해결된다.

### userNumber를 null로 만들면 시작 화면으로 돌아간다

현재 `App`의 화면 전환은 `userNumber`와 `gameIsOver` 조건에 따라 결정된다.

```jsx
let screen = <StartGameScreen onPickNumber={pickedNumberHandler} />;

if (userNumber) {
  screen = <GameScreen ... />;
}

if (gameIsOver && userNumber) {
  screen = <GameOverScreen ... />;
}
```

새 게임 시작 시 `userNumber`를 `null`로 만들면 두 조건이 모두 거짓이 된다.

```text
userNumber = null

if (userNumber)
-> false

if (gameIsOver && userNumber)
-> false
```

따라서 기본값으로 설정된 `StartGameScreen`이 다시 렌더링된다.

```text
Start New Game 클릭
-> setUserNumber(null)
-> GameOverScreen 조건 해제
-> StartGameScreen 표시
```

화면 전환을 위한 별도 네비게이션 API 없이도 상태만으로 화면을 바꿀 수 있다.

### GameScreen의 경계값도 초기화해야 한다

게임 진행 중에는 `minBoundary`, `maxBoundary`가 계속 바뀐다.

```jsx
let minBoundary = 1;
let maxBoundary = 100;
```

이 값들은 컴포넌트 함수 바깥에 있다.

이유는 `GameScreen`이 리렌더링될 때마다 초기값으로 돌아가면 안 되기 때문이다.

```text
컴포넌트 함수 내부 변수
-> 렌더링마다 초기화될 수 있음

컴포넌트 외부 변수
-> 추측 범위 유지 가능
```

하지만 새 게임이 시작되면 이 값들도 다시 초기 범위로 돌아가야 한다.

```text
새 게임 시작
-> minBoundary = 1
-> maxBoundary = 100
```

그렇지 않으면 이전 게임에서 좁혀진 범위가 다음 게임에 영향을 줄 수 있다.

### GameScreen 마운트 시 경계값 초기화하기

강의에서는 `GameScreen`에서 별도 `useEffect`를 사용해 경계값을 초기화한다.

```jsx
useEffect(() => {
  minBoundary = 1;
  maxBoundary = 100;
}, []);
```

빈 의존성 배열을 전달하면 이 effect는 컴포넌트가 처음 렌더링될 때 한 번 실행된다.

```text
[] 의존성 배열
-> 컴포넌트 마운트 시 한 번 실행
-> 같은 화면에서 상태 업데이트로 리렌더링될 때는 다시 실행되지 않음
```

게임이 끝나면 `GameScreen`은 화면에서 제거된다.

새 게임을 시작하고 다시 숫자를 선택하면 `GameScreen`이 다시 마운트되므로 이 effect가 다시 실행된다.

```text
첫 게임 시작
-> GameScreen 마운트
-> 경계값 초기화

게임 종료
-> GameScreen 언마운트

새 게임 시작
-> GameScreen 다시 마운트
-> 경계값 다시 초기화
```

### 다른 위치에서 초기화할 수도 있다

경계값은 게임이 끝나는 순간 초기화할 수도 있다.

예를 들어 `onGameOver`를 호출할 때 함께 처리할 수 있다.

```text
방법 1
-> 게임 종료 시 경계값 초기화

방법 2
-> GameScreen 마운트 시 경계값 초기화
```

강의에서는 두 번째 방식을 선택한다.

이 방식은 “새 게임 화면이 시작될 때 필요한 초기 조건을 설정한다”는 관점에서 이해하기 쉽다.

### GameScreen의 useEffect 두 개

현재 `GameScreen`에는 서로 다른 목적의 `useEffect`가 있을 수 있다.

```jsx
useEffect(() => {
  if (currentGuess === userNumber) {
    onGameOver();
  }
}, [currentGuess, userNumber, onGameOver]);
```

이 effect는 게임 종료 여부를 감지한다.

```jsx
useEffect(() => {
  minBoundary = 1;
  maxBoundary = 100;
}, []);
```

이 effect는 화면이 처음 렌더링될 때 경계값을 초기화한다.

```text
첫 번째 useEffect
-> 상태/props 변화에 반응

두 번째 useEffect
-> 마운트 시 한 번 실행
```

목적이 다르므로 별도 effect로 나누는 것이 자연스럽다.

### 이번 단계 후 동작

이번 단계가 끝나면 다음 흐름이 가능해진다.

```text
1. 시작 화면에서 숫자 선택
2. 게임 화면에서 추측 진행
3. 정답을 맞히면 게임 오버 화면 표시
4. 게임 오버 화면에서 Start New Game 클릭
5. 시작 화면으로 돌아감
6. 새 숫자를 선택해 다시 게임 진행
```

라운드 수는 아직 실제로 증가하지 않으므로 결과 화면에는 `0`이 표시될 수 있다.

이 부분은 다음 단계에서 구현한다.

### 프론트엔드 개발자 관점에서의 핵심 차이

웹 React 개발자 입장에서는 익숙한 상태 끌어올리기 패턴이다.

```text
App
-> 상태 소유
-> 화면 전환 조건 소유
-> 자식에게 데이터와 콜백 전달

GameOverScreen
-> props로 받은 데이터 표시
-> props로 받은 콜백 실행
```

React Native라고 해서 데이터 흐름이 달라지는 것은 아니다.

다만 화면 전환을 네비게이션 라이브러리 없이 상태 조건으로 처리하고 있다는 점이 현재 앱의 특징이다.

```text
웹 React
-> state에 따라 다른 컴포넌트 렌더링

React Native
-> state에 따라 다른 screen 컴포넌트 렌더링
```

또 하나 중요한 점은 컴포넌트 외부 변수 관리다.

```text
컴포넌트 외부 변수
-> 리렌더링 사이에서 값 유지
-> 하지만 새 플로우 시작 시 직접 초기화 필요
```

이런 값은 상태가 아니므로 React가 자동으로 초기화해주지 않는다.

따라서 새 게임 시작처럼 명확한 생명주기 지점에서 직접 되돌려야 한다.

### 실무 관점에서의 주의점

컴포넌트 외부 변수는 간단한 예제에서는 편리하지만, 실제 앱에서는 주의해서 사용해야 한다.

```text
장점
-> 리렌더링과 무관하게 값 유지
-> 상태 업데이트 없이 내부 계산 범위 유지 가능

주의점
-> 여러 인스턴스가 생기면 공유될 수 있음
-> 새 플로우 시작 시 직접 초기화해야 함
-> 상태와 다르게 변경되어도 렌더링을 유발하지 않음
```

이번 앱에서는 `GameScreen`이 하나만 존재하고 단순한 게임이므로 충분히 괜찮다.

하지만 앱이 복잡해지면 `useRef`, reducer, 상태 구조 개선 등을 고려할 수 있다.

## 정리

이번 강의에서는 `GameOverScreen`에 실제 데이터를 전달할 준비를 했다. `roundsNumber`, `userNumber`, `onStartNewGame`을 props로 받아 결과 문장에 라운드 수와 정답 숫자를 표시하고, 새 게임 버튼에는 `onStartNewGame` 함수를 연결했다.

`App`에서는 `guessRounds` 상태를 추가하고, `GameOverScreen`에 `guessRounds`, `userNumber`, `startNewGameHandler`를 전달했다. 새 게임 시작 시에는 `userNumber`를 `null`로, `guessRounds`를 `0`으로 되돌려 시작 화면으로 돌아가게 만들었다.

또한 `GameScreen` 외부에서 관리하는 `minBoundary`, `maxBoundary`는 새 게임마다 초기화되어야 하므로, `GameScreen`이 마운트될 때 실행되는 `useEffect`에서 각각 `1`, `100`으로 되돌리도록 정리했다. 라운드 수를 실제로 증가시키는 작업은 다음 단계에서 이어진다.
