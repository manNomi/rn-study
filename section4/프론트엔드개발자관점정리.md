# 섹션 4. 프론트엔드 개발자 관점 정리

## 한 줄 요약

React Native의 핵심 컴포넌트와 React 상태 관리 지식을 바탕으로 숫자 맞히기 게임 앱을 만들며, 모바일 앱다운 화면 구성, 커스텀 UI, 스타일링, 에셋, 폰트, 아이콘, 목록, 간단한 화면 전환을 종합적으로 연습하는 섹션이다.

## 이미 알고 있다고 보는 것

- React 함수 컴포넌트
- JSX와 props
- `children`을 활용한 래퍼 컴포넌트
- `useState`, `useEffect`
- 조건부 렌더링
- 이벤트 핸들러와 콜백 전달
- 배열 상태 업데이트
- `map()`과 리스트 렌더링
- CSS Flexbox의 기본 개념
- 공통 컴포넌트와 도메인 컴포넌트를 나누는 감각
- 디자인 토큰이나 상수 파일로 색상 값을 관리하는 개념

## 새로 잡아야 하는 관점

섹션 4는 React Native의 기본 문법을 넘어서 "앱 하나를 실제로 구성하는 감각"을 잡는 구간이다.

웹 프론트엔드에서는 보통 라우터, CSS, DOM, 브라우저 스크롤, 웹 폰트, SVG 아이콘 등을 자연스럽게 사용한다.

React Native에서는 같은 문제를 다른 도구로 푼다.

```text
웹 라우팅
-> 지금 단계에서는 상태 기반 조건부 렌더링

CSS 파일 / CSS 변수
-> StyleSheet 객체 / constants 파일

HTML button
-> Pressable 기반 커스텀 버튼

body 스크롤
-> FlatList / ScrollView와 부모 컨테이너

웹 폰트
-> expo-font로 앱 시작 시 로딩

SVG 아이콘 패키지
-> @expo/vector-icons

배경 이미지 / 그라디언트
-> ImageBackground / expo-linear-gradient
```

즉 React의 사고방식은 그대로 가져가되, 브라우저에서 당연했던 도구들을 React Native 방식으로 다시 매핑해야 한다.

## 앱 구조

이번 섹션에서 만든 앱은 세 개의 주요 화면으로 구성된다.

```text
StartGameScreen
-> 사용자가 1부터 99 사이의 숫자를 입력

GameScreen
-> 앱이 숫자를 추측하고 사용자가 higher/lower 힌트를 제공

GameOverScreen
-> 정답과 라운드 수를 보여주고 새 게임 시작
```

폴더도 역할에 맞게 나누었다.

```text
screens
-> 화면 단위 컴포넌트

components/ui
-> 여러 화면에서 재사용 가능한 UI 컴포넌트

components/game
-> 게임 도메인에 가까운 컴포넌트

constants
-> 색상 같은 공통 상수

assets
-> 이미지, 폰트 등 정적 자산
```

프론트엔드 개발자 관점에서는 Next.js의 `pages`나 `app`, `components`, `shared`, `entities`처럼 역할별로 파일을 나누는 것과 같은 흐름으로 이해하면 된다.

## 화면 전환

이 섹션에서는 React Navigation 같은 라이브러리를 쓰지 않고, `App` 컴포넌트의 상태와 조건문으로 화면을 바꿨다.

```jsx
let screen = <StartGameScreen onPickNumber={pickedNumberHandler} />;

if (userNumber) {
  screen = (
    <GameScreen
      userNumber={userNumber}
      onGameOver={gameOverHandler}
    />
  );
}

if (gameIsOver && userNumber) {
  screen = (
    <GameOverScreen
      roundsNumber={guessRounds}
      userNumber={userNumber}
      onStartNewGame={startNewGameHandler}
    />
  );
}
```

이 방식은 간단한 앱에서는 충분하다.

```text
userNumber 없음
-> 시작 화면

userNumber 있음 + 게임 진행 중
-> 게임 화면

userNumber 있음 + 게임 종료
-> 게임 오버 화면
```

웹 React에서도 `state`에 따라 모달, 탭, 단계별 폼 화면을 바꾸는 경우가 많다. 그 패턴을 앱 전체 화면 전환에 적용한 것이다.

다만 화면이 많아지고, 뒤로 가기, 스택, 탭, 딥링크, 헤더 제어가 필요해지면 전용 내비게이션 라이브러리를 써야 한다.

## 커스텀 UI 컴포넌트

이번 섹션의 핵심 중 하나는 React Native 핵심 컴포넌트를 조합해 직접 UI 컴포넌트를 만드는 것이다.

### PrimaryButton

기본 `Button` 대신 `Pressable`, `View`, `Text`를 조합해 직접 버튼을 만들었다.

```text
Pressable
-> 터치 이벤트와 pressed 상태 처리

View
-> 배경색, 둥근 모서리, 그림자, overflow 처리

Text
-> 버튼 라벨 출력
```

Android에서는 `android_ripple`로 물결 효과를 줄 수 있고, iOS에서는 `Pressable`의 `style` 함수로 눌림 상태를 반영할 수 있다.

```jsx
<Pressable
  onPress={onPress}
  android_ripple={{ color: Colors.primary600 }}
  style={({ pressed }) =>
    pressed
      ? [styles.buttonInnerContainer, styles.pressed]
      : styles.buttonInnerContainer
  }
>
  <Text style={styles.buttonText}>{children}</Text>
</Pressable>
```

웹의 `button:hover`, `button:active` 감각을 React Native에서는 `Pressable`의 상태 기반 스타일로 구현한다고 보면 된다.

### Title, Card, InstructionText

반복되는 스타일은 공통 UI 컴포넌트로 분리했다.

```text
Title
-> 화면 제목

Card
-> 입력 영역과 컨트롤 영역을 감싸는 공통 박스

InstructionText
-> 안내 문구
```

특히 `InstructionText`는 외부 스타일을 받아 기본 스타일과 병합했다.

```jsx
function InstructionText({ children, style }) {
  return (
    <Text style={[styles.instructionText, style]}>
      {children}
    </Text>
  );
}
```

React Native에는 CSS cascade가 없지만, 스타일 배열을 이용하면 비슷한 확장 패턴을 만들 수 있다.

```text
앞쪽 스타일
-> 기본값

뒤쪽 스타일
-> 화면별 override 또는 추가 스타일
```

### NumberContainer와 GuessLogItem

게임 도메인 전용 컴포넌트도 만들었다.

```text
NumberContainer
-> 현재 앱이 추측한 숫자 표시

GuessLogItem
-> 각 라운드의 추측 기록 표시
```

공통 UI 컴포넌트와 달리 이 컴포넌트들은 게임이라는 특정 도메인에 강하게 묶여 있다.

웹 프론트엔드에서도 `Button`, `Card` 같은 shared UI와 `OrderItem`, `ProductSummary` 같은 도메인 컴포넌트를 구분하는 것과 같은 감각이다.

## 스타일링

React Native 스타일은 `StyleSheet.create()`로 정의하고 `style` prop에 연결한다.

```jsx
const styles = StyleSheet.create({
  inputContainer: {
    marginHorizontal: 24,
    padding: 16,
    backgroundColor: Colors.primary800,
    borderRadius: 8,
  },
});
```

이번 섹션에서는 다음 스타일링 주제를 많이 연습했다.

```text
Flexbox
-> 버튼 행, 화면 중앙 정렬, 목록 영역 배치

borderRadius + overflow
-> 둥근 버튼, 원형 이미지

elevation
-> Android 그림자

shadowColor / shadowOffset / shadowOpacity / shadowRadius
-> iOS 그림자

fontFamily
-> 커스텀 폰트 적용

width / height / percentage
-> 이미지와 입력 필드 크기 제어
```

중요한 차이는 플랫폼별 스타일이다.

```text
Android 그림자
-> elevation

iOS 그림자
-> shadow* 속성 조합
```

하나의 스타일 객체로 끝나지 않을 때가 있다는 점을 받아들여야 한다.

## 디자인 토큰과 색상 상수

색상 값을 여러 파일에 하드코딩하지 않고 `constants/colors.js`로 분리했다.

```jsx
const Colors = {
  primary500: '#72063c',
  primary600: '#640233',
  primary700: '#4e0329',
  primary800: '#3b021f',
  accent500: '#ddb52f',
};

export default Colors;
```

사용하는 쪽에서는 import해서 쓴다.

```jsx
backgroundColor: Colors.primary800
```

웹의 CSS 변수, Tailwind config, theme token과 비슷한 역할이다.

```text
색상을 한곳에서 관리
-> 변경 비용 감소
-> 화면 간 일관성 유지
-> 컴포넌트 스타일의 의미가 명확해짐
```

나중에는 색상뿐 아니라 spacing, radius, typography도 이런 식으로 확장할 수 있다.

## 입력과 유효성 검사

숫자 입력은 `TextInput`으로 처리했다.

```jsx
<TextInput
  style={styles.numberInput}
  maxLength={2}
  keyboardType="number-pad"
  autoCapitalize="none"
  autoCorrect={false}
  value={enteredNumber}
  onChangeText={numberInputHandler}
/>
```

웹의 `input type="number"`와 비슷한 목적이지만, 모바일에서는 키보드 타입이 매우 중요하다.

```text
keyboardType="number-pad"
-> 숫자 입력에 맞는 모바일 키보드 표시
```

유효성 검사는 JavaScript로 수행했다.

```jsx
const chosenNumber = parseInt(enteredNumber);

if (isNaN(chosenNumber) || chosenNumber <= 0 || chosenNumber > 99) {
  Alert.alert(
    'Invalid number!',
    'Number has to be a number between 1 and 99.',
    [{ text: 'Okay', style: 'destructive', onPress: resetInputHandler }]
  );
  return;
}
```

React Native의 `Alert` API는 각 플랫폼의 네이티브 알림 UI를 띄운다.

```text
웹 alert
-> 브라우저 기본 alert

React Native Alert
-> iOS / Android 네이티브 다이얼로그
```

## 게임 로직과 상태 관리

게임의 핵심 상태는 다음과 같다.

```text
App
-> userNumber
-> gameIsOver
-> guessRounds 총 라운드 수

GameScreen
-> currentGuess
-> guessRounds 배열
```

앱이 숫자를 추측할 때는 최소/최대 경계를 좁혀 간다.

```text
사용자가 "더 작다"를 누름
-> maxBoundary를 currentGuess로 갱신

사용자가 "더 크다"를 누름
-> minBoundary를 currentGuess + 1로 갱신
```

잘못된 힌트를 주면 무한 루프에 빠질 수 있으므로 방어 로직도 추가했다.

```jsx
if (
  (direction === 'lower' && currentGuess < userNumber) ||
  (direction === 'greater' && currentGuess > userNumber)
) {
  Alert.alert("Don't lie!", 'You know that this is wrong...', [
    { text: 'Sorry!', style: 'cancel' },
  ]);
  return;
}
```

프론트엔드 실무에서도 사용자 입력이나 이벤트가 항상 올바르다고 가정하면 안 된다. UI 레벨에서도 상태가 불가능한 방향으로 흐르지 않도록 막아야 한다.

## useEffect의 역할

`GameScreen`에서는 `useEffect`로 게임 종료 여부를 감지했다.

```jsx
useEffect(() => {
  if (currentGuess === userNumber) {
    onGameOver(guessRounds.length);
  }
}, [currentGuess, userNumber, onGameOver, guessRounds]);
```

핵심은 "렌더링 결과에 따라 부수 효과를 실행한다"는 React의 기존 개념 그대로다.

```text
currentGuess 변경
-> 컴포넌트 리렌더링
-> effect 실행
-> 정답이면 App에 게임 종료 알림
```

또 컴포넌트가 새 게임에서 다시 마운트될 때 경계값을 초기화하기 위해 별도 `useEffect`를 사용할 수도 있다.

```jsx
useEffect(() => {
  minBoundary = 1;
  maxBoundary = 100;
}, []);
```

다만 컴포넌트 밖 변수는 React 상태가 아니므로 주의해서 써야 한다. 예제에서는 컴포넌트 리렌더링 때마다 경계값이 초기화되는 것을 막기 위해 사용했다.

## 목록 렌더링

추측 기록은 처음에는 배열을 직접 렌더링할 수 있지만, 최종적으로 `FlatList`를 사용했다.

```jsx
<FlatList
  data={guessRounds}
  keyExtractor={(item) => item.toString()}
  renderItem={(itemData) => (
    <GuessLogItem
      roundNumber={guessRoundsListLength - itemData.index}
      guess={itemData.item}
    />
  )}
/>
```

`FlatList`는 다음 세 가지가 핵심이다.

```text
data
-> 렌더링할 배열

renderItem
-> 각 항목을 어떻게 그릴지

keyExtractor
-> 각 항목의 고유 key를 어떻게 만들지
```

이번 예제에서는 최신 추측값을 배열 앞에 추가했으므로 라운드 번호 계산이 필요했다.

```jsx
roundNumber={guessRoundsListLength - itemData.index}
```

그리고 `FlatList`가 화면 하단을 벗어나지 않게 부모 컨테이너에 `flex: 1`을 주었다.

```jsx
listContainer: {
  flex: 1,
  padding: 16,
}
```

웹에서는 긴 컨텐츠가 자연스럽게 페이지 스크롤로 이어지는 경우가 많지만, React Native에서는 스크롤 영역과 부모 높이를 명시적으로 설계해야 한다.

## 에셋, 그라디언트, 이미지, 폰트, 아이콘

이번 섹션에서는 앱다운 시각 요소도 다뤘다.

### LinearGradient

Expo 패키지인 `expo-linear-gradient`를 사용했다.

```jsx
<LinearGradient
  colors={[Colors.primary700, Colors.accent500]}
  style={styles.rootScreen}
>
  {screen}
</LinearGradient>
```

Expo 프로젝트에서는 호환성을 위해 보통 `expo install`을 사용한다.

```bash
expo install expo-linear-gradient
```

### ImageBackground

배경 이미지는 `ImageBackground`로 처리했다.

```jsx
<ImageBackground
  source={require('./assets/images/background.png')}
  resizeMode="cover"
  style={styles.rootScreen}
  imageStyle={styles.backgroundImage}
>
  {screen}
</ImageBackground>
```

`imageStyle`을 사용하면 내부 이미지의 투명도 같은 스타일을 조정할 수 있다.

```jsx
backgroundImage: {
  opacity: 0.15,
}
```

### expo-font

커스텀 폰트는 `expo-font`의 `useFonts`로 앱 시작 시 로딩했다.

```jsx
const [fontsLoaded] = useFonts({
  'open-sans': require('./assets/fonts/OpenSans-Regular.ttf'),
  'open-sans-bold': require('./assets/fonts/OpenSans-Bold.ttf'),
});
```

폰트가 로딩되기 전에는 앱 UI를 바로 렌더링하지 않도록 처리했다.

웹의 `@font-face`나 Google Fonts와 목적은 비슷하지만, React Native에서는 앱 런타임에서 폰트를 로딩하고 `fontFamily`로 참조한다.

### @expo/vector-icons

`@expo/vector-icons`로 버튼 안의 `+`, `-` 문자를 아이콘으로 바꿨다.

```jsx
import { Ionicons } from '@expo/vector-icons';

<Ionicons name="md-add" size={24} color="white" />
```

Expo 프로젝트에서는 기본적으로 사용할 수 있는 아이콘 세트가 많기 때문에 모바일 앱 UI를 빠르게 구성하기 좋다.

## SafeAreaView와 모바일 화면 감각

모바일 기기에는 노치, 상태바, 둥근 모서리 등이 있다.

웹 브라우저의 직사각형 뷰포트만 생각하면 앱 UI가 기기 상단에 너무 붙거나 노치에 가릴 수 있다.

그래서 `SafeAreaView`를 사용했다.

```jsx
<SafeAreaView style={styles.rootScreen}>
  {screen}
</SafeAreaView>
```

중요한 점은 배경은 노치 뒤까지 깔고, 실제 콘텐츠만 안전 영역 안에 넣는 구조다.

```text
LinearGradient / ImageBackground
-> 전체 화면 배경

SafeAreaView
-> 실제 화면 콘텐츠 보호
```

이 감각은 웹 프론트엔드에는 상대적으로 덜 익숙하지만, 모바일 앱에서는 거의 기본이다.

## 프론트엔드 개발자가 특히 기억할 차이

```text
라우터 없이 화면 전환

CSS 변수
-> constants/colors.js

button 커스터마이징
-> Pressable + View + Text

active 상태
-> Pressable style 함수

긴 목록
-> FlatList + 부모 컨테이너 flex

브라우저 alert
-> React Native Alert

웹 폰트
-> expo-font + useFonts

아이콘 라이브러리
-> @expo/vector-icons

배경 이미지
-> ImageBackground + require()

노치 대응
-> SafeAreaView

플랫폼별 그림자
-> Android elevation / iOS shadow*
```

## 섹션 4에서 얻어야 할 결론

섹션 4는 React Native의 여러 기본기를 실제 앱에 통합하는 구간이다.

프론트엔드 개발자가 이 섹션에서 가져가야 할 핵심은 다음이다.

```text
React의 상태와 컴포넌트 사고방식은 그대로 통한다.

하지만 UI를 구성하는 기본 재료는 HTML/CSS가 아니라
React Native 핵심 컴포넌트와 StyleSheet다.

간단한 앱은 state와 조건부 렌더링만으로도 화면 전환이 가능하다.

모바일 앱은 입력 키보드, 터치 피드백, SafeArea, 플랫폼별 스타일,
에셋 로딩, 스크롤 영역 같은 요소를 더 명시적으로 다뤄야 한다.
```

이제 React Native가 낯선 "문법"이라기보다는, React를 모바일 앱 환경에 맞게 사용하는 방식이라는 감각을 가져가면 된다.

다음 단계에서는 같은 앱을 다양한 화면 크기와 iOS/Android 플랫폼 차이에 맞게 더 유연하게 조정하는 방법을 배우게 된다.
