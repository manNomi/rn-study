# 22. useEffect로 게임 종료 처리

## 핵심 요약

- 장치의 현재 추측값이 사용자가 고른 숫자와 같아지면 게임을 종료해야 한다.
- `currentGuess`가 바뀔 때마다 게임 종료 여부를 확인해야 하므로 `useEffect`를 사용한다.
- `GameScreen`은 게임 종료 사실을 직접 화면 전환으로 처리하지 않고, `App`에서 받은 `onGameOver` 함수를 호출한다.
- `App`은 `gameIsOver` 상태를 추가해 시작 화면, 게임 화면, 게임 오버 화면 중 무엇을 보여줄지 결정한다.
- 게임 시작 시에는 `gameIsOver`를 `false`로 바꾸고, 게임 종료 시에는 다시 `true`로 바꾼다.
- `useEffect`는 렌더링 이후 실행되므로, 렌더링 중 실행되는 계산 코드와의 타이밍 차이를 이해해야 한다.
- `initialGuess`를 매 렌더링마다 현재 경계값으로 다시 계산하면 무한 재귀 문제가 생길 수 있어, 초기 경계값을 고정해 해결한다.

## 학습 목표

- `useEffect`를 사용해 상태 변경에 반응하는 로직을 작성하는 방법을 익힌다.
- 자식 컴포넌트에서 부모 컴포넌트의 상태를 변경하도록 콜백 prop을 전달하는 패턴을 이해한다.
- 조건부 렌더링으로 시작 화면, 게임 화면, 게임 오버 화면을 전환하는 구조를 이해한다.
- `useEffect`의 의존성 배열에 어떤 값을 넣어야 하는지 배운다.
- React 컴포넌트 함수 실행과 `useEffect` 실행 타이밍의 차이를 이해한다.
- 렌더링 중 계산되는 값이 반복 실행될 때 어떤 문제가 생길 수 있는지 파악한다.

## 주요 내용

### 아직 남아 있는 문제

이전 단계에서 `+`, `-` 버튼을 통해 앱의 추측 범위를 좁힐 수 있게 되었다.

하지만 장치가 정답을 맞혔을 때 게임을 종료하는 로직은 아직 없다.

```text
사용자가 고른 숫자: 31
장치의 현재 추측값: 31

기대 동작
-> 게임 종료
-> GameOverScreen 표시

현재 문제
-> 종료 처리 없음
-> 특정 상황에서 무한 루프 또는 충돌 발생
```

따라서 현재 추측값과 정답을 비교하고, 두 값이 같아지는 순간 앱 전체 화면을 게임 오버 화면으로 전환해야 한다.

### 게임 종료 조건

게임 종료 조건은 단순하다.

```jsx
currentGuess === userNumber
```

`currentGuess`는 장치가 현재 추측한 숫자다.

`userNumber`는 사용자가 시작 화면에서 선택한 숫자다.

두 값이 같으면 장치가 정답을 맞힌 것이므로 게임이 끝난다.

```text
currentGuess
-> GameScreen 내부 상태

userNumber
-> App에서 관리하고 GameScreen에 전달한 값

두 값이 같음
-> 게임 종료
```

### 왜 useEffect를 사용하는가

게임 종료 여부는 `currentGuess`가 변경될 때마다 확인해야 한다.

React에서는 특정 상태나 prop이 바뀐 뒤에 부수 효과를 실행할 때 `useEffect`를 사용한다.

```jsx
import { useEffect } from 'react';
```

`GameScreen`에서 다음처럼 사용할 수 있다.

```jsx
useEffect(() => {
  if (currentGuess === userNumber) {
    onGameOver();
  }
}, [currentGuess, userNumber, onGameOver]);
```

이 코드는 `currentGuess`, `userNumber`, `onGameOver` 중 하나가 변경될 때마다 실행된다.

```text
currentGuess 변경
-> 새 추측값이 정답인지 확인

userNumber 변경
-> 정답이 바뀌었을 때 다시 확인

onGameOver 변경
-> 전달받은 콜백이 바뀌었을 때 다시 확인
```

### 의존성 배열에 넣어야 하는 값

`useEffect` 안에서 사용하는 외부 값은 의존성 배열에 넣는 것이 기본 규칙이다.

이번 effect 안에서는 세 값을 사용한다.

```jsx
currentGuess
userNumber
onGameOver
```

따라서 의존성 배열은 다음처럼 작성한다.

```jsx
[currentGuess, userNumber, onGameOver]
```

이렇게 하면 effect가 최신 값을 기준으로 실행된다.

```text
의존성 배열
-> effect가 어떤 값의 변화에 반응할지 알려주는 목록
```

React 웹 개발에서 사용하는 `useEffect` 규칙과 동일하다.

### GameScreen이 직접 화면을 바꾸지 않는다

`GameScreen`은 게임 종료 조건을 알 수 있다.

하지만 실제로 어떤 화면을 보여줄지는 `App`이 관리한다.

현재 화면 전환 구조가 `App` 안에 있기 때문이다.

```text
App
-> StartGameScreen 렌더링
-> GameScreen 렌더링
-> GameOverScreen 렌더링

GameScreen
-> 게임 진행 로직 담당
-> 종료 조건 감지
```

따라서 `GameScreen`은 게임이 끝났다는 사실을 부모인 `App`에 알려야 한다.

이를 위해 콜백 prop을 사용한다.

### App에 gameIsOver 상태 추가

`App`에서는 게임 종료 여부를 추적하는 상태를 추가한다.

```jsx
const [gameIsOver, setGameIsOver] = useState(true);
```

초기값을 `true`로 둔다.

게임이 아직 시작되지 않은 상태를 “게임 중이 아님”으로 볼 수 있기 때문이다.

```text
gameIsOver = true
-> 아직 게임이 시작되지 않음
-> 또는 게임이 끝남

gameIsOver = false
-> 게임 진행 중
```

이 상태는 화면 전환 조건에 사용된다.

### GameOverScreen 기본 구성

아직 완성된 게임 오버 화면은 아니지만, 우선 화면 컴포넌트를 만든다.

```jsx
import { Text } from 'react-native';

function GameOverScreen() {
  return <Text>Game is over!</Text>;
}

export default GameOverScreen;
```

이후 스타일링과 결과 요약은 다음 단계에서 추가한다.

지금은 화면 전환이 정상적으로 되는지 확인하는 것이 목적이다.

### App에서 화면 결정하기

`App`은 `userNumber`와 `gameIsOver` 상태를 보고 어떤 화면을 보여줄지 결정한다.

처음에는 시작 화면을 보여준다.

```jsx
let screen = <StartGameScreen onPickNumber={pickedNumberHandler} />;
```

사용자가 숫자를 골랐고 게임이 진행 중이면 게임 화면을 보여준다.

```jsx
if (userNumber) {
  screen = (
    <GameScreen
      userNumber={userNumber}
      onGameOver={gameOverHandler}
    />
  );
}
```

게임이 끝났고 숫자도 선택된 상태라면 게임 오버 화면을 보여준다.

```jsx
if (gameIsOver && userNumber) {
  screen = <GameOverScreen />;
}
```

여기서 `userNumber` 조건이 중요하다.

`gameIsOver`는 처음부터 `true`이기 때문에, 이 조건만 보면 앱 시작 직후 게임 오버 화면이 표시된다.

```text
잘못된 조건
-> gameIsOver만 확인
-> 앱 시작 시 바로 GameOverScreen 표시

올바른 조건
-> gameIsOver && userNumber
-> 게임이 시작된 뒤 끝났을 때만 GameOverScreen 표시
```

### 게임 시작 시 gameIsOver를 false로 바꾸기

숫자를 선택해 게임을 시작하면 `userNumber`를 설정한다.

이때 `gameIsOver`도 `false`로 바꿔야 한다.

```jsx
function pickedNumberHandler(pickedNumber) {
  setUserNumber(pickedNumber);
  setGameIsOver(false);
}
```

이렇게 해야 숫자를 선택하자마자 게임 오버 화면으로 넘어가는 문제를 막을 수 있다.

```text
숫자 선택
-> userNumber 설정
-> gameIsOver false
-> GameScreen 표시
```

React는 같은 이벤트 처리 안에서 여러 상태 업데이트를 묶어 처리할 수 있다.

그래서 `setUserNumber`와 `setGameIsOver`가 함께 실행되어도 불필요한 중간 화면이 보이지 않는다.

### 게임 종료 핸들러 만들기

게임이 끝났을 때 호출할 함수를 `App`에 만든다.

```jsx
function gameOverHandler() {
  setGameIsOver(true);
}
```

이 함수는 `GameScreen`으로 전달된다.

```jsx
<GameScreen
  userNumber={userNumber}
  onGameOver={gameOverHandler}
/>
```

`GameScreen`은 이 함수를 prop으로 받아 `useEffect` 안에서 호출한다.

```jsx
function GameScreen({ userNumber, onGameOver }) {
  useEffect(() => {
    if (currentGuess === userNumber) {
      onGameOver();
    }
  }, [currentGuess, userNumber, onGameOver]);
}
```

이 구조는 자식에서 부모에게 이벤트를 올려 보내는 일반적인 React 패턴이다.

```text
GameScreen
-> 게임 종료 감지
-> onGameOver 호출

App
-> gameIsOver 상태 변경
-> GameOverScreen 렌더링
```

### App 전체 흐름 예시

이번 단계의 `App` 화면 결정 흐름은 다음처럼 정리할 수 있다.

```jsx
export default function App() {
  const [userNumber, setUserNumber] = useState(null);
  const [gameIsOver, setGameIsOver] = useState(true);

  function pickedNumberHandler(pickedNumber) {
    setUserNumber(pickedNumber);
    setGameIsOver(false);
  }

  function gameOverHandler() {
    setGameIsOver(true);
  }

  let screen = (
    <StartGameScreen onPickNumber={pickedNumberHandler} />
  );

  if (userNumber) {
    screen = (
      <GameScreen
        userNumber={userNumber}
        onGameOver={gameOverHandler}
      />
    );
  }

  if (gameIsOver && userNumber) {
    screen = <GameOverScreen />;
  }

  return (
    // 배경 레이아웃 안에서 {screen} 렌더링
  );
}
```

조건이 늘어나면 화면 전환 상태를 더 명확하게 관리하는 구조가 필요할 수 있다.

하지만 이번 앱에서는 `userNumber`와 `gameIsOver` 조합으로 충분하다.

### 여전히 남아 있던 무한 루프 문제

`useEffect`로 게임 종료를 처리했는데도, 정답에 도달할 때 최대 호출 스택 초과 오류가 다시 발생할 수 있다.

원인은 `initialGuess` 계산 위치에 있다.

```jsx
const initialGuess = generateRandomBetween(
  minBoundary,
  maxBoundary,
  userNumber
);
```

이 코드가 `GameScreen` 컴포넌트 함수 안에 있으면, 컴포넌트가 재렌더링될 때마다 다시 실행된다.

```text
setCurrentGuess 호출
-> GameScreen 재렌더링
-> initialGuess 다시 계산
-> generateRandomBetween 다시 실행
```

문제는 게임이 끝나는 순간에도 컴포넌트 함수는 한 번 더 실행될 수 있다는 점이다.

### useEffect는 렌더링 이후 실행된다

React에서 `useEffect`는 렌더링이 끝난 뒤 실행된다.

따라서 흐름은 다음과 같다.

```text
1. 버튼 클릭
2. currentGuess 상태 업데이트
3. GameScreen 함수 다시 실행
4. JSX 렌더링 완료
5. useEffect 실행
6. currentGuess === userNumber 확인
7. onGameOver 호출
```

즉 `useEffect`가 실행되기 전에 `GameScreen` 함수 본문은 이미 실행된다.

그래서 함수 본문 안에 있는 `initialGuess` 계산도 먼저 실행된다.

이때 `minBoundary`와 `maxBoundary`가 이미 좁아져 같은 값이 되었다면, `generateRandomBetween`이 유효한 숫자를 만들 수 없어 무한 재귀에 빠진다.

```text
useEffect는 너무 늦게 실행되는 것이 아니라,
렌더 이후 실행되는 것이 원래 동작이다.
```

이 타이밍을 이해하는 것이 중요하다.

### 초기 추측값 계산을 고정하기

해결책은 `initialGuess`를 매번 동적 경계값으로 계산하지 않도록 하는 것이다.

처음 추측값은 게임 시작 시 한 번만 필요하다.

따라서 초기 난수 계산은 고정된 범위로 처리한다.

```jsx
const initialGuess = generateRandomBetween(1, 100, userNumber);
```

이렇게 하면 `minBoundary`, `maxBoundary`가 이후에 어떻게 바뀌든 초기 추측 계산은 영향을 받지 않는다.

```text
초기 추측
-> 항상 1부터 99 사이에서 생성

다음 추측
-> minBoundary, maxBoundary를 사용해 범위 축소
```

강의에서는 빠르고 단순한 해결책으로 이 방식을 선택했다.

### useMemo도 대안이 될 수 있다

다른 해결책으로는 `useMemo`를 사용할 수도 있다.

`useMemo`는 특정 계산이 불필요하게 다시 실행되지 않도록 결과를 메모이제이션할 때 사용한다.

```jsx
const initialGuess = useMemo(() => {
  return generateRandomBetween(1, 100, userNumber);
}, [userNumber]);
```

이렇게 하면 `userNumber`가 바뀔 때만 초기 추측값을 다시 계산할 수 있다.

하지만 이번 단계에서는 하드 코딩된 초기 범위를 사용하는 편이 더 간단하다.

```text
가능한 해결책
-> 초기 범위를 고정해서 계산
-> useMemo로 초기 계산을 메모이제이션

강의 선택
-> 더 단순한 하드 코딩 방식
```

### GameScreen 전체 흐름 정리

이번 단계의 `GameScreen` 핵심은 다음과 같다.

```jsx
function GameScreen({ userNumber, onGameOver }) {
  const initialGuess = generateRandomBetween(1, 100, userNumber);
  const [currentGuess, setCurrentGuess] = useState(initialGuess);

  useEffect(() => {
    if (currentGuess === userNumber) {
      onGameOver();
    }
  }, [currentGuess, userNumber, onGameOver]);

  // nextGuessHandler ...
}
```

이 구조에서 `useEffect`는 현재 추측값이 바뀐 뒤 정답 여부를 확인한다.

정답이면 부모 컴포넌트에 게임 종료를 알린다.

### 프론트엔드 개발자 관점에서의 핵심 차이

이번 강의는 React Native보다 React 자체의 렌더링 모델을 이해하는 데 더 중요하다.

```text
컴포넌트 함수 본문
-> 렌더링 중 실행

useEffect
-> 렌더링 이후 실행

상태 업데이트
-> 컴포넌트 함수 재실행 유발
```

웹 React에서도 동일한 문제가 생길 수 있다.

예를 들어 컴포넌트 본문에서 매번 무거운 계산이나 랜덤 계산을 수행하면 상태 업데이트 때마다 다시 실행된다.

```jsx
function Component() {
  const value = expensiveOrRandomCalculation();
  const [state, setState] = useState(value);
}
```

이런 계산이 “처음 한 번만 필요”하다면 lazy initializer나 `useMemo`를 고려할 수 있다.

```jsx
const [state] = useState(() => expensiveOrRandomCalculation());
```

또는 다음처럼 쓸 수 있다.

```jsx
const value = useMemo(() => expensiveOrRandomCalculation(), []);
```

React Native에서도 이 원리는 그대로 적용된다.

이번 버그는 모바일 특화 문제가 아니라 React 렌더링 타이밍을 이해해야 해결되는 문제다.

## 정리

이번 강의에서는 장치의 현재 추측값이 사용자가 선택한 숫자와 같아졌을 때 게임을 종료하는 로직을 추가했다. `GameScreen`에서는 `useEffect`로 `currentGuess`와 `userNumber`를 비교하고, 정답이면 `onGameOver` 콜백을 호출한다.

`App`에서는 `gameIsOver` 상태를 추가해 시작 화면, 게임 화면, 게임 오버 화면을 조건부 렌더링한다. 게임이 시작될 때는 `gameIsOver`를 `false`로 바꾸고, 게임 종료 시에는 `true`로 바꾼다.

또한 `useEffect`가 렌더링 이후 실행된다는 점 때문에 `initialGuess` 계산이 예상보다 한 번 더 실행되며 무한 재귀 문제가 생길 수 있음을 확인했다. 이를 해결하기 위해 초기 추측값은 고정 범위 `1`부터 `100` 사이에서 계산하도록 정리했다.
