# 30. 라운드 기록 상태와 추측 로그 출력

## 핵심 요약

- 게임 화면에서 장치가 추측한 모든 숫자를 라운드 로그로 기록한다.
- 라운드 기록은 여러 값을 담아야 하므로 배열 상태로 관리한다.
- `guessRounds` 상태의 초깃값은 첫 추측값인 `initialGuess`를 담은 배열이다.
- 새 추측값이 생성될 때마다 `setGuessRounds`로 기존 배열에 `newRndNumber`를 추가한다.
- 이전 상태를 기반으로 상태를 업데이트하므로 상태 업데이트 함수의 함수형 형태를 사용한다.
- 로그는 `guessRounds.map()`으로 `Text` 목록으로 출력할 수 있다.
- 목록에는 고유한 `key`가 필요하며, 이 게임에서는 같은 숫자를 다시 추측하지 않으므로 추측 숫자를 key로 사용할 수 있다.
- 목록이 아주 길어질 가능성이 작다면 `map` 방식도 충분하지만, 더 일반적인 목록 렌더링에는 `FlatList`가 더 적합하다.

## 학습 목표

- 여러 라운드 값을 배열 상태로 관리하는 방법을 익힌다.
- `setState(prevState => nextState)` 형태의 함수형 업데이트가 필요한 이유를 이해한다.
- 배열 앞쪽에 새 값을 추가해 최신 로그가 위에 오도록 만드는 방법을 배운다.
- `map()`으로 React Native의 `Text` 목록을 렌더링하는 흐름을 복습한다.
- 목록 렌더링에서 `key`가 필요한 이유와 이 예제에서 추측값을 key로 쓸 수 있는 이유를 이해한다.
- `map`과 `FlatList`를 선택하는 기준을 구분한다.

## 주요 내용

### 라운드 기록이 필요한 이유

현재 게임은 장치가 새 숫자를 추측할 때마다 화면의 현재 추측값만 바뀐다.

하지만 게임 화면에는 지금까지 어떤 숫자를 추측했는지 기록도 보여주고 싶다.

```text
현재 추측값
-> NumberContainer에 표시

추측 로그
-> 이전 라운드까지 포함한 목록 표시
```

예를 들어 장치가 다음 순서로 추측했다면 로그에도 이 값들이 남아야 한다.

```text
14
63
42
31
```

이런 값은 하나가 아니라 여러 개이므로 배열로 관리하는 것이 자연스럽다.

### GameScreen에 guessRounds 상태 추가하기

`GameScreen`에 새 상태를 추가한다.

```jsx
const [guessRounds, setGuessRounds] = useState([initialGuess]);
```

초깃값은 빈 배열이 아니라 `[initialGuess]`다.

게임이 시작되면 이미 첫 번째 추측값이 존재하기 때문이다.

```text
initialGuess
-> 게임 시작 시 장치의 첫 추측

guessRounds 초기값
-> [initialGuess]
```

이렇게 하면 첫 번째 추측도 로그에 바로 포함된다.

### 새 추측값을 로그에 추가하기

새로운 추측값은 `nextGuessHandler` 안에서 생성된다.

```jsx
const newRndNumber = generateRandomBetween(
  minBoundary,
  maxBoundary,
  currentGuess
);
```

이 값은 현재 추측값 상태에도 반영된다.

```jsx
setCurrentGuess(newRndNumber);
```

동시에 라운드 로그에도 추가해야 한다.

```jsx
setGuessRounds((prevGuessRounds) => [
  newRndNumber,
  ...prevGuessRounds,
]);
```

새로운 값을 배열 앞쪽에 넣으면 최신 라운드가 목록 상단에 표시된다.

```text
기존 로그
-> [14]

새 추측값
-> 63

업데이트 후
-> [63, 14]
```

### 왜 함수형 상태 업데이트를 사용하는가

새 `guessRounds` 상태는 이전 `guessRounds` 상태를 기반으로 만들어진다.

이럴 때는 상태 업데이트 함수에 새 값을 직접 넣기보다, 함수를 전달하는 방식이 좋다.

```jsx
setGuessRounds((prevGuessRounds) => [
  newRndNumber,
  ...prevGuessRounds,
]);
```

React는 상태 업데이트를 비동기적으로 처리할 수 있다.

따라서 이전 상태를 기반으로 다음 상태를 만들 때는 함수형 업데이트가 더 안전하다.

```text
직접 업데이트
-> 현재 렌더링 시점의 상태에 의존

함수형 업데이트
-> React가 제공하는 최신 이전 상태를 기반으로 계산
```

이 패턴은 React 웹에서도 동일하게 중요하다.

### 배열 앞에 추가할지 뒤에 추가할지

새 라운드를 배열 끝에 추가할 수도 있다.

```jsx
setGuessRounds((prevGuessRounds) => [
  ...prevGuessRounds,
  newRndNumber,
]);
```

하지만 강의에서는 배열 앞쪽에 추가한다.

```jsx
setGuessRounds((prevGuessRounds) => [
  newRndNumber,
  ...prevGuessRounds,
]);
```

이렇게 하면 최신 라운드가 화면에서 가장 위에 보인다.

```text
앞에 추가
-> 최신 로그가 상단

뒤에 추가
-> 오래된 로그가 상단
```

어떤 방식이 더 좋은지는 앱의 UX에 따라 달라진다.

이 게임에서는 최신 추측값을 바로 보고 싶으므로 앞에 추가하는 방식이 적절하다.

### 주의할 실수: 배열 자체를 다시 넣지 않기

강의 중 작은 실수가 있었다.

새 배열에 추가해야 하는 값은 `guessRounds` 배열 자체가 아니라 `newRndNumber`다.

잘못된 예:

```jsx
setGuessRounds((prevGuessRounds) => [
  guessRounds,
  ...prevGuessRounds,
]);
```

올바른 예:

```jsx
setGuessRounds((prevGuessRounds) => [
  newRndNumber,
  ...prevGuessRounds,
]);
```

잘못 작성하면 배열 안에 숫자가 아니라 배열이 들어가 로그 출력이 이상해질 수 있다.

```text
원하는 형태
-> [63, 14]

잘못된 형태
-> [[14], 14]
```

상태가 배열일 때는 새로 추가하려는 값이 무엇인지 정확히 확인해야 한다.

### 로그를 화면에 출력하기

라운드 로그는 `guessRounds` 배열을 JSX 목록으로 변환해 출력한다.

React에서는 배열의 `map()` 메서드를 사용한다.

```jsx
<View>
  {guessRounds.map((guessRound) => (
    <Text key={guessRound}>{guessRound}</Text>
  ))}
</View>
```

`Text` 컴포넌트를 사용하므로 `react-native`에서 import해야 한다.

```jsx
import { Text } from 'react-native';
```

React Native에서는 일반 문자열을 `View` 안에 직접 넣을 수 없으므로, 반드시 `Text`로 감싸야 한다.

### key가 필요한 이유

리스트를 렌더링할 때 React는 각 항목을 구분하기 위해 `key`가 필요하다.

```jsx
<Text key={guessRound}>{guessRound}</Text>
```

이 앱에서는 `guessRound` 자체를 key로 사용할 수 있다.

이유는 같은 숫자를 다시 추측하지 않도록 로직을 작성했기 때문이다.

```text
같은 숫자 재추측 방지
-> guessRound 값이 중복되지 않음
-> key로 사용 가능
```

다만 일반적인 앱에서는 숫자나 텍스트 값이 중복될 수 있으므로 id를 사용하는 것이 더 안전하다.

```text
일반적인 목록
-> item.id 사용 권장

이번 게임 로그
-> 추측 숫자가 중복되지 않으므로 guessRound 사용 가능
```

### map으로 렌더링해도 되는 이유

이전에 긴 목록은 `FlatList`를 사용하는 것이 좋다고 배웠다.

하지만 이 게임의 라운드 수는 많아야 8~10번 정도일 가능성이 높다.

```text
추측 가능한 숫자 범위
-> 1부터 99

범위를 계속 좁힘
-> 많은 라운드가 필요하지 않음

예상 로그 수
-> 대략 8~10개 수준
```

이 정도로 제한된 목록이라면 `map()`으로 직접 렌더링해도 성능 문제가 크지 않다.

```text
항목 수가 적고 제한적
-> map 사용 가능

항목 수가 많거나 동적 확장 가능
-> FlatList 권장
```

### FlatList를 사용할 수도 있다

더 일반적인 React Native 목록 렌더링 방식은 `FlatList`다.

```jsx
import { FlatList } from 'react-native';
```

`FlatList`는 화면에 보이는 항목 중심으로 렌더링하므로 긴 목록에서 효율적이다.

이번 게임 로그는 길어질 가능성이 낮아 `map()`도 충분하지만, `FlatList`로 바꾸는 연습을 해보는 것도 좋다.

```text
이번 강의
-> map으로 간단히 로그 출력

다음 흐름
-> FlatList로 목록 렌더링
-> 로그 항목 스타일링
```

### GameScreen 흐름 예시

이번 단계의 핵심 코드는 다음처럼 정리할 수 있다.

```jsx
function GameScreen({ userNumber, onGameOver }) {
  const initialGuess = generateRandomBetween(1, 100, userNumber);
  const [currentGuess, setCurrentGuess] = useState(initialGuess);
  const [guessRounds, setGuessRounds] = useState([initialGuess]);

  function nextGuessHandler(direction) {
    // boundary 업데이트

    const newRndNumber = generateRandomBetween(
      minBoundary,
      maxBoundary,
      currentGuess
    );

    setCurrentGuess(newRndNumber);
    setGuessRounds((prevGuessRounds) => [
      newRndNumber,
      ...prevGuessRounds,
    ]);
  }

  return (
    <View>
      {/* 현재 추측값 */}

      <View>
        {guessRounds.map((guessRound) => (
          <Text key={guessRound}>{guessRound}</Text>
        ))}
      </View>
    </View>
  );
}
```

이제 화면 하단에 장치가 지금까지 추측한 숫자들이 출력된다.

### 현재 단계에서 아직 남은 일

이번 강의에서는 라운드 로그를 화면에 출력했다.

하지만 아직 개선할 부분이 남아 있다.

```text
남은 작업
-> 로그 항목을 보기 좋게 스타일링
-> 필요하다면 FlatList로 변경
-> 라운드 수를 GameOverScreen에 실제 값으로 전달
```

이 중 `FlatList`와 로그 항목 스타일링은 다음 강의에서 이어질 예정이다.

### 프론트엔드 개발자 관점에서의 핵심 차이

웹 React 개발자라면 이번 내용은 익숙한 배열 상태와 리스트 렌더링이다.

```jsx
const [items, setItems] = useState([initialItem]);

setItems((prevItems) => [newItem, ...prevItems]);

items.map((item) => (
  <div key={item}>{item}</div>
));
```

React Native에서는 `div` 대신 `Text` 또는 `View`를 사용한다.

```jsx
guessRounds.map((guessRound) => (
  <Text key={guessRound}>{guessRound}</Text>
));
```

개념은 같다.

```text
React 웹
-> 배열 상태
-> map 렌더링
-> key 필요

React Native
-> 배열 상태
-> map 렌더링
-> key 필요
-> 텍스트는 Text 안에 렌더링
```

목록이 길어질 때 React Native는 `FlatList`를 적극적으로 고려한다는 점이 차이다.

### 실무 관점에서의 선택 기준

`map()`과 `FlatList` 선택은 목록의 성격에 따라 결정한다.

```text
map이 적합한 경우
-> 항목 수가 작고 고정적
-> 레이아웃이 단순함
-> 성능 부담이 거의 없음

FlatList가 적합한 경우
-> 항목 수가 많음
-> 스크롤 목록
-> 서버 데이터 목록
-> 계속 추가될 가능성이 있음
```

이번 라운드 로그는 짧은 목록이므로 `map()`도 충분하지만, React Native 목록 처리 연습을 위해 `FlatList`로 바꾸는 것도 좋은 학습이다.

## 정리

이번 강의에서는 장치가 추측한 모든 숫자를 `guessRounds` 배열 상태로 기록했다. 첫 라운드는 `initialGuess`를 담은 배열로 시작하고, 새 추측값이 나올 때마다 `setGuessRounds`의 함수형 업데이트를 사용해 `newRndNumber`를 배열 앞쪽에 추가했다.

또한 `guessRounds.map()`으로 로그를 화면에 출력했다. 각 항목은 `Text`로 감싸고, 같은 숫자를 다시 추측하지 않는 게임 로직 덕분에 `guessRound` 값을 `key`로 사용할 수 있다.

이번 목록은 길어질 가능성이 작기 때문에 `map()` 방식도 충분하지만, 더 일반적인 React Native 목록 렌더링에는 `FlatList`가 적합하다. 다음 단계에서는 로그 목록을 더 좋은 방식으로 렌더링하고 스타일링하는 작업을 이어가게 된다.
