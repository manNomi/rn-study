# 8. MealDetailScreen에서 Redux 사용하기

## 핵심 요약

- 이제 `MealDetailScreen`에서 Context 대신 Redux store를 사용한다.
- Context 관련 import와 `useContext(FavoritesContext)` 사용을 제거한다.
- Redux store에서 데이터를 읽을 때는 `react-redux`의 `useSelector` 훅을 사용한다.
- `useSelector((state) => state.favoriteMeals.ids)`로 즐겨찾기 meal id 배열을 읽는다.
- 현재 `mealId`가 favorite id 배열에 포함되어 있는지 `includes()`로 확인해 favorite 여부를 계산한다.
- Redux store에 변경을 요청할 때는 `useDispatch` 훅으로 `dispatch` 함수를 가져온다.
- `addFavorite`와 `removeFavorite` action creator를 import하고, `dispatch(addFavorite({ id: mealId }))`처럼 호출한다.
- Redux 방식에서는 Context 함수 호출과 달리 action을 dispatch해서 reducer를 실행한다.
- 이 변경 후에도 헤더의 `star`/`star-outline` 아이콘 토글은 Context 방식과 동일하게 동작한다.

## 학습 목표

- React Redux의 `useSelector`로 store 데이터를 읽는 방법을 익힌다.
- store에 등록한 reducer key가 selector 경로에 어떻게 반영되는지 이해한다.
- `useDispatch`로 action을 dispatch하는 방법을 익힌다.
- Redux Toolkit slice에서 export한 action creator를 화면 컴포넌트에서 사용하는 흐름을 이해한다.
- Context 기반 상태 변경 코드와 Redux 기반 상태 변경 코드의 차이를 비교한다.

## 주요 내용

### Context 코드 제거

이전 Context 방식에서는 `MealDetailScreen`에서 다음 코드를 사용했다.

```jsx
const favoriteMealsContext = useContext(FavoritesContext);
```

그리고 Context에서 상태와 함수를 꺼냈다.

```jsx
favoriteMealsContext.ids
favoriteMealsContext.addFavorite(mealId)
favoriteMealsContext.removeFavorite(mealId)
```

Redux 방식으로 바꾸면서 이 Context 코드는 더 이상 필요하지 않다.

따라서 다음 import들도 제거하거나 주석 처리한다.

```jsx
import { useContext } from 'react';
import { FavoritesContext } from '../store/context/favorites-context';
```

이제 상태 읽기와 상태 변경은 React Redux 훅으로 처리한다.

### useSelector로 Redux state 읽기

Redux store에서 데이터를 읽으려면 `useSelector`를 사용한다.

```jsx
import { useSelector } from 'react-redux';
```

`useSelector`에는 selector 함수를 전달한다.

```jsx
const favoriteMealIds = useSelector((state) => {
  return state.favoriteMeals.ids;
});
```

이 함수는 React Redux가 실행해 준다.

인자로 들어오는 `state`는 전체 Redux store 상태다.

```text
state
-> Redux store의 전체 상태 객체
```

### state.favoriteMeals.ids 경로

이전 강의에서 `store.js`에 favorites reducer를 다음처럼 등록했다.

```jsx
const store = configureStore({
  reducer: {
    favoriteMeals: favoritesReducer,
  },
});
```

따라서 Redux state 구조는 다음과 같다.

```jsx
{
  favoriteMeals: {
    ids: []
  }
}
```

그래서 favorite id 배열은 다음 경로로 읽는다.

```jsx
state.favoriteMeals.ids
```

```text
favoriteMeals
-> store.js에서 reducer를 등록한 key

ids
-> favorites slice의 initialState에 있는 property
```

store 구조와 selector 경로는 서로 직접 연결된다.

### 현재 meal이 favorite인지 계산하기

Context 방식과 마찬가지로 현재 화면의 `mealId`는 route params에서 가져온다.

```jsx
const mealId = route.params.mealId;
```

그리고 Redux store에서 읽은 `favoriteMealIds` 배열에 이 id가 있는지 확인한다.

```jsx
const mealIsFavorite = favoriteMealIds.includes(mealId);
```

`includes()`는 배열 안에 해당 id가 있으면 `true`, 없으면 `false`를 반환한다.

```text
favoriteMealIds.includes(mealId)
-> 현재 meal이 favorite이면 true
-> 아니면 false
```

이 값은 헤더 아이콘을 결정하는 데 그대로 사용할 수 있다.

### useDispatch로 action dispatch 준비

Redux store의 상태를 바꾸려면 action을 dispatch해야 한다.

이를 위해 `useDispatch`를 가져온다.

```jsx
import { useDispatch, useSelector } from 'react-redux';
```

컴포넌트 안에서 호출하면 dispatch 함수를 얻는다.

```jsx
const dispatch = useDispatch();
```

```text
dispatch
-> action을 Redux store에 보내는 함수
-> 해당 action에 맞는 reducer 실행
```

Context 방식에서는 상태 변경 함수를 직접 호출했다.

Redux 방식에서는 action creator로 action을 만들고 dispatch한다.

### action creator import

이전 강의에서 `favorites.js`에서 action creator를 export했다.

```jsx
export const addFavorite = favoritesSlice.actions.addFavorite;
export const removeFavorite = favoritesSlice.actions.removeFavorite;
```

이제 `MealDetailScreen`에서 가져온다.

```jsx
import {
  addFavorite,
  removeFavorite,
} from '../store/redux/favorites';
```

이 함수들은 reducer를 직접 실행하는 함수가 아니다.

action 객체를 만들어 주는 함수다.

```text
addFavorite({ id: mealId })
-> action object 생성

dispatch(addFavorite({ id: mealId }))
-> action을 store에 전달
-> reducer 실행
```

### action payload 전달

`favorites.js`의 reducer는 `action.payload.id`를 읽도록 작성했다.

```jsx
state.ids.push(action.payload.id);
```

따라서 action creator를 호출할 때 payload 객체에 `id`를 넣어야 한다.

```jsx
dispatch(addFavorite({ id: mealId }));
```

삭제할 때도 같은 형태를 사용한다.

```jsx
dispatch(removeFavorite({ id: mealId }));
```

```text
payload
-> { id: mealId }

reducer 내부
-> action.payload.id
```

payload 구조를 reducer와 dispatch 쪽에서 맞춰야 한다.

### 즐겨찾기 토글 함수

Context 방식에서 `changeFavoriteStatusHandler`는 Context 함수를 호출했다.

Redux 방식에서는 `dispatch()`를 호출한다.

```jsx
function changeFavoriteStatusHandler() {
  if (mealIsFavorite) {
    dispatch(removeFavorite({ id: mealId }));
  } else {
    dispatch(addFavorite({ id: mealId }));
  }
}
```

흐름은 다음과 같다.

```text
현재 favorite
-> dispatch(removeFavorite({ id: mealId }))
-> removeFavorite reducer 실행
-> ids에서 mealId 제거

현재 not favorite
-> dispatch(addFavorite({ id: mealId }))
-> addFavorite reducer 실행
-> ids에 mealId 추가
```

### 헤더 아이콘은 그대로 유지

헤더 아이콘 분기 로직은 Context 방식과 거의 같다.

```jsx
<IconButton
  icon={mealIsFavorite ? 'star' : 'star-outline'}
  color="white"
  onPress={changeFavoriteStatusHandler}
/>
```

달라진 것은 `mealIsFavorite`을 계산하는 데이터 출처다.

```text
Context 방식
-> FavoritesContext.ids

Redux 방식
-> state.favoriteMeals.ids
```

UI는 같은데 상태 공급원이 바뀐 것이다.

### 전체 동작 흐름

Redux 방식에서 별 아이콘을 누르면 다음 흐름으로 동작한다.

```text
1. IconButton onPress 실행
2. changeFavoriteStatusHandler 호출
3. mealIsFavorite 확인
4. addFavorite 또는 removeFavorite action creator 호출
5. dispatch(action)
6. favorites reducer 실행
7. Redux store의 favoriteMeals.ids 업데이트
8. useSelector가 새 상태를 감지
9. MealDetailScreen 리렌더링
10. star / star-outline 아이콘 갱신
```

Context 방식과 결과는 같지만, 상태 변경 흐름은 action dispatch 중심으로 바뀐다.

## 예시 코드

### MealDetailScreen 핵심 흐름

```jsx
import { useLayoutEffect } from 'react';
import { useDispatch, useSelector } from 'react-redux';

import IconButton from '../components/IconButton';
import {
  addFavorite,
  removeFavorite,
} from '../store/redux/favorites';

function MealDetailScreen({ route, navigation }) {
  const mealId = route.params.mealId;

  const favoriteMealIds = useSelector(
    (state) => state.favoriteMeals.ids
  );

  const dispatch = useDispatch();

  const mealIsFavorite = favoriteMealIds.includes(mealId);

  function changeFavoriteStatusHandler() {
    if (mealIsFavorite) {
      dispatch(removeFavorite({ id: mealId }));
    } else {
      dispatch(addFavorite({ id: mealId }));
    }
  }

  useLayoutEffect(() => {
    navigation.setOptions({
      headerRight: () => (
        <IconButton
          icon={mealIsFavorite ? 'star' : 'star-outline'}
          color="white"
          onPress={changeFavoriteStatusHandler}
        />
      ),
    });
  }, [navigation, changeFavoriteStatusHandler, mealIsFavorite]);

  // screen content...
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### useSelector는 Redux store의 구독 지점이다

`useSelector`는 단순히 값을 한 번 읽는 것이 아니다.

선택한 store 값이 바뀌면 컴포넌트를 다시 렌더링하게 해 준다.

```jsx
const favoriteMealIds = useSelector(
  (state) => state.favoriteMeals.ids
);
```

`favoriteMeals.ids`가 바뀌면 `MealDetailScreen`도 새 favorite 여부를 계산한다.

### dispatch는 상태 변경 요청이다

Context 방식에서는 함수를 직접 호출해 상태를 바꿨다.

Redux에서는 상태 변경을 action으로 표현하고 dispatch한다.

```jsx
dispatch(addFavorite({ id: mealId }));
```

이 구조는 상태 변경 흐름을 더 명시적으로 만든다.

```text
사용자 이벤트
-> action dispatch
-> reducer
-> store 업데이트
-> selector를 사용하는 컴포넌트 리렌더링
```

### payload 구조는 reducer와 맞아야 한다

reducer가 `action.payload.id`를 읽는다면 dispatch할 때도 `{ id: mealId }` 형태로 보내야 한다.

```jsx
dispatch(addFavorite({ id: mealId }));
```

만약 문자열만 보내면 reducer 코드도 그에 맞게 바꿔야 한다.

```jsx
dispatch(addFavorite(mealId));
// reducer에서는 action.payload 사용
```

중요한 것은 action payload의 구조를 일관되게 유지하는 것이다.

### Context와 Redux의 UI 코드는 비슷할 수 있다

상태 관리 방식은 달라졌지만, 화면 UI는 거의 같다.

```text
mealIsFavorite 계산
-> IconButton icon 결정
-> onPress로 토글
```

달라진 것은 `mealIsFavorite`을 어디서 계산하고, 상태 변경을 어떤 방식으로 요청하느냐다.

```text
Context
-> useContext
-> context.addFavorite()

Redux
-> useSelector
-> useDispatch
-> dispatch(addFavorite())
```

## 정리

이번 강의에서는 `MealDetailScreen`에서 Context 대신 Redux store를 사용하도록 변경했다.

`useSelector`로 `state.favoriteMeals.ids`를 읽어 현재 meal이 즐겨찾기인지 계산하고, `useDispatch`로 dispatch 함수를 가져와 `addFavorite` 또는 `removeFavorite` action을 dispatch했다.

그 결과 Context 방식과 동일하게 헤더의 별 아이콘이 `star`와 `star-outline` 사이에서 토글되고, Redux store의 favorite ids 상태가 업데이트된다.

다음 단계에서는 `FavoritesScreen`에서도 Redux store를 읽어 즐겨찾기 meal 목록을 표시한다.
