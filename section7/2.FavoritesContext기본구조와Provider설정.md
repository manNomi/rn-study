# 2. FavoritesContext 기본 구조와 Provider 설정

## 핵심 요약

- 이번 섹션에서는 앱 전역 상태 관리를 위해 Context API와 Redux를 차례로 살펴본다.
- 일반적으로는 Context와 Redux 중 하나만 선택해 앱 전역 상태를 관리한다.
- 먼저 React Context API 방식부터 구현한다.
- 앱 전역 상태 관련 코드는 보통 `store` 폴더에 모아 둔다.
- 이번 프로젝트에서는 `store/context`와 `store/redux` 폴더를 만들고, 먼저 `store/context/favorites-context.js`를 작성한다.
- `createContext()`로 즐겨찾기 상태를 위한 Context 객체를 만든다.
- Context 기본값에는 `ids`, `addFavorite`, `removeFavorite`를 넣어 나중에 자동완성과 사용 형태를 돕는다.
- `FavoritesContextProvider` 컴포넌트를 만들고, 내부에서 `FavoritesContext.Provider`로 `children`을 감싼다.
- `App.js`에서는 `NavigationContainer` 전체를 `FavoritesContextProvider`로 감싸 앱의 여러 화면에서 즐겨찾기 Context에 접근할 수 있게 한다.
- 아직 실제 상태 변경 로직은 구현하지 않았기 때문에 다음 단계에서 Provider 내부 로직을 채워야 한다.

## 학습 목표

- 앱 전역 상태 관련 코드를 어디에 둘지 폴더 구조 관점에서 이해한다.
- `createContext()`로 Context 객체를 만드는 방법을 복습한다.
- Context 기본값이 실제 동작보다 개발 경험과 사용 계약에 도움을 줄 수 있음을 이해한다.
- Provider 컴포넌트를 직접 만들어 앱을 감싸는 패턴을 익힌다.
- 여러 네비게이터와 화면이 Context를 사용할 수 있도록 Provider를 어디에 배치해야 하는지 이해한다.

## 주요 내용

### Context와 Redux를 모두 살펴보는 이유

앱 전역 상태를 관리할 때 Context API와 Redux를 동시에 사용할 필요는 보통 없다.

대부분의 프로젝트에서는 상태의 성격과 앱 규모에 따라 하나의 접근 방식을 선택한다.

```text
Context API
-> React 내장 기능
-> 비교적 단순한 전역 상태에 적합

Redux
-> 별도 라이브러리
-> 명시적인 상태 변경 흐름과 도구 생태계가 강함
```

강의에서는 두 방식을 모두 경험하기 위해 Context API를 먼저 구현하고, 이후 Redux 방식도 살펴본다.

### store 폴더 만들기

앱 전역 상태 관련 로직은 프로젝트의 여러 화면과 컴포넌트에서 사용된다.

따라서 화면 폴더나 UI 컴포넌트 폴더 안에 섞어 두기보다 별도의 폴더에 모아 두는 편이 좋다.

강의에서는 `store`라는 이름의 폴더를 사용한다.

```text
store/
  context/
  redux/
```

`store`라는 이름이 필수는 아니지만, 앱 전역 상태를 담는 위치로 자주 사용되는 관례다.

이번에는 먼저 `context` 폴더에서 작업한다.

```text
store/
  context/
    favorites-context.js
  redux/
```

`redux` 폴더는 이후 Redux 방식으로 같은 기능을 구현할 때 사용한다.

### favorites-context.js 파일

즐겨찾기 상태를 위한 Context는 `favorites-context.js` 파일에 작성한다.

이 파일의 책임은 크게 두 가지다.

```text
1. FavoritesContext 객체 생성
2. FavoritesContextProvider 컴포넌트 제공
```

Context 객체는 다른 컴포넌트에서 Context 값을 읽을 때 필요하다.

Provider 컴포넌트는 앱을 감싸고 실제 Context 값을 제공하는 역할을 한다.

### createContext로 Context 생성

React에서 Context 객체를 만들려면 `createContext`를 사용한다.

```jsx
import { createContext } from 'react';
```

```jsx
export const FavoritesContext = createContext({
  ids: [],
  addFavorite: (id) => {},
  removeFavorite: (id) => {},
});
```

여기서 `FavoritesContext`는 대문자로 시작한다.

이 Context 객체의 `Provider`를 JSX에서 컴포넌트처럼 사용할 것이기 때문이다.

```jsx
<FavoritesContext.Provider>
  ...
</FavoritesContext.Provider>
```

React 컴포넌트처럼 JSX에서 사용하는 값은 대문자로 시작해야 한다.

### Context 기본값의 역할

`createContext()`에 전달한 객체는 Context의 기본값이다.

```jsx
{
  ids: [],
  addFavorite: (id) => {},
  removeFavorite: (id) => {},
}
```

강의에서는 이 기본값이 실제 동작을 위해 꼭 필요한 것은 아니라고 설명한다.

Provider가 나중에 실제 값을 제공할 것이기 때문이다.

하지만 기본값을 넣어 두면 다음 장점이 있다.

```text
Context가 어떤 형태의 값을 제공하는지 한눈에 알 수 있다.
IDE 자동완성에 도움이 된다.
Context를 사용하는 컴포넌트에서 기대할 수 있는 API가 명확해진다.
```

즉 기본값은 일종의 사용 계약처럼 작동한다.

### ids

`ids`는 즐겨찾기로 저장된 meal id 목록이다.

```jsx
ids: []
```

실제 구현에서는 예를 들어 다음과 같은 배열이 될 수 있다.

```jsx
['m1', 'm3']
```

이 배열은 `FavoritesScreen`에서 즐겨찾기 meal 목록을 필터링하는 데 사용될 수 있다.

또한 `MealDetailScreen`에서는 현재 meal id가 이 배열 안에 있는지 확인해 별 아이콘의 상태를 결정할 수 있다.

### addFavorite와 removeFavorite

즐겨찾기 상태를 바꾸기 위한 함수도 Context 값에 포함한다.

```jsx
addFavorite: (id) => {},
removeFavorite: (id) => {},
```

현재는 빈 함수다.

아직 실제 구현이 아니라 Context가 제공할 함수의 형태만 정의한 것이다.

```text
addFavorite(id)
-> 특정 meal id를 즐겨찾기에 추가

removeFavorite(id)
-> 특정 meal id를 즐겨찾기에서 제거
```

실제 로직은 다음 강의에서 Provider 컴포넌트 내부에 구현한다.

### Provider 컴포넌트 만들기

Context 객체만 만들면 아직 앱에 값이 제공되지 않는다.

Context 값을 앱에 공급하려면 Provider가 필요하다.

강의에서는 별도의 Provider 컴포넌트를 만든다.

```jsx
function FavoritesContextProvider({ children }) {
  return (
    <FavoritesContext.Provider>
      {children}
    </FavoritesContext.Provider>
  );
}

export default FavoritesContextProvider;
```

`children`은 이 Provider로 감싼 하위 컴포넌트들이다.

```text
FavoritesContextProvider
-> FavoritesContext.Provider 렌더링
-> children을 내부에 배치
-> 하위 컴포넌트들이 Context에 접근 가능
```

이 패턴을 사용하면 `App.js`에서 간단히 앱 전체를 감쌀 수 있다.

### value prop은 아직 없다

위 코드에는 중요한 부분이 아직 빠져 있다.

```jsx
<FavoritesContext.Provider>
  {children}
</FavoritesContext.Provider>
```

`Provider`에는 실제로 제공할 값을 `value` prop으로 전달해야 한다.

```jsx
<FavoritesContext.Provider value={...}>
  {children}
</FavoritesContext.Provider>
```

이번 강의에서는 아직 Provider의 껍데기만 만들기 때문에, 이 상태로는 오류가 발생할 수 있다.

다음 강의에서 실제 상태와 함수를 만들고 `value`에 연결한다.

### App.js에서 Provider로 감싸기

Context가 여러 화면에서 필요하므로 Provider는 충분히 높은 위치에 있어야 한다.

이번 앱에서는 `MealDetailScreen`과 `FavoritesScreen` 모두 즐겨찾기 Context에 접근해야 한다.

```text
MealDetailScreen
-> 즐겨찾기 추가/삭제

FavoritesScreen
-> 즐겨찾기 목록 읽기
```

두 화면은 서로 다른 네비게이터 흐름에 포함되어 있다.

따라서 `NavigationContainer` 전체를 Provider로 감싸는 방식이 가장 간단하다.

```jsx
import FavoritesContextProvider from './store/context/favorites-context';
```

```jsx
export default function App() {
  return (
    <FavoritesContextProvider>
      <NavigationContainer>
        {/* navigators and screens */}
      </NavigationContainer>
    </FavoritesContextProvider>
  );
}
```

이렇게 하면 NavigationContainer 안의 모든 네비게이터와 모든 화면이 Context에 접근할 수 있다.

### Provider 위치의 의미

Provider는 그 아래에 있는 컴포넌트들에게만 값을 제공한다.

```text
FavoritesContextProvider
  -> NavigationContainer
     -> Stack Navigator
        -> Drawer Navigator
           -> FavoritesScreen
        -> MealDetailScreen
```

이 구조에서는 `FavoritesScreen`과 `MealDetailScreen` 모두 Provider의 하위에 있다.

따라서 나중에 `useContext(FavoritesContext)`를 사용해 같은 즐겨찾기 상태에 접근할 수 있다.

Provider를 너무 낮은 위치에 두면 일부 화면이 Context에 접근하지 못할 수 있다.

앱 전역 상태라면 보통 `App.js`의 높은 위치에 Provider를 두는 것이 자연스럽다.

## 예시 코드

### store/context/favorites-context.js

```jsx
import { createContext } from 'react';

export const FavoritesContext = createContext({
  ids: [],
  addFavorite: (id) => {},
  removeFavorite: (id) => {},
});

function FavoritesContextProvider({ children }) {
  return (
    <FavoritesContext.Provider>
      {children}
    </FavoritesContext.Provider>
  );
}

export default FavoritesContextProvider;
```

### App.js에서 Provider 적용

```jsx
import FavoritesContextProvider from './store/context/favorites-context';

export default function App() {
  return (
    <FavoritesContextProvider>
      <NavigationContainer>
        {/* navigators */}
      </NavigationContainer>
    </FavoritesContextProvider>
  );
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### Context 파일은 상태의 공개 API를 정의한다

`favorites-context.js`는 단순히 상태를 담는 파일이 아니다.

앱의 다른 화면들이 즐겨찾기 상태를 어떻게 사용할 수 있는지 정의하는 공개 API에 가깝다.

```text
ids
-> 현재 즐겨찾기 목록 읽기

addFavorite(id)
-> 즐겨찾기 추가

removeFavorite(id)
-> 즐겨찾기 제거
```

컴포넌트들은 내부 구현을 알 필요 없이 이 API만 사용하면 된다.

### Provider는 앱의 상태 경계다

Provider는 Context 값이 유효한 범위를 결정한다.

웹 React 앱에서도 `ThemeProvider`, `AuthProvider`, `QueryClientProvider` 등을 앱 루트 근처에 두는 것과 비슷하다.

React Native에서도 원리는 같다.

```text
Provider 아래
-> Context 사용 가능

Provider 밖
-> Context 사용 불가 또는 기본값만 사용
```

Navigation을 사용하는 앱에서는 여러 화면이 Provider 아래에 있도록 배치하는 것이 중요하다.

### NavigationContainer와 Provider의 관계

이번 강의에서는 Provider가 `NavigationContainer`를 감싼다.

```jsx
<FavoritesContextProvider>
  <NavigationContainer>
    ...
  </NavigationContainer>
</FavoritesContextProvider>
```

이 구조는 네비게이션 내부의 모든 화면에서 Context를 사용할 수 있게 한다.

반대로 특정 네비게이터나 특정 화면 그룹에서만 필요한 상태라면, Provider를 그 하위 범위에 둘 수도 있다.

중요한 것은 상태가 필요한 컴포넌트들이 모두 Provider의 하위에 있어야 한다는 점이다.

### 기본값과 실제 값은 다르다

`createContext()`에 넣은 기본값은 실제 앱 상태의 주 공급원이 아니다.

실제 값은 Provider의 `value` prop으로 전달된다.

```text
createContext(defaultValue)
-> 기본 형태와 fallback

Provider value
-> 실제 앱에서 사용되는 값
```

이번 단계에서는 아직 `value`를 넣지 않았기 때문에 Context 구조만 만든 상태다.

## 정리

이번 강의에서는 Favorites 기능을 위한 Context 구조를 만들었다.

`store/context/favorites-context.js`에서 `FavoritesContext`를 생성하고, `ids`, `addFavorite`, `removeFavorite` 형태의 기본값을 정의했다.

또한 `FavoritesContextProvider` 컴포넌트를 만들고, `App.js`에서 `NavigationContainer` 전체를 감싸 여러 화면이 나중에 즐겨찾기 상태에 접근할 수 있도록 준비했다.

아직 실제 상태 관리 로직과 Provider의 `value`는 구현되지 않았으므로, 다음 단계에서 `useState`를 이용해 즐겨찾기 id 배열을 관리하고 추가/삭제 함수를 연결한다.
