# 9. FavoritesScreen에서 Redux 상태 읽기

## 핵심 요약

- Redux 방식의 Favorites 기능을 완성하려면 `FavoritesScreen`도 Redux store를 읽어야 한다.
- Context 방식에서 사용하던 `useContext(FavoritesContext)`는 더 이상 사용하지 않는다.
- `react-redux`의 `useSelector` 훅으로 `state.favoriteMeals.ids`를 읽는다.
- 읽어 온 favorite id 배열을 기존 필터링 로직에 그대로 사용한다.
- `MEALS.filter()`와 `favoriteMealIds.includes(meal.id)`를 조합해 즐겨찾기 meal 목록을 만든다.
- 나머지 `MealsList` 렌더링과 empty state 로직은 Context 방식과 거의 동일하게 유지된다.
- 이제 `MealDetailScreen`에서 dispatch한 Redux 상태 변경이 `FavoritesScreen`에도 반영된다.

## 학습 목표

- `FavoritesScreen`에서 `useSelector`로 Redux store 데이터를 읽는 방법을 익힌다.
- Context 기반 코드에서 Redux 기반 코드로 바꿀 때 어떤 부분만 바뀌는지 이해한다.
- Redux store의 favorite id 배열을 원본 `MEALS` 데이터 필터링에 사용하는 흐름을 복습한다.
- 하나의 Redux store 상태가 여러 화면의 UI를 동시에 갱신하는 구조를 이해한다.

## 주요 내용

### FavoritesScreen도 Redux store가 필요하다

이전 단계에서는 `MealDetailScreen`에서 Redux store를 사용했다.

```text
MealDetailScreen
-> useSelector로 favorite ids 읽기
-> useDispatch로 add/remove action dispatch
```

이제 `FavoritesScreen`도 같은 Redux store를 읽어야 한다.

`FavoritesScreen`의 역할은 현재 즐겨찾기로 저장된 meal 목록을 보여주는 것이다.

```text
FavoritesScreen
-> Redux store에서 favorite ids 읽기
-> MEALS 원본 데이터 필터링
-> MealsList로 렌더링
```

### Context 코드 제거

Context 방식에서는 다음처럼 Context를 읽었다.

```jsx
const favoriteMealsContext = useContext(FavoritesContext);
```

Redux 방식에서는 이 코드를 사용하지 않는다.

따라서 다음 import도 제거한다.

```jsx
import { useContext } from 'react';
import { FavoritesContext } from '../store/context/favorites-context';
```

대신 `useSelector`를 사용한다.

### useSelector import

Redux store에서 값을 읽기 위해 `react-redux`에서 `useSelector`를 가져온다.

```jsx
import { useSelector } from 'react-redux';
```

`FavoritesScreen`에서는 dispatch할 action이 필요하지 않다.

이 화면은 favorite 상태를 변경하지 않고 읽기만 하기 때문이다.

```text
MealDetailScreen
-> 읽기 + 쓰기
-> useSelector + useDispatch

FavoritesScreen
-> 읽기
-> useSelector
```

### favoriteMealIds 읽기

`useSelector`에 selector 함수를 전달한다.

```jsx
const favoriteMealIds = useSelector(
  (state) => state.favoriteMeals.ids
);
```

이 경로는 store 설정에서 결정되었다.

```jsx
const store = configureStore({
  reducer: {
    favoriteMeals: favoritesReducer,
  },
});
```

그리고 favorites slice의 초기 상태는 다음 구조였다.

```jsx
initialState: {
  ids: [],
}
```

따라서 전체 state에서 favorite ids를 읽는 경로는 다음과 같다.

```text
state.favoriteMeals.ids
```

### MEALS 필터링

이제 읽어 온 `favoriteMealIds` 배열을 기존 필터링 로직에 사용한다.

```jsx
const favoriteMeals = MEALS.filter((meal) =>
  favoriteMealIds.includes(meal.id)
);
```

흐름은 다음과 같다.

```text
MEALS 전체 배열 순회
-> 각 meal.id 확인
-> favoriteMealIds에 포함되어 있으면 유지
-> 포함되어 있지 않으면 제외
```

Context 방식과 비교하면, 바뀐 것은 favorite id 배열을 가져오는 출처뿐이다.

```text
Context 방식
-> favoriteMealsContext.ids

Redux 방식
-> favoriteMealIds
```

필터링 로직은 동일하다.

### MealsList와 empty state는 그대로 유지

`favoriteMeals` 배열을 만든 뒤의 UI 로직은 이전과 같다.

즐겨찾기가 없으면 안내 메시지를 보여준다.

```jsx
if (favoriteMeals.length === 0) {
  return (
    <View style={styles.rootContainer}>
      <Text style={styles.text}>
        You have no favorite meals yet.
      </Text>
    </View>
  );
}
```

즐겨찾기가 있으면 `MealsList`에 전달한다.

```jsx
return <MealsList items={favoriteMeals} />;
```

즉 Redux로 바꿨다고 해서 화면 전체를 다시 작성할 필요는 없다.

상태를 가져오는 부분만 바꾸면 기존 UI 구조를 그대로 재사용할 수 있다.

### 상세 화면과의 연결

`MealDetailScreen`에서 별 아이콘을 누르면 Redux store의 `favoriteMeals.ids`가 바뀐다.

`FavoritesScreen`은 `useSelector`로 같은 값을 구독하고 있다.

따라서 favorite 상태가 바뀌면 `FavoritesScreen`의 필터링 결과도 바뀐다.

```text
MealDetailScreen
-> dispatch(addFavorite({ id: mealId }))
-> Redux store 업데이트

FavoritesScreen
-> useSelector로 favorite ids 구독
-> favoriteMeals 재계산
-> 화면 업데이트
```

즐겨찾기 화면에서 meal을 눌러 상세 화면으로 이동한 뒤, 그 meal을 즐겨찾기 해제하고 돌아오면 해당 item은 목록에서 사라진다.

이것이 앱 전역 상태를 사용하는 목적이다.

## 예시 코드

### FavoritesScreen

```jsx
import { StyleSheet, Text, View } from 'react-native';
import { useSelector } from 'react-redux';

import MealsList from '../components/MealsList/MealsList';
import { MEALS } from '../data/dummy-data';

function FavoritesScreen() {
  const favoriteMealIds = useSelector(
    (state) => state.favoriteMeals.ids
  );

  const favoriteMeals = MEALS.filter((meal) =>
    favoriteMealIds.includes(meal.id)
  );

  if (favoriteMeals.length === 0) {
    return (
      <View style={styles.rootContainer}>
        <Text style={styles.text}>
          You have no favorite meals yet.
        </Text>
      </View>
    );
  }

  return <MealsList items={favoriteMeals} />;
}

export default FavoritesScreen;

const styles = StyleSheet.create({
  rootContainer: {
    flex: 1,
    justifyContent: 'center',
    alignItems: 'center',
  },
  text: {
    fontSize: 18,
    fontWeight: 'bold',
    color: 'white',
  },
});
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### Redux로 바꿔도 UI 컴포넌트 구조는 유지된다

`MealsList`, `MealItem`, empty state 스타일은 그대로 사용할 수 있다.

바뀐 것은 데이터의 출처다.

```text
기존
-> Context에서 ids 읽기

변경 후
-> Redux store에서 ids 읽기
```

상태 관리 계층과 UI 렌더링 계층을 잘 나누면, 상태 관리 방식을 바꿔도 UI 코드 변경 범위가 작아진다.

### useSelector는 화면을 store에 연결한다

`FavoritesScreen`은 `useSelector`를 통해 Redux store의 특정 값에 연결된다.

```jsx
const favoriteMealIds = useSelector(
  (state) => state.favoriteMeals.ids
);
```

이 값이 바뀌면 화면이 다시 렌더링되고, favorite meal 목록도 다시 계산된다.

```text
store 업데이트
-> selector 값 변경
-> FavoritesScreen 리렌더링
-> favoriteMeals 재계산
```

### 읽기 전용 화면에는 useDispatch가 필요 없다

`FavoritesScreen`은 favorite 상태를 직접 바꾸지 않는다.

따라서 `useDispatch`가 필요 없다.

```text
상태 읽기만 필요
-> useSelector

상태 변경 필요
-> useDispatch
```

화면이 어떤 역할을 하는지에 따라 필요한 Redux 훅만 사용하면 된다.

### Context와 Redux 모두 같은 결과를 낼 수 있다

이번 섹션에서 같은 Favorites 기능을 Context와 Redux 양쪽으로 구현했다.

두 방식 모두 결과는 같다.

```text
MealDetailScreen에서 favorite 토글
FavoritesScreen에서 favorite 목록 표시
```

차이는 상태 변경과 상태 접근을 어떤 패턴으로 구성하느냐다.

```text
Context
-> Provider 내부 useState
-> useContext로 값과 함수 사용

Redux
-> store + slice reducer
-> useSelector로 읽기
-> useDispatch로 action dispatch
```

## 정리

이번 강의에서는 `FavoritesScreen`도 Redux store를 사용하도록 변경했다.

`useSelector((state) => state.favoriteMeals.ids)`로 favorite id 배열을 읽고, `MEALS.filter()`와 `includes()`를 사용해 실제 favorite meal 목록을 만들었다.

나머지 `MealsList` 렌더링과 empty state 코드는 그대로 유지되며, 이제 `MealDetailScreen`에서 dispatch한 favorite 상태 변경이 `FavoritesScreen`에도 자동으로 반영된다.

이로써 Context API로 구현했던 Favorites 기능과 동일한 결과를 Redux Toolkit 기반으로도 완성했다.
