# 10. 섹션 마무리와 전역 상태 관리 핵심 정리

## 핵심 요약

- 이번 섹션에서는 Meals 앱의 즐겨찾기 기능을 완성했다.
- 사용자는 meal 상세 화면에서 즐겨찾기를 추가하거나 제거할 수 있다.
- `FavoritesScreen`에서는 즐겨찾기로 저장된 meal 목록을 볼 수 있다.
- 즐겨찾기 목록에서 meal을 눌러 다시 상세 화면으로 들어갈 수도 있다.
- 같은 기능을 React Context API 방식과 Redux Toolkit 방식으로 각각 구현했다.
- Context API와 Redux는 모두 React Native 앱에서 사용할 수 있는 앱 전역 상태 관리 방법이다.
- 중요한 결론은 둘 다 React Native 전용 기능이 아니라 일반 React 상태 관리 방식이라는 점이다.
- React Native에서는 렌더링 대상이 네이티브 UI일 뿐, Context와 Redux 사용법 자체는 React 웹 앱과 거의 같다.

## 학습 목표

- 이번 섹션에서 완성한 Favorites 기능의 전체 흐름을 정리한다.
- Context API 방식과 Redux Toolkit 방식의 구현 차이를 비교한다.
- React Native에서 앱 전역 상태 관리가 React의 일반 상태 관리 개념 위에 있다는 점을 이해한다.
- 여러 화면에서 같은 상태를 읽고 변경하는 구조를 복습한다.
- 앞으로 다른 React Native 앱에서도 Context 또는 Redux를 사용할 수 있는 감각을 정리한다.

## 주요 내용

### 완성된 기능

이번 섹션이 끝나면서 Meals 앱에는 실제 즐겨찾기 기능이 들어갔다.

```text
MealDetailScreen
-> 별 아이콘으로 favorite 추가
-> 다시 누르면 favorite 제거

FavoritesScreen
-> favorite meal 목록 표시
-> 목록 item을 눌러 MealDetailScreen으로 이동

MealDetailScreen에서 해제
-> FavoritesScreen 목록에서도 제거됨
```

즉 한 화면에서 바꾼 상태가 다른 화면에서도 반영된다.

이것이 앱 전역 상태 관리가 필요한 대표적인 상황이다.

### 앱 전역 상태로 관리한 데이터

이번 섹션에서 관리한 핵심 상태는 즐겨찾기 meal id 목록이다.

```jsx
['m1', 'm3']
```

이 배열은 두 화면에서 사용된다.

```text
MealDetailScreen
-> 현재 mealId가 favorite ids에 있는지 확인
-> add/remove로 상태 변경

FavoritesScreen
-> favorite ids를 읽음
-> MEALS 원본 데이터에서 favorite meal만 필터링
```

상태가 특정 화면 하나에만 속하지 않고 여러 화면에서 함께 필요하기 때문에 app-wide state로 관리했다.

### Context API 방식

먼저 React Context API로 구현했다.

핵심 구조는 다음과 같다.

```text
FavoritesContext
-> ids
-> addFavorite(id)
-> removeFavorite(id)

FavoritesContextProvider
-> useState로 favoriteMealIds 관리
-> value로 ids와 함수 제공

화면 컴포넌트
-> useContext(FavoritesContext)
```

Context 방식에서는 Provider 컴포넌트 내부에서 `useState`로 상태를 관리했다.

```jsx
const [favoriteMealIds, setFavoriteMealIds] = useState([]);
```

그리고 value 객체로 상태와 변경 함수를 공급했다.

```jsx
const value = {
  ids: favoriteMealIds,
  addFavorite,
  removeFavorite,
};
```

화면에서는 `useContext`로 이 값을 읽었다.

```jsx
const favoriteMealsContext = useContext(FavoritesContext);
```

### Context 방식의 특징

Context 방식은 React에 내장되어 있다.

별도 라이브러리를 설치하지 않아도 된다.

```text
장점
-> React 내장 기능
-> 작은 규모 전역 상태에 간단히 사용 가능
-> Provider와 useContext만으로 구현 가능

주의점
-> 상태 변경 구조가 커지면 Provider가 복잡해질 수 있음
-> 여러 상태 도메인이 많아지면 관리 방식 정리가 필요함
```

이번 Favorites 기능처럼 작은 규모의 공유 상태에는 Context API도 충분히 사용할 수 있다.

### Redux Toolkit 방식

이후 같은 기능을 Redux Toolkit으로 다시 구현했다.

핵심 구조는 다음과 같다.

```text
store/redux/store.js
-> configureStore()

store/redux/favorites.js
-> createSlice()
-> initialState: { ids: [] }
-> reducers: addFavorite, removeFavorite

App.js
-> react-redux Provider로 store 공급

화면 컴포넌트
-> useSelector로 읽기
-> useDispatch로 action dispatch
```

Redux 방식에서는 상태 변경 로직을 slice reducer로 분리했다.

```jsx
const favoritesSlice = createSlice({
  name: 'favorites',
  initialState: {
    ids: [],
  },
  reducers: {
    addFavorite: (state, action) => {
      state.ids.push(action.payload.id);
    },
    removeFavorite: (state, action) => {
      const index = state.ids.indexOf(action.payload.id);
      state.ids.splice(index, 1);
    },
  },
});
```

그리고 store에 등록했다.

```jsx
const store = configureStore({
  reducer: {
    favoriteMeals: favoritesReducer,
  },
});
```

### Redux 방식의 특징

Redux는 상태 변경 흐름을 action과 reducer 중심으로 명확하게 만든다.

```text
사용자 이벤트
-> dispatch(action)
-> reducer 실행
-> store 업데이트
-> useSelector를 사용하는 컴포넌트 리렌더링
```

화면에서는 store를 읽을 때 `useSelector`를 사용했다.

```jsx
const favoriteMealIds = useSelector(
  (state) => state.favoriteMeals.ids
);
```

상태를 바꿀 때는 `useDispatch`와 action creator를 사용했다.

```jsx
dispatch(addFavorite({ id: mealId }));
dispatch(removeFavorite({ id: mealId }));
```

Redux Toolkit을 사용하면 store 설정과 reducer/action 작성이 일반 Redux보다 간단해진다.

### Context와 Redux의 공통점

두 방식 모두 같은 문제를 해결했다.

```text
한 화면에서 favorite 상태 변경
다른 화면에서 favorite 목록 읽기
상태 변경 시 관련 화면 업데이트
```

둘 다 Provider를 앱 루트 근처에 배치했다.

```text
Context
-> FavoritesContextProvider

Redux
-> React Redux Provider
```

둘 다 `NavigationContainer` 전체를 감싸 여러 화면에서 상태에 접근할 수 있게 했다.

```text
Provider
  -> NavigationContainer
     -> 여러 navigators
        -> 여러 screens
```

### Context와 Redux의 차이

같은 결과를 만들 수 있지만 사용 패턴은 다르다.

```text
Context
-> Provider 내부 useState
-> useContext로 값과 함수 사용
-> 함수 호출로 상태 변경

Redux
-> 중앙 store
-> slice reducer
-> useSelector로 상태 읽기
-> useDispatch로 action dispatch
```

Context 방식은 더 직접적이고 간단하다.

Redux 방식은 구조가 더 명시적이고, 상태 변경 흐름이 action/reducer로 분리된다.

어느 쪽이 항상 더 좋은 것은 아니다.

앱 규모, 상태 복잡도, 팀 경험, 디버깅 요구에 따라 선택할 수 있다.

### React Native 전용 개념은 아니었다

이번 섹션의 가장 중요한 결론은 이것이다.

Context와 Redux 사용 방식은 React Native라고 해서 특별히 달라지지 않는다.

```text
React 웹 앱
-> Context API 사용 가능
-> Redux Toolkit 사용 가능

React Native 앱
-> Context API 사용 가능
-> Redux Toolkit 사용 가능
```

React Native는 UI를 네이티브 컴포넌트로 렌더링하지만, 상태 관리의 기반은 React다.

따라서 React에서 배운 Context와 Redux 지식은 React Native 앱에서도 그대로 활용된다.

### Favorites 기능의 전체 데이터 흐름

최종적으로 앱은 다음 흐름으로 동작한다.

```text
1. 사용자가 MealDetailScreen에서 별 아이콘을 누른다.
2. 전역 상태에 mealId가 추가 또는 제거된다.
3. MealDetailScreen은 현재 favorite 여부에 따라 아이콘을 바꾼다.
4. FavoritesScreen은 전역 상태의 favorite ids를 읽는다.
5. MEALS 원본 데이터에서 favorite meal만 필터링한다.
6. MealsList로 favorite meals를 렌더링한다.
7. favorite이 없으면 empty state 메시지를 보여준다.
```

상태 관리 방식이 Context든 Redux든 앱의 사용자 경험은 동일하다.

## 프론트엔드 개발자 관점에서의 핵심 차이

### 전역 상태는 화면 간 계약이다

즐겨찾기 상태는 특정 화면의 내부 구현이 아니라 여러 화면이 공유하는 계약이다.

```text
MealDetailScreen
-> favorite 상태 변경

FavoritesScreen
-> favorite 상태 읽기
```

이런 상태는 명확한 위치에서 관리해야 한다.

Context든 Redux든 핵심은 상태 소유권을 화면 바깥으로 끌어올리는 것이다.

### 상태 관리 도구보다 데이터 흐름이 먼저다

Context를 쓸지 Redux를 쓸지보다 먼저 생각해야 할 것은 데이터 흐름이다.

```text
누가 상태를 바꾸는가?
누가 상태를 읽는가?
상태 변경이 어떤 UI에 영향을 주는가?
상태가 어느 범위까지 필요할까?
```

이 질문에 답하면 어떤 도구를 써야 할지도 더 분명해진다.

### UI 컴포넌트 분리는 상태 관리 방식 변경을 쉽게 만든다

`MealsList`와 `MealItem`을 분리했기 때문에 Context에서 Redux로 바뀌어도 목록 UI는 거의 그대로 유지됐다.

```text
상태 선택
-> 화면 컴포넌트

목록 렌더링
-> MealsList

개별 카드
-> MealItem
```

상태 관리와 UI 렌더링 책임이 분리되어 있으면 구현 방식을 바꿀 때 수정 범위가 줄어든다.

### React Native 앱도 React 앱이다

이번 섹션에서 사용한 핵심 API는 모두 React 생태계의 일반 개념이다.

```text
useState
createContext
useContext
Provider
Redux store
useSelector
useDispatch
```

React Native에서는 이 상태가 화면에 반영될 때 `View`, `Text`, `FlatList` 같은 네이티브 컴포넌트가 업데이트될 뿐이다.

상태 관리 사고방식은 React 웹 앱과 같다.

## 정리

이번 섹션에서는 Meals 앱의 즐겨찾기 기능을 앱 전역 상태로 구현했다.

먼저 Context API를 사용해 `FavoritesContextProvider`에서 favorite id 배열과 추가/삭제 함수를 제공했고, 이후 같은 기능을 Redux Toolkit과 React Redux로 다시 구현했다.

두 방식 모두 React Native 앱에서 자연스럽게 사용할 수 있으며, 중요한 것은 React Native 전용 문법이 아니라 React의 상태 관리 개념을 여러 화면을 가진 모바일 앱 구조에 적용하는 것이다.

이제 다른 React Native 앱에서도 Context API나 Redux를 사용해 여러 화면이 공유하는 상태를 관리할 수 있다.
