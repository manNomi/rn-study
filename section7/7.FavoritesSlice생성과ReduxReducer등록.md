# 7. Favorites Slice 생성과 Redux Reducer 등록

## 핵심 요약

- Redux store를 만들었지만 아직 실제 favorites 상태 로직은 없었다.
- Redux Toolkit에서는 `createSlice()`로 특정 기능의 상태, reducer, action을 한 번에 정의할 수 있다.
- `store/redux/favorites.js` 파일을 만들고 favorites 관련 Redux 로직을 분리한다.
- slice의 `initialState`에는 즐겨찾기 meal id 배열인 `ids: []`를 둔다.
- `reducers` 객체 안에 `addFavorite`와 `removeFavorite` reducer를 정의한다.
- Redux Toolkit reducer는 `state`와 `action`을 인자로 받는다.
- 추가 데이터는 `action.payload`를 통해 전달된다.
- Redux Toolkit은 내부적으로 불변 업데이트를 처리해 주므로 reducer 안에서 `push`, `splice`처럼 mutation처럼 보이는 코드를 작성할 수 있다.
- slice에서 생성된 action creator를 export하고, slice reducer를 default export한다.
- `store.js`의 `configureStore()`에 favorites reducer를 등록해야 앱에서 사용할 수 있다.

## 학습 목표

- Redux Toolkit의 `createSlice()` 기본 구조를 이해한다.
- favorites 상태를 위한 slice를 별도 파일에 분리하는 이유를 이해한다.
- `initialState`, `reducers`, `action.payload`의 역할을 익힌다.
- Redux Toolkit에서 reducer 내부 mutation 코드가 허용되는 이유를 개념적으로 이해한다.
- action creator와 reducer를 export하는 패턴을 익힌다.
- `configureStore()`에 slice reducer를 등록하는 방법을 이해한다.

## 주요 내용

### 실제 Redux 로직 추가하기

이전 단계에서는 Redux store만 만들었다.

```jsx
const store = configureStore({
  reducer: {},
});
```

하지만 이 상태에는 아직 관리할 데이터도, 데이터를 바꿀 로직도 없다.

이번 단계에서는 Favorites 기능을 위한 Redux 상태 조각을 만든다.

### 기능별 파일 분리

Redux 로직을 모두 `store.js`에 작성할 수도 있다.

하지만 앱이 커지면 기능별로 파일을 분리하는 편이 유지보수에 좋다.

강의에서는 다음 파일을 만든다.

```text
store/
  redux/
    store.js
    favorites.js
```

`favorites.js`는 즐겨찾기 상태와 그 상태를 변경하는 로직을 담당한다.

```text
favorites.js
-> favorite meal ids 상태
-> addFavorite reducer
-> removeFavorite reducer
-> action exports
-> reducer export
```

### createSlice import

Redux Toolkit에서 slice를 만들려면 `createSlice`를 가져온다.

```jsx
import { createSlice } from '@reduxjs/toolkit';
```

`createSlice()`는 특정 상태 영역의 이름, 초기 상태, reducer 함수들을 받아 slice 객체를 만들어 준다.

```jsx
const favoritesSlice = createSlice({
  name: 'favorites',
  initialState: {
    ids: [],
  },
  reducers: {},
});
```

### slice란?

slice는 앱 상태의 한 조각과 그 조각을 변경하는 reducer/action 묶음이다.

```text
favorites slice
-> state: favorite meal ids
-> reducers: addFavorite, removeFavorite
-> actions: addFavorite action, removeFavorite action
```

Context 방식에서는 하나의 Context 객체에 상태와 함수 형태를 함께 정의했다.

Redux Toolkit에서는 slice 안에서 상태와 reducer를 정의하고, action creator는 자동으로 생성된다.

### name

slice에는 이름이 필요하다.

```jsx
name: 'favorites'
```

이 이름은 Redux Toolkit 내부에서 action type을 만들 때 사용된다.

예를 들어 `favorites/addFavorite` 같은 식의 action type이 만들어질 수 있다.

일반적으로 해당 상태 영역을 설명하는 이름을 사용한다.

### initialState

즐겨찾기 상태의 초기값은 빈 id 배열이다.

```jsx
initialState: {
  ids: [],
}
```

Context 방식에서 `favoriteMealIds` 배열을 `useState([])`로 시작했던 것과 같은 의미다.

```text
Context 방식
-> useState([])

Redux 방식
-> initialState: { ids: [] }
```

`ids`에는 즐겨찾기로 저장된 meal id들이 들어간다.

```jsx
{
  ids: ['m1', 'm3']
}
```

### reducers 객체

상태를 변경하는 함수들은 `reducers` 안에 정의한다.

```jsx
reducers: {
  addFavorite: (state, action) => {},
  removeFavorite: (state, action) => {},
}
```

각 reducer 함수는 Redux Toolkit이 호출한다.

자동으로 두 가지 인자를 받는다.

```text
state
-> 현재 slice 상태

action
-> dispatch된 action 객체
-> payload에 추가 데이터가 들어갈 수 있음
```

### action.payload

즐겨찾기를 추가하거나 삭제하려면 어떤 meal id를 대상으로 할지 알아야 한다.

그 id는 action의 payload로 전달한다.

```jsx
action.payload.id
```

나중에 화면에서 action을 dispatch할 때 다음과 같은 형태로 데이터를 넘길 수 있다.

```jsx
dispatch(addFavorite({ id: mealId }));
```

그러면 reducer 안에서는 이 값을 읽을 수 있다.

```jsx
action.payload.id
```

```text
dispatch(addFavorite({ id: 'm1' }))
-> action.payload.id === 'm1'
```

### addFavorite reducer

즐겨찾기 id 배열에 새 id를 추가한다.

```jsx
addFavorite: (state, action) => {
  state.ids.push(action.payload.id);
}
```

일반 Redux만 사용할 때는 상태를 직접 mutate하면 안 된다.

하지만 Redux Toolkit은 내부적으로 Immer를 사용해 mutation처럼 보이는 코드를 안전한 불변 업데이트로 변환한다.

```text
작성 코드
-> state.ids.push(...)

Redux Toolkit 내부
-> 불변 업데이트로 처리
```

그래서 Redux Toolkit reducer 안에서는 이런 방식이 허용된다.

### removeFavorite reducer

즐겨찾기 id 배열에서 특정 id를 제거한다.

```jsx
removeFavorite: (state, action) => {
  const index = state.ids.indexOf(action.payload.id);
  state.ids.splice(index, 1);
}
```

`indexOf()`로 삭제할 id의 위치를 찾는다.

```jsx
const index = state.ids.indexOf(action.payload.id);
```

`splice(index, 1)`은 해당 index에서 1개 항목을 제거한다.

```jsx
state.ids.splice(index, 1);
```

이 코드도 mutation처럼 보이지만 Redux Toolkit reducer 안에서는 사용할 수 있다.

### reducer 코드 작성 시 주의점

강의 흐름에서는 `indexOf()`로 찾은 값을 바로 `splice()`에 사용한다.

실제 앱에서는 id가 없을 가능성까지 고려해 `index >= 0`인지 확인하는 방어 코드를 넣을 수도 있다.

```jsx
if (index >= 0) {
  state.ids.splice(index, 1);
}
```

현재 흐름에서는 favorite인 상태에서만 remove를 호출할 예정이므로 강의 코드는 간단하게 유지한다.

### action creator export

`createSlice()`는 reducers에 정의한 함수 이름을 기반으로 action creator를 자동 생성한다.

이를 화면 컴포넌트에서 dispatch하려면 export해야 한다.

```jsx
export const addFavorite = favoritesSlice.actions.addFavorite;
export const removeFavorite = favoritesSlice.actions.removeFavorite;
```

이제 다른 파일에서 다음처럼 가져와 사용할 수 있다.

```jsx
import { addFavorite, removeFavorite } from '../store/redux/favorites';
```

이 함수들은 직접 상태를 바꾸는 함수가 아니다.

dispatch할 action 객체를 만들어 주는 action creator다.

```text
addFavorite({ id: mealId })
-> action 객체 생성

dispatch(addFavorite({ id: mealId }))
-> Redux store에 action 전달
-> reducer 실행
```

### reducer export

store에 등록할 reducer도 export해야 한다.

```jsx
export default favoritesSlice.reducer;
```

이 reducer는 `configureStore()`의 `reducer` 설정에 들어간다.

### favorites.js 전체 예시

```jsx
import { createSlice } from '@reduxjs/toolkit';

const favoritesSlice = createSlice({
  name: 'favorites',
  initialState: {
    ids: [],
  },
  reducers: {
    addFavorite: (state, action) => {
      state.ids.push(action.payload.id);
    },
    removeFavorite: (state, action) => {
      const index = state.ids.indexOf(action.payload.id);
      state.ids.splice(index, 1);
    },
  },
});

export const addFavorite = favoritesSlice.actions.addFavorite;
export const removeFavorite = favoritesSlice.actions.removeFavorite;

export default favoritesSlice.reducer;
```

### store.js에 reducer 등록

이제 `favorites.js`에서 default export한 reducer를 `store.js`에서 가져온다.

```jsx
import favoritesReducer from './favorites';
```

그리고 `configureStore()`의 `reducer` 객체에 등록한다.

```jsx
const store = configureStore({
  reducer: {
    favoriteMeals: favoritesReducer,
  },
});
```

여기서 `favoriteMeals`는 Redux state 안에서 이 slice가 저장될 key다.

```text
전체 Redux state
-> state.favoriteMeals
   -> ids
```

따라서 나중에 컴포넌트에서 ids를 읽을 때는 다음과 같은 경로를 사용하게 된다.

```jsx
state.favoriteMeals.ids
```

key 이름은 자유지만, 이후 selector에서 그대로 사용해야 하므로 의미 있는 이름을 고르는 것이 좋다.

## 예시 코드

### store/redux/store.js

```jsx
import { configureStore } from '@reduxjs/toolkit';

import favoritesReducer from './favorites';

const store = configureStore({
  reducer: {
    favoriteMeals: favoritesReducer,
  },
});

export default store;
```

### state 구조

```jsx
{
  favoriteMeals: {
    ids: []
  }
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### Context의 함수와 Redux의 action은 다르다

Context 방식에서는 직접 만든 함수를 Context value로 제공했다.

```jsx
favoriteMealsContext.addFavorite(mealId);
```

Redux 방식에서는 action creator를 dispatch한다.

```jsx
dispatch(addFavorite({ id: mealId }));
```

```text
Context
-> 함수 호출로 상태 변경

Redux
-> action dispatch로 상태 변경
```

상태 변경 흐름이 더 명시적으로 분리된다는 점이 Redux의 특징이다.

### Redux Toolkit은 불변 업데이트 부담을 줄인다

일반 Redux에서는 reducer에서 상태를 직접 mutate하면 안 된다.

Redux Toolkit에서는 reducer 내부에서 mutation처럼 보이는 코드를 작성할 수 있다.

```jsx
state.ids.push(action.payload.id);
state.ids.splice(index, 1);
```

Redux Toolkit이 내부적으로 안전한 불변 업데이트로 처리하기 때문이다.

그래서 reducer 코드가 짧고 읽기 쉬워진다.

### store에 등록한 key가 selector 경로가 된다

`store.js`에서 다음처럼 등록했다.

```jsx
reducer: {
  favoriteMeals: favoritesReducer,
}
```

그러면 컴포넌트에서 상태를 읽을 때도 `favoriteMeals`라는 key를 사용한다.

```jsx
state.favoriteMeals.ids
```

store 구조를 어떻게 잡는지가 이후 컴포넌트 코드의 selector 경로를 결정한다.

### 파일 분리는 상태 도메인을 드러낸다

`favorites.js` 파일을 따로 만들면 상태 도메인이 명확해진다.

```text
store/redux/favorites.js
-> favorites 상태와 액션

store/redux/store.js
-> 전체 store 조립
```

작은 앱에서는 한 파일에 작성해도 되지만, 앱이 커질수록 기능별 slice 분리가 훨씬 관리하기 쉽다.

## 정리

이번 강의에서는 Redux Toolkit의 `createSlice()`를 사용해 Favorites 상태 로직을 만들었다.

`favorites.js`에서 `ids` 배열을 초기 상태로 두고, `addFavorite`와 `removeFavorite` reducer를 작성했다.

또한 slice가 자동 생성한 action creator들을 export하고, slice reducer를 `store.js`의 `configureStore()`에 `favoriteMeals` key로 등록했다.

이제 Redux store에는 즐겨찾기 상태와 상태 변경 로직이 준비되었고, 다음 단계에서는 화면 컴포넌트에서 `useDispatch`와 `useSelector`로 이 store를 실제로 사용한다.
