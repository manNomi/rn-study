# 4. MealDetailScreen에서 Context 사용과 즐겨찾기 토글

## 핵심 요약

- 이제 `FavoritesContext`를 실제 화면에서 사용한다.
- 먼저 `MealDetailScreen`에서 Context를 소비한다.
- React의 `useContext` 훅을 사용해 `FavoritesContext`에 접근한다.
- `favoriteMealsContext.ids` 배열에 현재 `mealId`가 포함되어 있는지 확인해 현재 meal이 즐겨찾기인지 판단한다.
- `includes(mealId)`는 배열에 해당 id가 있으면 `true`, 없으면 `false`를 반환한다.
- 현재 meal이 즐겨찾기라면 채워진 `star` 아이콘을 보여주고, 아니면 `star-outline` 아이콘을 보여준다.
- 헤더 버튼을 누르면 현재 상태에 따라 `addFavorite(mealId)` 또는 `removeFavorite(mealId)`를 호출한다.
- 이렇게 하면 `MealDetailScreen`에서 즐겨찾기 상태를 토글할 수 있다.
- Context API와 `useContext`는 React Native 전용 기능이 아니라 표준 React 기능이다.

## 학습 목표

- 화면 컴포넌트에서 `useContext`로 Context 값을 읽는 방법을 익힌다.
- Context의 상태 값과 상태 변경 함수를 함께 사용하는 패턴을 이해한다.
- 현재 meal이 즐겨찾기인지 `ids.includes(mealId)`로 판단하는 방법을 익힌다.
- Context 상태를 기반으로 헤더 아이콘을 조건부 렌더링하는 흐름을 이해한다.
- 헤더 버튼 클릭으로 전역 상태를 변경하는 구조를 익힌다.

## 주요 내용

### MealDetailScreen에서 Context가 필요한 이유

`MealDetailScreen`에는 헤더 오른쪽에 별 아이콘 버튼이 있다.

이전까지 이 버튼은 눌러도 콘솔에 로그만 찍었다.

이제 이 버튼은 실제로 현재 meal의 즐겨찾기 상태를 바꿔야 한다.

이를 위해 `MealDetailScreen`에는 두 가지 정보가 필요하다.

```text
1. 현재 meal이 이미 즐겨찾기인지 여부
2. 즐겨찾기 추가/삭제 함수
```

두 정보는 모두 `FavoritesContext`에서 제공된다.

```text
FavoritesContext.ids
-> 즐겨찾기 meal id 배열

FavoritesContext.addFavorite(id)
-> 즐겨찾기 추가

FavoritesContext.removeFavorite(id)
-> 즐겨찾기 제거
```

### useContext 가져오기

Context를 사용하려면 React의 `useContext` 훅을 사용한다.

```jsx
import { useContext } from 'react';
```

이미 `useLayoutEffect`를 사용하고 있다면 같은 import에서 함께 가져올 수 있다.

```jsx
import { useContext, useLayoutEffect } from 'react';
```

`useContext`는 React Native 전용 기능이 아니다.

React 웹 앱에서도 동일하게 사용하는 표준 React 훅이다.

### FavoritesContext import

`useContext`에는 Context 객체를 전달해야 한다.

따라서 `favorites-context.js`에서 export한 `FavoritesContext`를 가져온다.

```jsx
import { FavoritesContext } from '../store/context/favorites-context';
```

여기서 가져오는 것은 Provider 컴포넌트가 아니라 Context 객체다.

```text
FavoritesContextProvider
-> App.js에서 앱을 감싸는 Provider 컴포넌트

FavoritesContext
-> useContext에 전달해 값을 읽는 Context 객체
```

### useContext로 Context 값 읽기

`MealDetailScreen` 안에서 `useContext`를 호출한다.

```jsx
const favoriteMealsContext = useContext(FavoritesContext);
```

이제 `favoriteMealsContext`에는 Provider의 `value`로 전달한 객체가 들어 있다.

```jsx
{
  ids: favoriteMealIds,
  addFavorite: addFavorite,
  removeFavorite: removeFavorite,
}
```

따라서 다음과 같이 사용할 수 있다.

```jsx
favoriteMealsContext.ids
favoriteMealsContext.addFavorite(mealId)
favoriteMealsContext.removeFavorite(mealId)
```

### 현재 meal이 즐겨찾기인지 확인하기

`MealDetailScreen`은 route params에서 현재 화면의 meal id를 이미 가지고 있다.

```jsx
const mealId = route.params.mealId;
```

현재 meal이 즐겨찾기인지 확인하려면 이 `mealId`가 Context의 `ids` 배열 안에 있는지 보면 된다.

```jsx
const mealIsFavorite = favoriteMealsContext.ids.includes(mealId);
```

`includes()`는 JavaScript 배열 메서드다.

```text
ids.includes(mealId)
-> ids 배열 안에 mealId가 있으면 true
-> 없으면 false
```

여기서는 id가 문자열 같은 원시값이므로 `includes()`를 사용하기에 적합하다.

### 아이콘 조건부 변경

현재 meal이 즐겨찾기인지에 따라 헤더 아이콘을 다르게 보여줄 수 있다.

```jsx
<IconButton
  icon={mealIsFavorite ? 'star' : 'star-outline'}
  color="white"
  onPress={changeFavoriteStatusHandler}
/>
```

```text
mealIsFavorite === true
-> 'star'
-> 채워진 별

mealIsFavorite === false
-> 'star-outline'
-> 빈 별
```

이렇게 하면 전역 상태가 UI에 바로 반영된다.

### 핸들러 이름 변경

기존 함수 이름은 단순히 헤더 버튼을 눌렀다는 의미에 가까웠다.

이제 실제 역할은 즐겨찾기 상태 변경이다.

따라서 함수 이름을 더 명확하게 바꾼다.

```jsx
function changeFavoriteStatusHandler() {
  ...
}
```

이 함수는 헤더 버튼의 `onPress`에 연결된다.

```jsx
onPress={changeFavoriteStatusHandler}
```

### 즐겨찾기 상태 토글

현재 meal이 이미 즐겨찾기라면 제거해야 한다.

```jsx
if (mealIsFavorite) {
  favoriteMealsContext.removeFavorite(mealId);
}
```

아직 즐겨찾기가 아니라면 추가해야 한다.

```jsx
else {
  favoriteMealsContext.addFavorite(mealId);
}
```

전체 함수는 다음과 같다.

```jsx
function changeFavoriteStatusHandler() {
  if (mealIsFavorite) {
    favoriteMealsContext.removeFavorite(mealId);
  } else {
    favoriteMealsContext.addFavorite(mealId);
  }
}
```

이제 버튼 하나로 즐겨찾기 추가와 제거를 모두 처리한다.

```text
현재 favorite
-> 클릭 시 removeFavorite

현재 not favorite
-> 클릭 시 addFavorite
```

### useLayoutEffect 의존성 업데이트

헤더 버튼은 `navigation.setOptions()` 안에서 설정된다.

```jsx
useLayoutEffect(() => {
  navigation.setOptions({
    headerRight: () => (
      <IconButton
        icon={mealIsFavorite ? 'star' : 'star-outline'}
        color="white"
        onPress={changeFavoriteStatusHandler}
      />
    ),
  });
}, [navigation, changeFavoriteStatusHandler]);
```

이제 헤더 아이콘은 `mealIsFavorite` 값에 따라 달라진다.

따라서 의존성 배열에는 헤더 설정 안에서 사용하는 값들이 포함되어야 한다.

```jsx
[
  navigation,
  changeFavoriteStatusHandler,
  mealIsFavorite
]
```

다만 `changeFavoriteStatusHandler`가 컴포넌트 렌더링마다 새로 만들어지는 함수라면 의존성 관리가 신경 쓰일 수 있다.

강의 흐름에서는 이 함수를 의존성에 넣고, 버튼이 현재 상태를 기준으로 다시 설정되도록 만든다.

핵심은 `mealIsFavorite`이 바뀌면 헤더 아이콘도 다시 계산되어야 한다는 점이다.

### 동작 결과

앱을 새로고침하고 meal 상세 화면으로 들어가면 별 아이콘이 표시된다.

버튼을 누르면 다음 일이 일어난다.

```text
1. changeFavoriteStatusHandler 실행
2. 현재 favorite 여부 확인
3. addFavorite 또는 removeFavorite 호출
4. Context 상태 업데이트
5. MealDetailScreen 리렌더링
6. mealIsFavorite 값 변경
7. 헤더 아이콘이 star 또는 star-outline으로 변경
```

이제 `MealDetailScreen`에서는 즐겨찾기 상태 토글이 동작한다.

다만 아직 `FavoritesScreen`에서는 이 상태를 사용하지 않는다.

다음 단계에서는 `FavoritesScreen`에서 Context를 읽어 실제 즐겨찾기 meal 목록을 표시한다.

## 예시 코드

### MealDetailScreen 핵심 흐름

```jsx
import { useContext, useLayoutEffect } from 'react';

import IconButton from '../components/IconButton';
import { FavoritesContext } from '../store/context/favorites-context';

function MealDetailScreen({ route, navigation }) {
  const favoriteMealsContext = useContext(FavoritesContext);

  const mealId = route.params.mealId;

  const mealIsFavorite =
    favoriteMealsContext.ids.includes(mealId);

  function changeFavoriteStatusHandler() {
    if (mealIsFavorite) {
      favoriteMealsContext.removeFavorite(mealId);
    } else {
      favoriteMealsContext.addFavorite(mealId);
    }
  }

  useLayoutEffect(() => {
    navigation.setOptions({
      headerRight: () => (
        <IconButton
          icon={mealIsFavorite ? 'star' : 'star-outline'}
          color="white"
          onPress={changeFavoriteStatusHandler}
        />
      ),
    });
  }, [navigation, changeFavoriteStatusHandler, mealIsFavorite]);

  // screen content...
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### Context 소비는 React Native에서도 동일하다

`useContext`는 React Native만의 특별한 API가 아니다.

React 웹 앱에서 Context를 읽는 방식과 동일하다.

```jsx
const contextValue = useContext(SomeContext);
```

React Native에서는 DOM 대신 네이티브 컴포넌트를 렌더링할 뿐, Context의 사용 방식은 그대로다.

### 전역 상태가 UI를 결정한다

헤더 아이콘은 이제 단순히 고정된 `star`가 아니다.

전역 상태인 `favoriteMealsContext.ids`에 따라 달라진다.

```text
Context state
-> mealIsFavorite 계산
-> IconButton icon prop 결정
```

즉 UI가 로컬 상태가 아니라 앱 전역 상태에 반응한다.

### 상태 읽기와 상태 변경을 같은 Context에서 가져온다

`MealDetailScreen`은 Context에서 상태와 액션을 모두 꺼낸다.

```text
읽기
-> favoriteMealsContext.ids

쓰기
-> favoriteMealsContext.addFavorite
-> favoriteMealsContext.removeFavorite
```

이 패턴은 작은 규모의 전역 상태 관리에서 흔히 사용된다.

### params와 Context의 역할이 함께 쓰인다

`MealDetailScreen`은 두 종류의 정보를 조합한다.

```text
route.params.mealId
-> 현재 화면이 어떤 meal을 보여주는지 알려줌

FavoritesContext.ids
-> 현재 앱에서 어떤 meal들이 favorite인지 알려줌
```

현재 화면의 `mealId`와 전역 favorite 목록을 비교해서 `mealIsFavorite`을 계산한다.

따라서 params와 app-wide state는 서로 대체 관계가 아니라 함께 사용되는 경우가 많다.

## 정리

이번 강의에서는 `MealDetailScreen`에서 `FavoritesContext`를 실제로 사용했다.

`useContext(FavoritesContext)`로 즐겨찾기 id 배열과 추가/삭제 함수를 가져오고, 현재 `mealId`가 `ids`에 포함되어 있는지 `includes()`로 확인했다.

그 결과에 따라 헤더의 별 아이콘을 `star` 또는 `star-outline`으로 바꾸고, 버튼을 누르면 `addFavorite` 또는 `removeFavorite`를 호출해 즐겨찾기 상태를 토글하도록 연결했다.

이제 상세 화면에서는 즐겨찾기 상태가 실제로 바뀌며, 다음 단계에서는 이 상태를 `FavoritesScreen`에서도 읽어 화면에 표시한다.
