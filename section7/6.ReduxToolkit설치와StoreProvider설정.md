# 6. Redux Toolkit 설치와 Store Provider 설정

## 핵심 요약

- Context API만으로 앱 전역 상태를 관리할 수 있다면 Redux를 반드시 사용할 필요는 없다.
- 이번 강의부터는 같은 Favorites 기능을 Redux 방식으로 다시 구현한다.
- 강의에서는 Redux의 공식 권장 방식인 Redux Toolkit을 사용한다.
- Redux를 React 또는 React Native 앱에서 사용하려면 `@reduxjs/toolkit`과 `react-redux`를 설치한다.
- `store/redux/store.js` 파일을 만들고 `configureStore()`로 Redux store를 생성한다.
- `configureStore()`에는 `reducer` 설정이 필요하지만, 이번 단계에서는 아직 실제 reducer 로직 없이 빈 객체로 시작한다.
- 앱에서 Redux store를 사용하려면 `react-redux`의 `Provider`로 앱 컴포넌트 트리를 감싸야 한다.
- `Provider`에는 `store` prop으로 생성한 Redux store를 전달한다.
- Context Provider와 마찬가지로, Redux Provider도 store를 필요로 하는 모든 화면보다 높은 위치에 배치해야 한다.

## 학습 목표

- Context API 방식과 Redux 방식이 같은 문제를 해결할 수 있음을 이해한다.
- Redux Toolkit이 Redux 사용의 공식 권장 방식이라는 점을 정리한다.
- `@reduxjs/toolkit`과 `react-redux`의 역할 차이를 이해한다.
- `configureStore()`로 Redux store를 만드는 기본 구조를 익힌다.
- React Redux의 `Provider`로 앱에 store를 공급하는 방법을 익힌다.
- `NavigationContainer`와 Redux Provider의 배치 관계를 이해한다.

## 주요 내용

### Context만으로 충분할 수도 있다

이전 강의들에서는 Context API로 Favorites 기능을 구현했다.

```text
FavoritesContext
-> favorite meal ids 관리
-> addFavorite
-> removeFavorite
-> MealDetailScreen과 FavoritesScreen에서 사용
```

이 방식만으로 충분하다면 Redux를 사용할 필요는 없다.

Redux는 앱 전역 상태를 관리하는 또 다른 방법이다.

따라서 Context가 목적에 맞고 복잡도가 낮다면 그대로 사용해도 된다.

이번 강의부터는 Redux를 사용하는 개발자를 위해 같은 결과를 Redux 방식으로 구현한다.

### Redux Toolkit 사용

강의에서는 Redux Toolkit을 사용한다.

Redux Toolkit은 Redux를 더 간단하고 일관되게 사용할 수 있도록 만든 공식 권장 도구다.

```text
Redux Toolkit
-> Redux 공식 권장 방식
-> store 설정 단순화
-> slice/reducer/action 작성 간소화
```

Redux 자체나 Redux Toolkit의 개념을 깊게 설명하는 것이 이번 강의의 주목적은 아니다.

이번 목표는 React Native 앱에서 Redux store를 어떻게 설정하고 앱에 연결하는지 보는 것이다.

### 필요한 패키지 설치

Redux Toolkit과 React Redux를 설치한다.

```bash
npm install @reduxjs/toolkit react-redux
```

두 패키지의 역할은 다르다.

```text
@reduxjs/toolkit
-> Redux store, reducer, slice 등을 더 쉽게 만들기 위한 도구

react-redux
-> React 컴포넌트에서 Redux store에 접근하기 위한 바인딩
-> Provider, useSelector, useDispatch 등을 제공
```

React Native 앱도 React 앱이므로 `react-redux`를 사용해 Redux store를 컴포넌트 트리에 연결한다.

### redux 폴더 사용

이전 Context 구현을 위해 `store/context` 폴더를 만들었다.

Redux 구현은 `store/redux` 폴더에서 진행한다.

```text
store/
  context/
    favorites-context.js
  redux/
    store.js
```

Context 폴더는 지워도 되지만, 강의에서는 두 방식을 비교할 수 있도록 남겨 둔다.

실제 프로젝트에서는 둘 중 하나의 접근만 사용한다면 사용하지 않는 코드는 정리하는 편이 좋다.

### store.js 파일 만들기

Redux store는 `store/redux/store.js` 파일에서 만든다.

먼저 Redux Toolkit에서 `configureStore`를 가져온다.

```jsx
import { configureStore } from '@reduxjs/toolkit';
```

그리고 `configureStore()`를 호출해 store를 만든다.

```jsx
const store = configureStore({
  reducer: {},
});

export default store;
```

`configureStore()`는 Redux store를 생성하는 함수다.

강의에서는 아직 실제 상태 로직을 만들지 않았으므로 `reducer`를 빈 객체로 둔다.

### reducer 설정의 의미

Redux store는 여러 상태 조각과 그 상태를 바꾸는 로직을 합쳐 만든다.

Redux Toolkit에서는 보통 slice 단위로 상태와 reducer를 만든 뒤 store에 등록한다.

```text
slice
-> 특정 상태 영역
-> initial state
-> reducers
-> actions
```

이번 단계에서는 아직 favorites slice를 만들지 않았기 때문에 빈 reducer 객체로 시작한다.

```jsx
reducer: {}
```

다음 단계에서 Favorites 상태를 위한 reducer 로직을 추가한다.

### App.js에서 Context Provider 제거

이전에는 `FavoritesContextProvider`로 앱 전체를 감쌌다.

```jsx
<FavoritesContextProvider>
  <NavigationContainer>
    ...
  </NavigationContainer>
</FavoritesContextProvider>
```

Redux 방식으로 바꾸기 위해 Context Provider는 사용하지 않는다.

강의에서는 비교를 위해 import나 JSX를 주석 처리할 수 있다.

```jsx
// import FavoritesContextProvider from './store/context/favorites-context';
```

이후 Redux의 `Provider`를 사용한다.

### React Redux Provider

Redux store를 React 컴포넌트 트리에 공급하려면 `react-redux`의 `Provider`를 사용한다.

```jsx
import { Provider } from 'react-redux';
```

여기서 `Provider`는 Context API의 Provider와 개념적으로 비슷하다.

상위에서 값을 제공하고, 하위 컴포넌트들이 그 값에 접근할 수 있게 한다.

다만 Redux Provider는 Context 객체를 직접 제공하는 것이 아니라 Redux store를 제공한다.

### store import

`store/redux/store.js`에서 만든 store를 `App.js`로 가져온다.

```jsx
import store from './store/redux/store';
```

이 store를 `Provider`의 `store` prop으로 전달한다.

```jsx
<Provider store={store}>
  <NavigationContainer>
    ...
  </NavigationContainer>
</Provider>
```

이제 `NavigationContainer` 내부의 모든 화면과 컴포넌트가 Redux store에 접근할 준비가 된다.

### Provider 위치

Context Provider와 마찬가지로 Redux Provider도 store를 필요로 하는 컴포넌트보다 위에 있어야 한다.

이번 앱에서는 `MealDetailScreen`과 `FavoritesScreen`이 모두 Redux store를 사용할 예정이다.

```text
MealDetailScreen
-> favorite 추가/삭제 action dispatch

FavoritesScreen
-> favorite ids selector로 읽기
```

따라서 `NavigationContainer` 전체를 Redux `Provider`로 감싸는 것이 자연스럽다.

```text
Provider
  -> NavigationContainer
     -> Stack Navigator
        -> Drawer Navigator
           -> FavoritesScreen
        -> MealDetailScreen
```

이 구조에서는 네비게이션 안의 모든 화면이 Redux store에 접근할 수 있다.

### 아직 로직은 없다

이번 단계는 Redux를 앱에 연결하는 준비 단계다.

아직 favorites 상태나 action은 없다.

```text
현재 완료
-> Redux Toolkit 설치
-> store.js 생성
-> configureStore 호출
-> App.js에서 Provider로 감싸기

아직 필요
-> favorites slice 생성
-> add/remove favorite reducer 작성
-> 화면에서 useDispatch/useSelector 사용
```

즉 Context 구현 때와 마찬가지로, 먼저 전역 상태를 공급할 껍데기를 만들고 다음 단계에서 실제 로직을 채운다.

## 예시 코드

### store/redux/store.js

```jsx
import { configureStore } from '@reduxjs/toolkit';

const store = configureStore({
  reducer: {},
});

export default store;
```

### App.js

```jsx
import { NavigationContainer } from '@react-navigation/native';
import { Provider } from 'react-redux';

import store from './store/redux/store';

export default function App() {
  return (
    <Provider store={store}>
      <NavigationContainer>
        {/* navigators and screens */}
      </NavigationContainer>
    </Provider>
  );
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### Redux Toolkit과 React Redux는 역할이 다르다

두 패키지를 함께 설치하지만 담당하는 역할은 다르다.

```text
@reduxjs/toolkit
-> Redux store와 상태 로직을 만드는 도구

react-redux
-> React 컴포넌트와 Redux store를 연결하는 도구
```

React Native에서도 이 구분은 동일하다.

### Redux store는 앱의 중앙 상태 저장소다

Context 구현에서는 `FavoritesContextProvider` 안의 `useState`가 상태를 관리했다.

Redux 구현에서는 store가 앱 전역 상태의 중심이 된다.

```text
Context 방식
-> Provider 컴포넌트 내부 useState

Redux 방식
-> Redux store
-> reducer
-> action dispatch
```

이번 강의에서는 아직 store만 만들었고, 실제 상태 조각은 다음 단계에서 추가한다.

### Provider는 여전히 앱의 상태 경계다

Context든 Redux든 Provider의 위치는 중요하다.

```text
Provider 아래
-> 전역 상태 접근 가능

Provider 밖
-> 전역 상태 접근 불가
```

여러 네비게이터와 화면이 store를 사용해야 하므로, 앱 루트 근처에서 `NavigationContainer`를 감싸는 방식이 적절하다.

### React Native라고 Redux 사용법이 달라지지는 않는다

Redux Toolkit과 React Redux의 기본 사용법은 React 웹 앱과 같다.

```text
store 생성
-> Provider로 앱 감싸기
-> useSelector로 상태 읽기
-> useDispatch로 action dispatch
```

React Native에서는 렌더링 대상이 DOM이 아니라 네이티브 컴포넌트일 뿐, Redux의 상태 관리 방식은 동일하다.

## 정리

이번 강의에서는 Context API로 구현했던 Favorites 기능을 Redux 방식으로 다시 구현하기 위한 준비를 했다.

`@reduxjs/toolkit`과 `react-redux`를 설치하고, `store/redux/store.js`에서 `configureStore()`로 Redux store를 만들었다.

또한 `App.js`에서 `react-redux`의 `Provider`로 `NavigationContainer` 전체를 감싸고, `store` prop으로 생성한 store를 전달했다.

아직 reducer와 action 로직은 없으므로, 다음 단계에서 favorites 상태를 관리할 Redux slice를 추가한다.
