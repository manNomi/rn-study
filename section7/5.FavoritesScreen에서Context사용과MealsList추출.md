# 5. FavoritesScreen에서 Context 사용과 MealsList 추출

## 핵심 요약

- `FavoritesScreen`은 더 이상 더미 텍스트가 아니라 즐겨찾기 meal 목록을 보여줘야 한다.
- 이미 `MealsOverviewScreen`에는 meal 목록을 렌더링하는 `FlatList` 로직이 있다.
- 같은 목록 UI를 여러 화면에서 사용하기 위해 `MealsList` 컴포넌트로 추출한다.
- `MealItem`은 목록 안에서만 사용되므로 `components/MealsList` 폴더로 함께 옮긴다.
- `MealsList`는 `items` prop을 받아 해당 meal 배열을 `FlatList`로 렌더링한다.
- `MealsOverviewScreen`은 필터링된 `displayedMeals`를 `MealsList`에 전달한다.
- `FavoritesScreen`은 `useContext(FavoritesContext)`로 즐겨찾기 id 배열을 읽고, `MEALS` 원본 데이터에서 즐겨찾기 meal만 필터링한다.
- 즐겨찾기 목록이 비어 있으면 빈 리스트 대신 안내 메시지를 보여준다.
- `MealItem` 안에 detail 화면으로 이동하는 로직이 이미 있으므로, `FavoritesScreen`에서 렌더링한 meal도 상세 화면으로 이동할 수 있다.

## 학습 목표

- 화면에 묶여 있던 목록 렌더링 로직을 재사용 컴포넌트로 추출하는 방법을 익힌다.
- `MealsList` 컴포넌트를 만들어 `MealsOverviewScreen`과 `FavoritesScreen`에서 함께 사용하는 구조를 이해한다.
- Context의 `ids` 배열을 기반으로 원본 `MEALS` 데이터를 필터링하는 방법을 익힌다.
- 즐겨찾기 데이터가 비어 있을 때 empty state를 렌더링하는 패턴을 이해한다.
- 네비게이션 로직이 `MealItem` 안에 있을 때 재사용 목록에서도 상세 이동이 유지되는 이유를 이해한다.

## 주요 내용

### FavoritesScreen의 목표

현재 `FavoritesScreen`은 더미 텍스트만 출력한다.

```jsx
<Text>The favorites screen!</Text>
```

하지만 실제 목표는 사용자가 즐겨찾기로 저장한 meal 목록을 보여주는 것이다.

```text
MealDetailScreen
-> 별 아이콘으로 favorite 상태 변경

FavoritesScreen
-> favorite 상태를 읽음
-> favorite meal 목록 출력
```

이제 Context에 저장된 즐겨찾기 id 배열을 사용해 화면을 구성해야 한다.

### 이미 존재하는 meal 목록 UI

Meals 앱에는 이미 meal 목록을 출력하는 화면이 있다.

카테고리를 선택하면 `MealsOverviewScreen`이 해당 카테고리에 속한 meal들을 보여준다.

이 화면에는 다음 로직이 들어 있었다.

```text
FlatList 렌더링
renderMealItem 함수
MealItem 컴포넌트 렌더링
목록 컨테이너 스타일
```

이 로직은 특정 카테고리 화면에만 묶여 있을 필요가 없다.

즐겨찾기 화면에서도 같은 형태의 meal 목록을 보여줘야 하기 때문이다.

### MealsList 컴포넌트로 추출

목록 렌더링 로직을 재사용하기 위해 새 컴포넌트를 만든다.

```text
components/
  MealsList/
    MealsList.js
    MealItem.js
```

`MealItem`은 meal 목록에서만 사용되므로 `MealsList` 폴더 안으로 옮긴다.

이렇게 하면 목록 관련 컴포넌트들이 한곳에 모인다.

```text
MealsList.js
-> FlatList와 목록 렌더링 담당

MealItem.js
-> 개별 meal 카드 담당
```

### MealsList의 책임

`MealsList` 컴포넌트는 meal 배열을 받아 목록을 렌더링한다.

```jsx
function MealsList({ items }) {
  ...
}
```

`items`는 이 컴포넌트가 화면에 출력할 meal 배열이다.

```text
MealsOverviewScreen
-> categoryId로 필터링한 meals 전달

FavoritesScreen
-> favorite ids로 필터링한 meals 전달
```

즉 `MealsList`는 데이터가 어디에서 왔는지 알 필요가 없다.

받은 `items`를 리스트로 보여주는 일만 맡는다.

### renderMealItem 이동

기존 `MealsOverviewScreen`에 있던 `renderMealItem` 함수도 `MealsList`로 옮긴다.

```jsx
function renderMealItem(itemData) {
  const item = itemData.item;

  const mealItemProps = {
    id: item.id,
    title: item.title,
    imageUrl: item.imageUrl,
    affordability: item.affordability,
    complexity: item.complexity,
    duration: item.duration,
  };

  return <MealItem {...mealItemProps} />;
}
```

이 함수는 특정 화면의 상태에 의존하지 않고, item 데이터를 `MealItem` props로 바꿔 주는 역할만 한다.

따라서 별도 목록 컴포넌트로 옮기기 적합하다.

### FlatList도 MealsList 안으로 이동

`FlatList` 렌더링도 `MealsList` 안으로 들어간다.

```jsx
return (
  <View style={styles.container}>
    <FlatList
      data={items}
      keyExtractor={(item) => item.id}
      renderItem={renderMealItem}
    />
  </View>
);
```

`data`에는 `items` prop을 사용한다.

기존에는 `displayedMeals`라는 화면 내부 변수를 사용했지만, 이제 `MealsList`는 범용 컴포넌트이므로 더 일반적인 이름인 `items`를 받는다.

### 필요한 import

`MealsList.js`에서는 React Native 컴포넌트와 스타일 도구가 필요하다.

```jsx
import { FlatList, StyleSheet, View } from 'react-native';
```

그리고 같은 폴더에 있는 `MealItem`도 가져온다.

```jsx
import MealItem from './MealItem';
```

`MealItem` 파일을 `components/MealsList` 폴더로 옮겼다면 import 경로도 새 위치에 맞게 수정해야 한다.

### MealsList 예시 코드

```jsx
import { FlatList, StyleSheet, View } from 'react-native';

import MealItem from './MealItem';

function renderMealItem(itemData) {
  const item = itemData.item;

  const mealItemProps = {
    id: item.id,
    title: item.title,
    imageUrl: item.imageUrl,
    affordability: item.affordability,
    complexity: item.complexity,
    duration: item.duration,
  };

  return <MealItem {...mealItemProps} />;
}

function MealsList({ items }) {
  return (
    <View style={styles.container}>
      <FlatList
        data={items}
        keyExtractor={(item) => item.id}
        renderItem={renderMealItem}
      />
    </View>
  );
}

export default MealsList;

const styles = StyleSheet.create({
  container: {
    flex: 1,
    padding: 16,
  },
});
```

### MealsOverviewScreen 단순화

이제 `MealsOverviewScreen`은 목록 렌더링을 직접 하지 않는다.

카테고리 id로 meal을 필터링한 뒤 `MealsList`에 전달하기만 한다.

```jsx
import MealsList from '../components/MealsList/MealsList';
```

```jsx
function MealsOverviewScreen({ route, navigation }) {
  const catId = route.params.categoryId;

  const displayedMeals = MEALS.filter((mealItem) => {
    return mealItem.categoryIds.indexOf(catId) >= 0;
  });

  return <MealsList items={displayedMeals} />;
}
```

`FlatList`, `View`, `StyleSheet` 같은 목록 렌더링용 import는 더 이상 필요하지 않다.

`MealsOverviewScreen`의 책임이 훨씬 명확해진다.

```text
MealsOverviewScreen
-> categoryId로 데이터 선택
-> MealsList에 전달

MealsList
-> 전달받은 meal 배열 렌더링
```

### FavoritesScreen에서 Context 사용

이제 `FavoritesScreen`에서도 `MealsList`를 사용할 수 있다.

먼저 Context를 읽는다.

```jsx
import { useContext } from 'react';
import { FavoritesContext } from '../store/context/favorites-context';
```

```jsx
const favoriteMealsContext = useContext(FavoritesContext);
```

`favoriteMealsContext.ids`에는 즐겨찾기로 저장된 meal id들이 들어 있다.

### 원본 MEALS 데이터 필터링

즐겨찾기 화면에는 전체 meal 중에서 favorite id에 포함된 것만 보여줘야 한다.

따라서 원본 데이터 `MEALS`를 가져온다.

```jsx
import { MEALS } from '../data/dummy-data';
```

그리고 `filter`와 `includes`를 조합한다.

```jsx
const favoriteMeals = MEALS.filter((meal) =>
  favoriteMealsContext.ids.includes(meal.id)
);
```

흐름은 다음과 같다.

```text
MEALS 전체 배열 순회
-> 각 meal.id 확인
-> favoriteMealsContext.ids에 포함되어 있으면 유지
-> 포함되지 않으면 제외
```

결과적으로 `favoriteMeals`에는 즐겨찾기 meal 객체들만 들어간다.

### FavoritesScreen에서 MealsList 렌더링

필터링한 결과를 `MealsList`에 전달한다.

```jsx
return <MealsList items={favoriteMeals} />;
```

이제 `FavoritesScreen`에서도 `MealsOverviewScreen`과 같은 목록 UI를 재사용한다.

```text
FavoritesScreen
-> Context에서 favorite ids 읽기
-> MEALS에서 favorite meal만 필터링
-> MealsList로 렌더링
```

### 상세 화면 이동이 그대로 되는 이유

`MealItem` 안에는 이미 meal detail 화면으로 이동하는 로직이 들어 있다.

```text
MealItem
-> pressable
-> navigation.navigate('MealDetail', { mealId })
```

따라서 `MealItem`이 `MealsOverviewScreen`에서 렌더링되든, `FavoritesScreen`에서 렌더링되든 상세 이동은 그대로 동작한다.

React Navigation 입장에서는 같은 네비게이션 설정 안에 있는 컴포넌트이기 때문이다.

```text
FavoritesScreen
-> Drawer로 열린 화면
-> 전체 Stack Navigator 설정 안에 있음
-> MealItem에서 MealDetail로 이동 가능
```

또한 상세 화면에서 즐겨찾기를 해제하고 뒤로 돌아오면, Context 상태가 바뀌었기 때문에 `FavoritesScreen`의 필터 결과도 업데이트된다.

### 빈 즐겨찾기 상태 처리

즐겨찾기 meal이 하나도 없을 때 빈 리스트만 보여주면 사용자에게 불친절하다.

따라서 `favoriteMeals.length`가 `0`이면 안내 메시지를 렌더링한다.

```jsx
if (favoriteMeals.length === 0) {
  return (
    <View style={styles.rootContainer}>
      <Text style={styles.text}>
        You have no favorite meals yet.
      </Text>
    </View>
  );
}
```

이런 UI를 empty state라고 부를 수 있다.

```text
데이터 있음
-> MealsList 렌더링

데이터 없음
-> 안내 메시지 렌더링
```

### Empty state 스타일링

안내 메시지는 화면 중앙에 표시한다.

```jsx
const styles = StyleSheet.create({
  rootContainer: {
    flex: 1,
    justifyContent: 'center',
    alignItems: 'center',
  },
  text: {
    fontSize: 18,
    fontWeight: 'bold',
    color: 'white',
  },
});
```

`flex: 1`로 가능한 공간을 모두 차지하고, `justifyContent`와 `alignItems`로 수직/수평 중앙 정렬한다.

```text
justifyContent: 'center'
-> 세로 중앙 정렬

alignItems: 'center'
-> 가로 중앙 정렬
```

## 예시 코드

### FavoritesScreen

```jsx
import { useContext } from 'react';
import { StyleSheet, Text, View } from 'react-native';

import MealsList from '../components/MealsList/MealsList';
import { MEALS } from '../data/dummy-data';
import { FavoritesContext } from '../store/context/favorites-context';

function FavoritesScreen() {
  const favoriteMealsContext = useContext(FavoritesContext);

  const favoriteMeals = MEALS.filter((meal) =>
    favoriteMealsContext.ids.includes(meal.id)
  );

  if (favoriteMeals.length === 0) {
    return (
      <View style={styles.rootContainer}>
        <Text style={styles.text}>
          You have no favorite meals yet.
        </Text>
      </View>
    );
  }

  return <MealsList items={favoriteMeals} />;
}

export default FavoritesScreen;

const styles = StyleSheet.create({
  rootContainer: {
    flex: 1,
    justifyContent: 'center',
    alignItems: 'center',
  },
  text: {
    fontSize: 18,
    fontWeight: 'bold',
    color: 'white',
  },
});
```

### MealsOverviewScreen

```jsx
import MealsList from '../components/MealsList/MealsList';
import { MEALS } from '../data/dummy-data';

function MealsOverviewScreen({ route }) {
  const catId = route.params.categoryId;

  const displayedMeals = MEALS.filter((mealItem) => {
    return mealItem.categoryIds.indexOf(catId) >= 0;

  return <MealsList items={displayedMeals} />;
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### 화면은 데이터 선택, 컴포넌트는 렌더링

이번 리팩터링의 핵심은 책임 분리다.

```text
MealsOverviewScreen
-> categoryId로 어떤 meals를 보여줄지 결정

FavoritesScreen
-> Context ids로 어떤 meals를 보여줄지 결정

MealsList
-> 전달받은 meals를 어떻게 보여줄지 담당
```

이렇게 나누면 같은 UI를 여러 데이터 소스에 재사용할 수 있다.

### Context 상태는 다른 화면의 렌더링도 바꾼다

`MealDetailScreen`에서 즐겨찾기를 추가하거나 제거하면 Context 상태가 바뀐다.

`FavoritesScreen`은 같은 Context를 읽고 있으므로, 그 상태에 따라 렌더링 결과가 달라진다.

```text
MealDetailScreen
-> context.addFavorite('m1')

FavoritesScreen
-> context.ids에 'm1' 포함
-> MEALS에서 m1 필터링
-> MealsList에 표시
```

이것이 앱 전역 상태를 사용하는 핵심 이유다.

### UI 재사용과 네비게이션 재사용이 함께 일어난다

`MealsList`를 재사용하면 목록 UI가 재사용된다.

`MealItem` 안의 navigation 로직도 함께 유지되므로 즐겨찾기 목록에서도 상세 화면으로 이동할 수 있다.

```text
MealsOverviewScreen의 MealItem
-> 상세 이동 가능

FavoritesScreen의 MealItem
-> 같은 MealItem이므로 상세 이동 가능
```

단, 이런 구조는 `MealItem`이 네비게이션에 직접 의존한다는 뜻이기도 하다.

작은 앱에서는 편리하지만, 더 큰 앱에서는 navigation handler를 prop으로 넘기는 식으로 분리할 수도 있다.

### Empty state는 실제 앱 품질의 일부다

데이터가 없을 때 아무것도 보여주지 않으면 사용자는 앱이 깨졌다고 느낄 수 있다.

```text
나쁜 상태
-> 빈 화면

좋은 상태
-> "You have no favorite meals yet."
```

앱 전역 상태를 다룰 때는 데이터가 있는 경우뿐 아니라 비어 있는 경우도 UI로 명확히 표현해야 한다.

## 정리

이번 강의에서는 `MealsOverviewScreen`에 있던 meal 목록 렌더링 로직을 `MealsList` 컴포넌트로 추출했다.

그 결과 `MealsOverviewScreen`은 카테고리별 meal을, `FavoritesScreen`은 Context의 favorite ids를 기준으로 필터링한 meal을 같은 `MealsList`로 렌더링할 수 있게 되었다.

또한 `FavoritesScreen`에서 `useContext(FavoritesContext)`로 즐겨찾기 id 배열을 읽고, `MEALS.filter()`와 `includes()`를 사용해 실제 favorite meal 목록을 만들었다.

마지막으로 즐겨찾기 목록이 비어 있을 때 안내 메시지를 보여주는 empty state까지 추가해, Context API 기반 즐겨찾기 기능의 기본 흐름을 완성했다.
