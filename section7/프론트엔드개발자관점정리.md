# 섹션 7. 프론트엔드 개발자 관점 정리

## 한 줄 요약

여러 화면이 공유해야 하는 즐겨찾기 상태를 Context API와 Redux Toolkit 두 방식으로 구현하며, React Native에서도 앱 전역 상태 관리는 React 웹 앱에서 쓰던 Context/Redux 지식이 거의 그대로 적용된다는 점을 확인하는 섹션이다.

## 이미 알고 있다고 보는 것

- React 함수 컴포넌트와 hooks
- `useState`
- `createContext`, `useContext`
- Provider 패턴
- Redux의 store, reducer, action 개념
- Redux Toolkit의 `configureStore`, `createSlice`
- React Redux의 `Provider`, `useSelector`, `useDispatch`
- 배열 필터링과 id 기반 데이터 선택
- React Navigation의 screen과 route params

## 새로 잡아야 하는 관점

React Native라고 해서 전역 상태 관리 방식이 달라지지 않는다.

상태 관리 관점에서는 React 웹 앱과 거의 같다.

```text
React 웹 앱
-> Context API
-> Redux Toolkit
-> Provider
-> selector / dispatch

React Native 앱
-> Context API
-> Redux Toolkit
-> Provider
-> selector / dispatch
```

차이는 렌더링 대상이다.

웹에서는 DOM을 업데이트하고, React Native에서는 `View`, `Text`, `FlatList` 같은 네이티브 컴포넌트를 업데이트한다.

## 왜 앱 전역 상태가 필요한가

Meals 앱에는 두 화면이 같은 상태를 공유해야 한다.

```text
MealDetailScreen
-> 현재 meal을 favorite으로 추가/삭제
-> 현재 meal이 favorite인지에 따라 별 아이콘 변경

FavoritesScreen
-> favorite meal 목록 표시
-> favorite이 없으면 empty state 표시
```

이 상태는 특정 화면 내부에만 있으면 안 된다.

```text
로컬 상태
-> 한 화면에서만 필요할 때

앱 전역 상태
-> 여러 화면에서 읽고 변경할 때
```

이번 섹션의 전역 상태는 favorite meal id 배열이다.

```jsx
['m1', 'm3']
```

## params와 전역 상태의 차이

섹션 6에서는 `route.params`로 `categoryId`, `mealId`를 전달했다.

이 값들은 특정 화면을 열기 위한 식별자다.

```text
route.params
-> 현재 화면을 그리기 위한 작은 식별자
-> categoryId
-> mealId
```

반면 favorite 목록은 여러 화면에서 계속 변하고 읽히는 상태다.

```text
app-wide state
-> favoriteMealIds
-> 로그인 사용자
-> 장바구니
-> 테마
```

프론트엔드 개발자 관점에서는 URL param과 global store를 구분하는 것과 같다.

## Context API 방식

먼저 Context로 구현했다.

```text
store/context/favorites-context.js
```

Context 객체는 상태의 공개 API를 정의했다.

```jsx
export const FavoritesContext = createContext({
  ids: [],
  addFavorite: (id) => {},
  removeFavorite: (id) => {},
});
```

Provider는 실제 상태와 함수를 관리했다.

```jsx
function FavoritesContextProvider({ children }) {
  const [favoriteMealIds, setFavoriteMealIds] = useState([]);

  function addFavorite(id) {
    setFavoriteMealIds((currentFavIds) => [
      ...currentFavIds,
      id,
    ]);
  }

  function removeFavorite(id) {
    setFavoriteMealIds((currentFavIds) =>
      currentFavIds.filter((mealId) => mealId !== id)
    );
  }

  const value = {
    ids: favoriteMealIds,
    addFavorite,
    removeFavorite,
  };

  return (
    <FavoritesContext.Provider value={value}>
      {children}
    </FavoritesContext.Provider>
  );
}
```

Context 자체가 상태를 자동 관리하는 것은 아니다.

상태는 `useState`가 관리하고, Context는 그 상태와 함수를 멀리 있는 화면까지 전달하는 통로다.

## Context 소비

`MealDetailScreen`에서는 Context를 읽고 favorite 여부를 계산했다.

```jsx
const favoriteMealsContext = useContext(FavoritesContext);

const mealIsFavorite =
  favoriteMealsContext.ids.includes(mealId);
```

별 아이콘을 누르면 상태에 따라 추가 또는 제거했다.

```jsx
if (mealIsFavorite) {
  favoriteMealsContext.removeFavorite(mealId);
} else {
  favoriteMealsContext.addFavorite(mealId);
}
```

`FavoritesScreen`에서는 favorite ids로 원본 데이터를 필터링했다.

```jsx
const favoriteMeals = MEALS.filter((meal) =>
  favoriteMealsContext.ids.includes(meal.id)
);
```

## Redux Toolkit 방식

같은 기능을 Redux Toolkit으로도 구현했다.

```text
store/redux/store.js
store/redux/favorites.js
```

store는 `configureStore`로 만들었다.

```jsx
const store = configureStore({
  reducer: {
    favoriteMeals: favoritesReducer,
  },
});
```

favorite 상태 로직은 slice로 분리했다.

```jsx
const favoritesSlice = createSlice({
  name: 'favorites',
  initialState: {
    ids: [],
  },
  reducers: {
    addFavorite: (state, action) => {
      state.ids.push(action.payload.id);
    },
    removeFavorite: (state, action) => {
      const index = state.ids.indexOf(action.payload.id);
      state.ids.splice(index, 1);
    },
  },
});
```

Redux Toolkit은 내부적으로 불변 업데이트를 처리하므로 reducer 안에서 mutation처럼 보이는 코드를 작성할 수 있다.

```text
state.ids.push(...)
state.ids.splice(...)
```

일반 Redux만 사용할 때의 reducer 규칙과 헷갈리지 않도록 Redux Toolkit의 특징으로 기억해야 한다.

## Redux 소비

`MealDetailScreen`에서는 `useSelector`로 favorite ids를 읽었다.

```jsx
const favoriteMealIds = useSelector(
  (state) => state.favoriteMeals.ids
);
```

상태 변경은 `dispatch`로 요청했다.

```jsx
const dispatch = useDispatch();

dispatch(addFavorite({ id: mealId }));
dispatch(removeFavorite({ id: mealId }));
```

`FavoritesScreen`도 같은 selector를 사용했다.

```jsx
const favoriteMealIds = useSelector(
  (state) => state.favoriteMeals.ids
);

const favoriteMeals = MEALS.filter((meal) =>
  favoriteMealIds.includes(meal.id)
);
```

## Context와 Redux 비교

두 방식 모두 같은 결과를 만들었다.

```text
MealDetailScreen에서 favorite 토글
FavoritesScreen에서 favorite 목록 표시
```

차이는 상태 변경 흐름이다.

```text
Context
-> Provider 내부 useState
-> useContext로 상태와 함수 읽기
-> 함수 직접 호출

Redux
-> 중앙 store
-> createSlice로 reducer/action 정의
-> useSelector로 상태 읽기
-> useDispatch로 action dispatch
```

Context는 더 직접적이다.

Redux는 더 구조적이고 명시적이다.

작은 상태에는 Context가 충분할 수 있고, 상태 도메인이 많거나 변경 흐름을 명확히 추적해야 하는 앱에서는 Redux가 유리할 수 있다.

## Provider 위치

두 방식 모두 Provider를 앱 루트 근처에 배치했다.

```jsx
<FavoritesContextProvider>
  <NavigationContainer>
    ...
  </NavigationContainer>
</FavoritesContextProvider>
```

또는 Redux 방식에서는 다음과 같다.

```jsx
<Provider store={store}>
  <NavigationContainer>
    ...
  </NavigationContainer>
</Provider>
```

핵심은 상태가 필요한 모든 화면이 Provider 아래에 있어야 한다는 것이다.

```text
Provider
-> NavigationContainer
   -> Stack
      -> Drawer
         -> FavoritesScreen
      -> MealDetailScreen
```

웹에서 `App` 루트에 `QueryClientProvider`, `ThemeProvider`, `Redux Provider`를 두는 감각과 같다.

## MealsList 추출의 의미

이번 섹션에서는 `MealsOverviewScreen`의 목록 렌더링을 `MealsList`로 추출했다.

이 작업은 단순 리팩터링이 아니다.

상태 관리 방식을 바꿔도 UI 렌더링 컴포넌트는 그대로 재사용할 수 있게 만든다.

```text
MealsOverviewScreen
-> categoryId로 meals 선택
-> MealsList 렌더링

FavoritesScreen
-> favorite ids로 meals 선택
-> MealsList 렌더링
```

화면 컴포넌트는 “어떤 데이터를 보여줄지”를 결정하고, `MealsList`는 “어떻게 보여줄지”를 담당한다.

이 분리가 되어 있으면 Context에서 Redux로 바꿀 때 수정 범위가 줄어든다.

## Empty state

즐겨찾기가 없을 때 빈 화면을 보여주지 않고 안내 메시지를 렌더링했다.

```jsx
if (favoriteMeals.length === 0) {
  return (
    <View style={styles.rootContainer}>
      <Text style={styles.text}>
        You have no favorite meals yet.
      </Text>
    </View>
  );
}
```

전역 상태를 다루는 화면에서는 데이터가 있을 때뿐 아니라 없을 때의 UX도 중요하다.

```text
데이터 있음
-> 목록 렌더링

데이터 없음
-> empty state
```

## 실무 체크리스트

- 이 상태가 정말 여러 화면에서 필요한가?
- `route.params`로 넘길 값과 전역 상태로 둘 값을 구분했는가?
- Provider가 상태를 필요한 모든 화면보다 높은 위치에 있는가?
- Context value의 읽기 값과 변경 함수 이름이 명확한가?
- Redux store key와 selector 경로가 일관적인가?
- action payload 구조가 reducer와 dispatch 쪽에서 맞는가?
- UI 컴포넌트와 상태 선택 로직이 적절히 분리되어 있는가?
- empty state를 처리했는가?
- Context로 충분한 문제에 Redux를 과하게 쓰고 있지는 않은가?
- Redux가 필요한 규모인데 Context Provider에 너무 많은 로직을 몰아넣고 있지는 않은가?

## 정리

섹션 7은 React Native 앱에서 앱 전역 상태를 다루는 방법을 정리한 구간이다.

중요한 결론은 Context API와 Redux Toolkit 모두 React Native에서 특별한 방식으로 쓰는 것이 아니라, React 웹 앱에서 쓰던 방식과 거의 동일하게 적용된다는 점이다.

React Native 개발자는 네이티브 UI 컴포넌트만 배우는 것이 아니라, React의 상태 관리 감각을 여러 화면을 가진 모바일 앱 구조에 연결할 수 있어야 한다.
