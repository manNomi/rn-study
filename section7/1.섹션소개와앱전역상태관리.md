# 1. 섹션 소개와 앱 전역 상태 관리

## 핵심 요약

- 이전 섹션에서 React Navigation을 추가하면서 앱이 여러 화면으로 구성되었다.
- Meals 앱에는 `MealDetailScreen`의 즐겨찾기 아이콘과 `FavoritesScreen`이 있지만, 아직 실제 기능은 연결되어 있지 않다.
- 즐겨찾기 기능을 구현하려면 한 화면에서 바꾼 상태를 다른 화면에서도 읽을 수 있어야 한다.
- 이런 상태를 앱 전역 상태 또는 app-wide state라고 부른다.
- 앱 전역 상태는 React 웹 앱에서도 중요하고, React Native 앱에서도 동일하게 중요하다.
- 이번 섹션에서는 React Context API와 Redux를 사용해 React Native 앱에서 앱 전역 상태를 관리하는 방법을 살펴본다.
- 단, Context와 Redux 자체의 기초 개념을 깊게 설명하는 섹션은 아니다.
- 이번 섹션의 초점은 이미 알고 있는 Context/Redux를 React Native 앱에 어떻게 적용하는지에 있다.

## 학습 목표

- 여러 화면을 가진 앱에서 앱 전역 상태가 왜 필요한지 이해한다.
- Meals 앱의 즐겨찾기 기능이 어떤 상태 관리 문제를 만드는지 파악한다.
- `MealDetailScreen`과 `FavoritesScreen` 사이에 공유되어야 하는 데이터 흐름을 이해한다.
- React Context API와 Redux가 앱 전역 상태 관리 도구로 사용될 수 있음을 정리한다.
- React Native에서의 상태 관리가 React 웹 앱과 본질적으로 같은 React 개념 위에 있다는 점을 이해한다.
- 이번 섹션에서 다루는 범위와 다루지 않는 범위를 구분한다.

## 주요 내용

### 이전 섹션에서 만든 기반

이전 섹션에서는 React Navigation을 사용해 Meals 앱에 여러 화면을 추가했다.

```text
Categories
-> MealsOverview
-> MealDetail

Drawer
-> Categories
-> Favorites
```

이제 앱은 단일 화면 앱이 아니라 여러 화면을 오가는 구조가 되었다.

또한 `MealDetailScreen`에는 별 모양 아이콘 버튼이 있고, Drawer에는 `FavoritesScreen`도 추가했다.

하지만 아직 이 둘은 실제로 연결되어 있지 않다.

```text
MealDetailScreen의 별 아이콘
-> 현재는 실제 즐겨찾기 상태를 바꾸지 않음

FavoritesScreen
-> 현재는 실제 즐겨찾기 목록을 보여주지 않음
```

이번 섹션은 바로 이 빈 기능을 채우면서 시작한다.

### 즐겨찾기 기능이 요구하는 상태

즐겨찾기 기능을 제대로 만들려면 다음 흐름이 필요하다.

```text
1. 사용자가 MealDetailScreen에서 별 아이콘을 누른다.
2. 해당 meal이 즐겨찾기로 저장된다.
3. 같은 meal을 다시 누르면 즐겨찾기에서 제거될 수 있다.
4. FavoritesScreen에서는 즐겨찾기로 저장된 meal 목록을 보여준다.
5. MealDetailScreen은 현재 meal이 즐겨찾기인지 아닌지도 알아야 한다.
```

즉 한 화면에서 상태를 변경하고, 다른 화면에서 그 상태를 읽어야 한다.

```text
MealDetailScreen
-> favorite 여부 변경

FavoritesScreen
-> favorite meal 목록 읽기
```

이 상태는 특정 화면 하나에만 속하지 않는다.

앱의 여러 부분에서 필요하므로 더 넓은 범위에서 관리해야 한다.

### 로컬 상태만으로는 부족한 이유

React의 `useState`는 컴포넌트 내부 상태를 관리하기 좋다.

예를 들어 입력값, 모달 열림 여부, 특정 화면 안에서만 쓰이는 UI 상태는 로컬 상태로 충분하다.

```text
로컬 상태에 적합한 예
-> 입력 중인 텍스트
-> 현재 화면의 로딩 토글
-> 특정 컴포넌트의 펼침/접힘 상태
```

하지만 즐겨찾기 상태는 다르다.

`MealDetailScreen` 안에서만 `useState`로 즐겨찾기 여부를 관리하면 `FavoritesScreen`이 그 값을 알 수 없다.

```text
MealDetailScreen 내부 useState
-> MealDetailScreen 안에서는 사용 가능
-> FavoritesScreen에서는 직접 접근 불가
```

물론 props로 끌어올려서 전달할 수도 있지만, 화면 구조가 복잡해지면 prop drilling이 발생한다.

특히 navigation으로 연결된 여러 화면 사이에서는 단순 부모-자식 관계처럼 props를 전달하기 어렵다.

그래서 앱 전체 또는 큰 화면 그룹에서 공유할 수 있는 상태 관리 방식이 필요하다.

### 앱 전역 상태란?

앱 전역 상태는 앱의 여러 화면이나 여러 컴포넌트에서 함께 사용하는 상태를 의미한다.

```text
앱 전역 상태 예시
-> 로그인한 사용자 정보
-> 인증 토큰
-> 테마 설정
-> 장바구니 목록
-> 즐겨찾기 목록
-> 앱 설정값
```

이번 Meals 앱에서는 즐겨찾기 meal id 목록이 앱 전역 상태가 된다.

```text
favoriteMealIds
-> MealDetailScreen에서 추가/삭제
-> FavoritesScreen에서 목록 렌더링
```

핵심은 “상태가 어디에서 변경되고, 어디에서 읽히는가?”를 보는 것이다.

여러 화면에서 필요하다면 앱 전역 상태 후보가 된다.

### React 웹 앱과 같은 문제

이 문제는 React Native에만 있는 문제가 아니다.

React 웹 앱에서도 여러 페이지나 여러 컴포넌트가 같은 데이터를 필요로 하면 앱 전역 상태 관리가 필요하다.

```text
React 웹 앱
-> 여러 라우트에서 사용자 정보 필요
-> 장바구니 상태를 여러 페이지에서 읽음
-> 테마 상태를 앱 전체에 적용

React Native 앱
-> 여러 화면에서 사용자 정보 필요
-> FavoritesScreen과 MealDetailScreen이 favorite 상태 공유
-> 테마나 설정을 앱 전체에 적용
```

React Native는 UI 렌더링 대상이 웹 DOM이 아니라 네이티브 UI일 뿐, React의 상태 관리 문제는 동일하게 존재한다.

### 이번 섹션에서 사용할 도구

이번 섹션에서는 두 가지 대표적인 앱 전역 상태 관리 방법을 살펴본다.

```text
React Context API
Redux
```

Context API는 React에 내장된 기능이다.

컴포넌트 트리의 상위에서 값을 제공하고, 하위 컴포넌트들이 필요한 값을 꺼내 쓸 수 있게 해 준다.

Redux는 별도 상태 관리 라이브러리다.

중앙 store를 두고 action과 reducer를 통해 상태 변경을 관리한다.

둘 다 React Native 앱에서도 사용할 수 있다.

React Native 앱이라고 해서 상태 관리 방식이 완전히 달라지는 것은 아니다.

### 이번 섹션의 초점

강의에서는 Context와 Redux를 처음부터 깊게 설명하지 않는다.

Context와 Redux는 React의 기본 또는 중급 개념에 해당하고, 이 과정에서는 선행 지식으로 전제한다.

이번 섹션의 초점은 다음이다.

```text
이미 알고 있는 Context API를 React Native 앱에 적용하기
이미 알고 있는 Redux를 React Native 앱에 적용하기
Meals 앱의 Favorites 기능에 연결하기
```

즉 “Context가 무엇인가?”, “Redux reducer란 무엇인가?”를 처음부터 배우는 섹션이 아니라, React Native 프로젝트에서 실제로 어떻게 연결하는지를 보는 섹션이다.

### Context와 Redux 중 무엇을 써야 할까?

강의에서는 Context와 Redux를 모두 다룰 예정이다.

어떤 도구가 항상 정답인 것은 아니다.

앱 규모, 상태 변경 복잡도, 팀 선호도, 디버깅 요구사항에 따라 선택이 달라질 수 있다.

```text
Context API
-> React 내장 기능
-> 작은 규모 또는 단순 공유 상태에 적합
-> 별도 라이브러리 불필요

Redux
-> 명시적인 상태 변경 흐름
-> 큰 앱이나 복잡한 상태 로직에 유리
-> 디버깅과 도구 생태계가 강함
```

이번 섹션에서는 두 방식을 모두 Meals 앱에 적용해 보면서 차이를 경험한다.

## 예시 흐름

### 현재 상태

```text
MealDetailScreen
-> 별 아이콘 있음
-> 누르면 아직 실제 favorite 상태 변경 없음

FavoritesScreen
-> 화면은 있음
-> 아직 favorite meal 목록 없음
```

### 구현하고 싶은 상태

```text
MealDetailScreen
-> 별 아이콘 누름
-> 현재 meal id를 favorites에 추가 또는 제거
-> 아이콘 상태도 favorite 여부에 따라 달라질 수 있음

FavoritesScreen
-> favorites 상태를 읽음
-> favorite meal 목록을 필터링해서 렌더링
```

### 필요한 공유 상태

```jsx
const favoriteMealIds = ['m1', 'm3'];
```

이 배열은 한 화면 안에만 있으면 안 된다.

앱의 여러 화면에서 접근할 수 있어야 한다.

## 프론트엔드 개발자 관점에서의 핵심 차이

### 화면이 늘어나면 상태의 위치가 중요해진다

단일 화면에서는 `useState`만으로도 충분한 경우가 많다.

하지만 화면이 늘어나면 상태를 어디에 둘지 결정하는 일이 중요해진다.

```text
한 컴포넌트만 사용
-> 로컬 상태

부모와 자식 몇 개가 사용
-> props 또는 state lifting

여러 화면이 사용
-> 앱 전역 상태 고려
```

상태 관리 도구를 선택하기 전에 상태의 사용 범위를 먼저 봐야 한다.

### Navigation params와 app-wide state는 다르다

이전 섹션에서 `route.params`로 화면 간 데이터를 전달했다.

하지만 params는 주로 “이 화면을 열기 위해 필요한 식별자나 작은 데이터”를 전달하는 데 적합하다.

```text
route.params
-> categoryId
-> mealId
-> 현재 화면을 그리기 위한 식별자
```

즐겨찾기 목록처럼 여러 화면에서 계속 바뀌고 읽히는 데이터는 params로 관리하기 어렵다.

```text
app-wide state
-> favoriteMealIds
-> 로그인 사용자
-> 장바구니
```

따라서 params와 앱 전역 상태를 구분해야 한다.

### React Native에서도 React 상태 관리 지식이 그대로 중요하다

React Native는 모바일 앱을 만들지만, 컴포넌트와 상태 관리는 React 위에서 동작한다.

따라서 Context, Redux, reducer, provider, store 같은 React 상태 관리 개념은 React Native에서도 그대로 중요하다.

```text
React 웹 앱에서 배운 상태 관리
-> React Native 앱에서도 거의 그대로 적용
```

차이는 DOM 대신 네이티브 컴포넌트를 렌더링한다는 점이지, 상태 관리의 핵심 원리가 바뀌는 것은 아니다.

### 이번 섹션은 상태 관리 도구의 RN 적용편이다

Context와 Redux 자체를 처음 배우는 섹션은 아니다.

오히려 다음 질문에 답하는 섹션에 가깝다.

```text
React Native 프로젝트에서 Provider를 어디에 둘까?
NavigationContainer와 Provider는 어떻게 배치할까?
화면 컴포넌트에서 Context나 Redux 상태를 어떻게 읽을까?
헤더 버튼의 onPress와 전역 상태 변경을 어떻게 연결할까?
```

이런 질문들이 이번 섹션의 실제 실습 포인트가 된다.

## 정리

이번 섹션에서는 여러 화면이 함께 사용하는 상태, 즉 앱 전역 상태를 관리하는 방법을 배운다.

Meals 앱에서는 `MealDetailScreen`의 별 아이콘으로 즐겨찾기 상태를 바꾸고, `FavoritesScreen`에서 그 즐겨찾기 목록을 보여주는 기능이 대표 예시가 된다.

이를 위해 React Context API와 Redux를 React Native 앱에 적용해 보고, 네비게이션으로 구성된 여러 화면 사이에서 상태를 공유하는 방법을 익힌다.
