# 2. 에러 메시지 읽기

## 핵심 요약

- 디버깅의 첫 단계는 에러 메시지를 차분히 읽고, 문제가 발생한 컴포넌트와 prop을 찾는 것이다.
- React Native의 에러 메시지는 완벽하지 않을 수 있지만, 대부분 문제 해결을 위한 첫 힌트를 제공한다.
- 터미널 로그와 에뮬레이터 화면의 에러 메시지를 함께 확인하면 원인을 더 빨리 좁힐 수 있다.
- 스택트레이스(Stack Trace)는 에러가 어떤 컴포넌트 흐름에서 발생했는지 추적하는 데 도움이 된다.
- 컴포넌트 사용법이 헷갈릴 때는 공식 문서를 확인하는 것이 가장 안전하다.

## 학습 목표

- 의도적으로 에러를 만들고 에러 메시지가 어떻게 표시되는지 확인한다.
- `Image` 컴포넌트의 잘못된 `source` prop 예시를 통해 문제 위치를 찾는 법을 배운다.
- 터미널 로그와 앱 화면의 에러 메시지 차이를 이해한다.
- 스택트레이스를 보고 어떤 컴포넌트를 살펴봐야 하는지 파악한다.
- 에러 메시지가 불친절할 때 공식 문서로 사용법을 확인하는 습관을 익힌다.

## 주요 내용

### 의도적으로 에러 만들기

에러 메시지를 살펴보기 위해 `GoalInput.js`에서 이미지의 `source` 설정을 잘못 바꾼다.

원래 로컬 이미지는 `require()`로 불러와야 한다.

```jsx
<Image
  style={styles.image}
  source={require('../assets/images/goal.png')}
/>
```

하지만 아래처럼 존재하지 않는 문자열 경로를 직접 넣으면 문제가 발생한다.

```jsx
<Image
  style={styles.image}
  source="some/path/to/image.png"
/>
```

이 방식은 웹의 `<img src="...">`처럼 보이지만, React Native의 로컬 이미지 처리 방식과 맞지 않는다.

### 터미널에서 에러 확인하기

파일을 저장하면 Expo 개발 서버가 변경 사항을 반영하고, 터미널에 에러 메시지가 출력된다.

강의에서 확인한 핵심 힌트는 다음과 같다.

```text
Image 컴포넌트의 source prop이 잘못되었다.
유효하지 않은 prop 값을 전달했다.
```

메시지의 마지막 부분은 다소 헷갈릴 수 있다. 예를 들어 특정 타입이 와야 한다는 설명이 완벽하게 직관적이지 않을 수 있다.

그래도 중요한 힌트는 충분하다.

```text
문제가 발생한 컴포넌트: Image
문제가 발생한 prop: source
```

이 두 가지를 알면 확인 범위를 크게 줄일 수 있다.

### 에러 메시지는 완벽하지 않을 수 있다

에러 메시지가 항상 친절하고 정확하게 원인을 설명해 주지는 않는다. 때로는 추상적이거나, 내부 구현 기준의 표현이 섞여 있어 처음 보면 혼란스러울 수 있다.

하지만 에러 메시지는 보통 다음 중 하나 이상의 정보를 준다.

```text
어떤 컴포넌트에서 문제가 났는가
어떤 prop이나 값이 문제인가
어떤 타입이나 형식을 기대했는가
어느 컴포넌트 경로에서 문제가 발생했는가
```

따라서 메시지를 바로 닫지 말고, 문제를 좁힐 단서를 먼저 찾아야 한다.

### 스택트레이스 확인하기

에러 메시지 아래에는 스택트레이스가 표시된다. 스택트레이스는 에러가 발생하기까지 어떤 컴포넌트들이 호출되었는지 보여준다.

첫 줄이 항상 가장 유용한 것은 아니다. 번들된 내부 코드 경로나 React Native 내부 컴포넌트가 먼저 나올 수 있기 때문이다.

더 유용한 정보는 아래쪽에 있는 앱 컴포넌트 이름이다.

```text
Image
GoalInput
App
```

이런 식으로 표시된다면 `App`이 렌더링한 `GoalInput` 안에서 `Image` 관련 문제가 발생했다는 뜻이다.

같은 컴포넌트를 여러 곳에서 사용하는 앱이라면, 스택트레이스는 어떤 인스턴스 흐름에서 문제가 발생했는지 파악하는 데 특히 유용하다.

### 공식 문서 확인하기

`Image`의 `source` prop에 어떤 값을 전달해야 하는지 확실하지 않다면 공식 문서를 확인해야 한다.

검색 예시:

```text
react native image
```

또는 React Native 공식 문서의 Components 목록에서 `Image` 문서를 찾을 수 있다.

공식 문서에서는 다음 내용을 확인할 수 있다.

- 컴포넌트의 기본 사용법
- 필수 또는 선택 prop
- prop에 전달해야 하는 값의 형태
- 로컬 이미지와 네트워크 이미지 사용 예시

에러 메시지로 문제의 위치를 좁히고, 공식 문서로 올바른 사용법을 확인하는 흐름이 중요하다.

### 앱 화면의 경고와 에러

에뮬레이터나 실제 기기 화면에도 에러나 경고가 표시될 수 있다.

때로는 화면 아래쪽에 경고 메시지가 뜨고, 이를 누르면 자세한 내용을 볼 수 있다.

다만 앱 화면에 표시되는 메시지가 터미널 로그보다 덜 유용할 때도 있다. 이번 예시에서도 터미널 로그가 `Image`와 `source` 문제를 더 분명하게 알려 주었다.

그래도 앱 화면의 메시지는 최소한 다음과 같은 방향을 알려줄 수 있다.

```text
앱이 완전히 멈춘 에러인지
실행은 되지만 의도대로 동작하지 않는 경고인지
어떤 기능이나 컴포넌트 근처의 문제인지
```

따라서 터미널과 앱 화면의 메시지를 함께 보는 것이 좋다.

### 에러 수정하기

문제 원인을 파악했다면 다시 올바른 `source` 설정으로 되돌린다.

```jsx
<Image
  style={styles.image}
  source={require('../assets/images/goal.png')}
/>
```

저장하면 Expo가 앱을 다시 업데이트하고, 에러 메시지도 사라진다.

## 정리

이번 강의에서는 `Image` 컴포넌트의 `source` prop을 일부러 잘못 설정해 에러 메시지를 읽는 방법을 살펴봤다.

에러 메시지가 완벽하지 않더라도 컴포넌트 이름, prop 이름, 스택트레이스 같은 힌트를 통해 문제 위치를 좁힐 수 있다. 터미널 로그와 앱 화면의 메시지를 함께 확인하고, 사용법이 헷갈릴 때는 공식 문서를 확인하는 것이 좋은 디버깅 흐름이다.
