# 섹션 3. 프론트엔드 개발자 관점 정리

## 한 줄 요약

React Native 디버깅은 웹 프론트엔드에서 익숙한 에러 메시지 읽기, `console.log`, DevTools 활용 습관을 모바일 앱 실행 환경에 맞게 옮겨 쓰는 과정이다.

## 이미 알고 있다고 보는 것

- 브라우저 DevTools 사용 경험
- `console.log()`로 값과 실행 흐름 확인
- 에러 메시지와 스택트레이스 읽기
- React 컴포넌트 트리 개념
- props와 state 확인의 중요성
- Network 패널로 API 요청 확인
- 개발 서버를 실행한 터미널 로그 확인

## 새로 잡아야 하는 관점

웹 앱은 브라우저에서 실행되므로 브라우저 DevTools가 디버깅의 중심이다.

```text
웹 React 앱
-> 브라우저
-> Chrome DevTools
-> Console / Network / React DevTools 확장
```

React Native 앱은 브라우저가 아니라 iOS 또는 Android 앱으로 실행된다. 그래서 터미널, 에뮬레이터 화면, Expo 개발자 메뉴, 독립 실행형 React DevTools를 함께 사용한다.

```text
React Native 앱
-> Expo 개발 서버
-> 실제 기기 / Emulator / Simulator
-> 터미널 로그
-> 개발자 메뉴
-> React DevTools
```

즉 디버깅의 기본 감각은 같지만, 도구에 접근하는 위치와 실행 환경이 다르다.

## 디버깅의 기본 순서

React Native에서 문제가 생기면 다음 순서로 좁혀 가는 것이 좋다.

```text
1. 에러 메시지를 읽는다.
2. 터미널 로그를 확인한다.
3. 스택트레이스에서 컴포넌트 흐름을 본다.
4. 필요하면 console.log를 추가한다.
5. 개발자 메뉴나 DevTools를 연다.
6. 공식 문서로 컴포넌트 사용법을 확인한다.
```

도구를 많이 아는 것보다 중요한 것은 문제를 차분히 좁혀 가는 습관이다.

## 에러 메시지 읽기

React Native도 에러가 발생하면 터미널과 앱 화면에 메시지를 보여준다.

예를 들어 `Image` 컴포넌트의 `source` prop을 잘못 전달하면 터미널에서 다음 단서를 얻을 수 있다.

```text
문제가 발생한 컴포넌트: Image
문제가 발생한 prop: source
```

에러 메시지가 항상 완벽하지는 않다. 어떤 타입을 기대했는지 같은 설명이 조금 헷갈릴 수도 있다.

그래도 에러 메시지는 보통 다음 힌트를 준다.

```text
어떤 컴포넌트에서 문제가 났는가
어떤 prop이나 값이 문제인가
어떤 파일이나 컴포넌트 흐름에서 발생했는가
```

웹 프론트엔드에서 콘솔 에러를 읽듯이, React Native에서도 에러 메시지를 먼저 읽는 것이 출발점이다.

## 스택트레이스 읽기

스택트레이스는 에러가 어떤 컴포넌트 흐름에서 발생했는지 보여준다.

첫 줄이 항상 유용한 것은 아니다. React Native 내부 코드나 번들 코드가 먼저 나올 수 있기 때문이다.

대신 직접 만든 컴포넌트 이름을 찾는 것이 중요하다.

```text
Image
GoalInput
App
```

이런 흐름이 보이면 `App`이 렌더링한 `GoalInput` 안의 `Image`에서 문제가 발생했다는 식으로 추론할 수 있다.

웹 React에서도 스택트레이스에서 내 코드 위치를 찾는 것과 같은 감각이다.

## console.log 디버깅

React Native에서도 `console.log()`는 그대로 사용할 수 있다.

```jsx
console.log('GoalInput 컴포넌트 렌더링 됨');
```

출력은 `npm start`를 실행한 터미널에서 확인한다.

```text
GoalInput 컴포넌트 렌더링 됨
```

변수나 상태 값도 함께 출력할 수 있다.

```jsx
console.log('현재 입력값:', enteredGoalText);
```

이 방식으로 다음을 확인할 수 있다.

```text
컴포넌트가 언제 렌더링되는지
이벤트 핸들러가 실제로 호출되는지
상태 값이 어떤 시점에 어떤 값을 가지는지
props가 제대로 전달되는지
```

여러 에뮬레이터나 시뮬레이터를 동시에 실행하면 같은 로그가 여러 번 찍힐 수 있다. 앱 인스턴스가 여러 개 실행 중이기 때문이다.

## Expo 개발 서버와 단축키

Expo 개발 서버를 실행한 터미널에서 `?`를 누르면 사용할 수 있는 단축키 목록을 볼 수 있다.

자주 쓰는 단축키는 다음과 같다.

| 단축키 | 역할 |
| --- | --- |
| `a` | Android Emulator에서 앱 열기 |
| `i` | iOS Simulator에서 앱 열기 |
| `r` | 앱 새로고침 |
| `m` | 개발자 메뉴 열기 / 닫기 |
| `?` | 단축키 목록 보기 |

웹 개발에서 터미널은 보통 빌드 로그나 서버 로그를 보는 곳에 가깝다. Expo에서는 터미널이 앱 실행과 개발자 메뉴 접근을 제어하는 입력 창 역할도 한다.

## 개발자 메뉴

React Native 앱에는 개발 중 사용할 수 있는 개발자 메뉴가 있다.

여는 방법은 다음과 같다.

```text
터미널: m
iOS Simulator: Command + D
Android Emulator: Ctrl + M
```

개발자 메뉴에서는 다음 작업을 할 수 있다.

```text
앱 새로고침
홈으로 이동
원격 JavaScript 디버깅 실행
기타 개발 도구 기능 실행
```

브라우저에서 새로고침하거나 DevTools를 여는 것처럼, React Native에서는 개발자 메뉴가 앱 내부 디버깅 기능의 진입점이 된다.

## Chrome DevTools와 원격 디버깅

개발자 메뉴에서 JavaScript 원격 디버깅을 켜면 Chrome DevTools를 사용할 수 있다.

주로 확인하는 패널은 다음이다.

| 패널 | 용도 |
| --- | --- |
| `Console` | `console.log()` 출력 확인 |
| `Network` | 앱에서 발생한 네트워크 요청 확인 |

현재 목표 관리 앱처럼 네트워크 요청이 없는 앱에서는 `Network` 패널이 크게 유용하지 않을 수 있다.

하지만 나중에 백엔드 API를 호출하는 앱에서는 웹 프론트엔드처럼 요청 URL, 상태 코드, 응답 데이터, 요청 시간을 확인할 수 있다.

원격 디버깅은 유용하지만 항상 켜둘 필요는 없다. 환경에 따라 조금 불안정할 수 있고, 앱의 동작이나 표시가 평소와 다르게 보일 수 있다.

## React DevTools

React Native에서는 브라우저 확장 형태의 React DevTools가 아니라 독립 실행형 React DevTools를 사용할 수 있다.

전역 설치 예시는 다음과 같다.

```bash
npm install -g react-devtools
```

macOS에서 권한 문제가 있으면 `sudo`를 붙일 수 있다.

```bash
sudo npm install -g react-devtools
```

Windows에서는 `sudo`를 사용하지 않는다.

설치 후 별도 터미널에서 실행한다.

```bash
react-devtools
```

React DevTools에서는 다음을 볼 수 있다.

```text
컴포넌트 트리
선택한 컴포넌트의 props
선택한 컴포넌트의 state
state 직접 수정에 따른 UI 변화
```

예를 들어 `GoalInput` 컴포넌트를 선택하면 `visible`, `onAddGoal`, `onCancel` 같은 props와 입력값 state를 확인할 수 있다.

웹 React 개발에서 React DevTools로 컴포넌트 props와 state를 보는 것과 같은 목적이다. 차이는 React Native에서는 독립 실행형 도구를 실행해 앱과 연결한다는 점이다.

## 도구별 사용 기준

상황에 따라 사용할 도구를 고르면 된다.

| 상황 | 먼저 볼 것 |
| --- | --- |
| 앱이 에러로 멈춤 | 에러 메시지, 스택트레이스 |
| 값이 예상과 다름 | `console.log()`, React DevTools state |
| 함수가 실행되는지 모르겠음 | `console.log()` |
| props 전달이 의심됨 | React DevTools |
| 네트워크 요청 문제 | Chrome DevTools Network |
| 앱을 강제로 새로고침해야 함 | Expo 터미널 `r`, 개발자 메뉴 |
| 공식 사용법이 헷갈림 | React Native 공식 문서 |

처음부터 복잡한 도구를 켤 필요는 없다. 보통은 에러 메시지와 로그로 시작하고, 더 깊게 봐야 할 때 DevTools를 연다.

## 공식 문서 확인

React Native 공식 문서에는 디버깅 가이드가 따로 있다.

- [React Native Debugging Basics](https://reactnative.dev/docs/debugging.html)

공식 문서에서는 개발자 메뉴, React Native DevTools, LogBox, 성능 모니터 같은 기능을 더 자세히 볼 수 있다.

React Native의 디버깅 도구와 권장 방식은 버전에 따라 조금씩 달라질 수 있으므로, 실제 프로젝트에서는 공식 문서를 함께 확인하는 습관이 좋다.

## 프론트엔드 개발자가 특히 기억할 차이

```text
브라우저 콘솔
-> Expo 터미널 로그 / Chrome DevTools Console

브라우저 DevTools 열기
-> 개발자 메뉴에서 원격 디버깅 또는 DevTools 열기

React DevTools 브라우저 확장
-> 독립 실행형 react-devtools

페이지 새로고침
-> Expo 터미널 r / 개발자 메뉴 Reload

DOM 트리 확인
-> React 컴포넌트 트리 확인

Network 패널
-> 원격 디버깅 상태에서 앱 요청 확인
```

프론트엔드 개발자가 이 섹션에서 가져가야 할 핵심은 다음이다.

```text

에러 메시지를 읽고, 로그를 찍고, 상태와 props를 확인한다.

다만 실행 환경이 브라우저가 아니므로
Expo 터미널, 개발자 메뉴, 에뮬레이터, 독립 실행형 React DevTools를 함께 사용한다.
```
