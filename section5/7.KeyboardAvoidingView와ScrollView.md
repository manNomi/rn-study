# 7. KeyboardAvoidingView와 ScrollView

## 핵심 요약

- 가로 모드에서 소프트웨어 키보드가 열리면 입력 필드와 버튼이 키보드에 가려져 앱을 사용할 수 없게 될 수 있다.
- React Native는 이런 문제를 해결하기 위해 `KeyboardAvoidingView` 컴포넌트를 제공한다.
- `KeyboardAvoidingView`는 키보드가 열릴 때 입력 영역을 위로 이동시켜 사용자가 계속 입력 UI에 접근할 수 있게 돕는다.
- `behavior` prop에는 `height`, `padding`, `position` 같은 값을 줄 수 있다.
- `position`은 입력 영역을 위로 밀어내는 데 유용하지만, 이 경우 내부 콘텐츠가 스크롤 가능해야 제대로 사용할 수 있다.
- 그래서 `KeyboardAvoidingView`를 `ScrollView`로 감싸 입력 UI 전체를 스크롤 가능하게 만든다.
- `ScrollView`와 `KeyboardAvoidingView` 모두 화면 전체 공간을 차지하도록 `flex: 1` 스타일을 적용해야 레이아웃이 깨지지 않는다.

## 학습 목표

- 모바일 앱에서 키보드가 입력 UI를 가리는 문제가 왜 중요한지 이해한다.
- `KeyboardAvoidingView`의 역할과 사용 방법을 익힌다.
- `behavior="position"` 설정의 의미와 주의점을 이해한다.
- 입력 화면에서 `ScrollView`와 `KeyboardAvoidingView`를 함께 사용하는 패턴을 배운다.
- `flex: 1`이 루트 래퍼 컴포넌트에서 왜 필요한지 다시 복습한다.

## 주요 내용

### 문제 상황: 키보드가 입력 UI를 가림

가로 모드를 허용한 뒤 시작 화면에서 숫자 입력창을 누르면 문제가 드러난다.

Android에서는 전체 화면에 가까운 키보드가 뜨고, 닫기 버튼이나 시스템 동작으로 비교적 처리할 수 있다.

하지만 iOS에서는 소프트웨어 키보드가 입력창과 버튼을 가릴 수 있다.

```text
iOS 가로 모드 문제
-> 키보드가 입력 필드를 덮음
-> Confirm / Reset 버튼에 접근하기 어려움
-> 배경을 눌러도 키보드가 닫히지 않음
-> 사용자가 앱을 더 이상 정상적으로 사용할 수 없음
```

이 상태는 단순히 보기 안 좋은 문제가 아니라 앱 사용성을 깨뜨리는 문제다.

사용자가 입력을 시작했는데 완료 버튼에 접근할 수 없다면 실제 앱에서는 치명적인 UX 버그다.

### KeyboardAvoidingView란?

React Native는 키보드가 UI를 가리는 문제를 완화하기 위해 `KeyboardAvoidingView`를 제공한다.

```jsx
import { KeyboardAvoidingView } from 'react-native';
```

`KeyboardAvoidingView`는 JSX에서 사용하는 컴포넌트다.

입력 필드를 포함하는 영역을 감싸면, 키보드가 열릴 때 해당 영역을 적절히 이동하거나 크기를 조정한다.

```jsx
<KeyboardAvoidingView>
  <View>
    {/* TextInput과 버튼 등 */}
  </View>
</KeyboardAvoidingView>
```

이번 예제에서는 `StartGameScreen`의 루트 `View`를 `KeyboardAvoidingView`로 감싼다.

```text
기존 구조
-> View(rootContainer)
   -> Title
   -> Card
      -> TextInput
      -> Buttons

변경 구조
-> KeyboardAvoidingView
   -> View(rootContainer)
      -> Title
      -> Card
         -> TextInput
         -> Buttons
```

### 루트 래퍼에 flex: 1이 필요한 이유

`KeyboardAvoidingView`가 최상위 래퍼가 되면 기존 루트 `View`가 더 이상 화면 전체 공간을 직접 차지하지 않을 수 있다.

그 결과 레이아웃이 깨질 수 있다.

해결하려면 `KeyboardAvoidingView`에도 `flex: 1`을 적용해야 한다.

```jsx
const styles = StyleSheet.create({
  screen: {
    flex: 1,
  },
});
```

```jsx
<KeyboardAvoidingView style={styles.screen}>
  <View style={[styles.rootContainer, { marginTop: marginTopDistance }]}>
    {/* ... */}
  </View>
</KeyboardAvoidingView>
```

React Native 레이아웃에서는 상위 컨테이너가 사용 가능한 공간을 차지해야 자식 요소들이 그 공간 안에서 정상적으로 배치된다.

```text
루트 래퍼가 flex: 1을 가지지 않음
-> 자식이 사용할 공간이 불안정함
-> 기존 레이아웃이 깨질 수 있음

루트 래퍼가 flex: 1을 가짐
-> 화면 전체 공간 확보
-> 내부 레이아웃이 기존처럼 작동
```

### behavior prop

`KeyboardAvoidingView`는 `behavior` prop으로 키보드를 피하는 방식을 설정할 수 있다.

```jsx
<KeyboardAvoidingView behavior="position">
```

주요 값은 다음과 같다.

```text
height
-> 키보드 높이에 따라 영역 높이를 조정

padding
-> 키보드 높이만큼 패딩을 추가

position
-> 콘텐츠 위치를 이동
```

강의에서는 `position`이 많은 경우 잘 동작한다고 설명한다.

하지만 실제 프로젝트에서는 화면 구조와 플랫폼에 따라 다르게 보일 수 있으므로 직접 비교해보는 것이 좋다.

```jsx
<KeyboardAvoidingView behavior="position" style={styles.screen}>
  {/* content */}
</KeyboardAvoidingView>
```

### position만으로는 부족한 이유

`behavior="position"`을 설정하면 키보드가 열릴 때 콘텐츠를 위로 이동시키려 한다.

하지만 콘텐츠가 스크롤 가능한 컨테이너 안에 있지 않으면 이동된 UI의 일부에 접근하기 어렵거나 레이아웃이 다시 깨질 수 있다.

```text
KeyboardAvoidingView만 사용
-> 콘텐츠를 위로 이동
-> 하지만 화면 공간이 부족하면 일부 UI에 접근하기 어려움

ScrollView와 함께 사용
-> 콘텐츠를 위로 이동
-> 사용자가 스크롤해서 필요한 UI에 접근 가능
```

따라서 이번 케이스에서는 `KeyboardAvoidingView`를 `ScrollView`로 감싼다.

### ScrollView로 감싸기

입력 화면은 무한히 늘어나는 목록이 아니다.

항목 수가 정해진 UI이며, 화면이 부족할 때만 스크롤이 필요하다.

따라서 `FlatList`가 아니라 `ScrollView`를 사용한다.

```jsx
import { KeyboardAvoidingView, ScrollView } from 'react-native';
```

구조는 다음과 같다.

```jsx
<ScrollView style={styles.screen}>
  <KeyboardAvoidingView behavior="position" style={styles.screen}>
    <View style={[styles.rootContainer, { marginTop: marginTopDistance }]}>
      {/* Title, Card, TextInput, Buttons */}
    </View>
  </KeyboardAvoidingView>
</ScrollView>
```

`ScrollView`에도 `style={styles.screen}`을 적용한다.

```text
ScrollView flex: 1
-> 전체 화면 영역 차지

KeyboardAvoidingView flex: 1
-> 내부 입력 화면도 충분한 공간 확보
```

이렇게 하면 키보드가 열렸을 때 콘텐츠가 위로 이동하고, 사용자는 스크롤해서 입력창과 버튼에 접근할 수 있다.

### 키보드 닫기

`KeyboardAvoidingView`를 사용하면 일부 영역을 탭했을 때 키보드가 닫히는 부가적인 동작도 얻을 수 있다.

강의에서는 카드 영역을 탭하면 키보드가 닫히는 모습을 확인한다.

다만 모든 영역에서 항상 닫히는 것은 아닐 수 있다.

```text
개선된 점
-> 사용자가 입력창과 버튼에 다시 접근 가능
-> 일부 영역을 탭해 키보드를 닫을 수 있음

남은 한계
-> 모든 영역 탭이 항상 키보드 닫기로 이어지지는 않을 수 있음
```

그래도 가장 중요한 목표인 “키보드가 떠도 입력 UI를 계속 사용할 수 있게 만들기”는 해결된다.

## 예시 코드

```jsx
import {
  KeyboardAvoidingView,
  ScrollView,
  StyleSheet,
  TextInput,
  useWindowDimensions,
  View,
} from 'react-native';

function StartGameScreen({ onPickNumber }) {
  const { height } = useWindowDimensions();

  const marginTopDistance = height < 380 ? 30 : 100;

  return (
    <ScrollView style={styles.screen}>
      <KeyboardAvoidingView behavior="position" style={styles.screen}>
        <View style={[styles.rootContainer, { marginTop: marginTopDistance }]}>
          {/* Title */}
          {/* Card */}
          {/* TextInput */}
          {/* Buttons */}
        </View>
      </KeyboardAvoidingView>
    </ScrollView>
  );
}

const styles = StyleSheet.create({
  screen: {
    flex: 1,
  },
  rootContainer: {
    flex: 1,
    alignItems: 'center',
  },
});
```

실제 코드에서는 `Title`, `Card`, `InstructionText`, `PrimaryButton` 등이 함께 사용된다.

핵심은 입력 필드를 포함한 화면을 `KeyboardAvoidingView`로 감싸고, 필요한 경우 그 바깥을 `ScrollView`로 감싸는 것이다.

## 프론트엔드 개발자 관점에서의 핵심 차이

### 웹보다 키보드 대응이 중요하다

웹에서는 데스크톱 키보드가 화면을 덮지 않는다.

하지만 모바일 앱에서는 소프트웨어 키보드가 화면의 큰 영역을 차지한다.

```text
웹 데스크톱
-> 키보드가 UI를 가리지 않음

모바일 앱
-> 소프트웨어 키보드가 화면 일부를 덮음
-> 입력 필드와 버튼이 가려질 수 있음
```

따라서 모바일 입력 화면에서는 키보드 대응이 레이아웃의 일부다.

### ScrollView는 목록이 아니어도 필요하다

웹 개발자는 스크롤을 페이지 전체 흐름에 맡기는 경우가 많다.

React Native에서는 화면이 부족할 때 사용자가 접근해야 하는 고정 UI에도 `ScrollView`가 필요할 수 있다.

```text
FlatList
-> 데이터 목록
-> 항목 수가 많거나 동적일 때

ScrollView
-> 정해진 UI 전체를 스크롤 가능하게 만들 때
-> 키보드 때문에 공간이 부족해지는 입력 화면에 유용
```

이번 시작 화면은 동적 목록이 아니라 고정 입력 UI이므로 `ScrollView`가 적합하다.

### 루트 컨테이너의 flex를 잊으면 레이아웃이 깨진다

React Native에서 래퍼 컴포넌트를 추가하면 기존 레이아웃이 갑자기 깨질 수 있다.

대부분은 새 래퍼가 충분한 공간을 차지하지 않기 때문이다.

```text
새 래퍼 추가
-> ScrollView
-> KeyboardAvoidingView

확인할 것
-> style={styles.screen}
-> flex: 1
```

웹에서 wrapper div를 추가하면서 display/flex 스타일이 달라지는 것과 비슷한 문제다.

## 정리

이번 강의에서는 가로 모드에서 소프트웨어 키보드가 입력창과 버튼을 가리는 문제를 해결하기 위해 `KeyboardAvoidingView`를 사용했다.

`KeyboardAvoidingView`는 키보드가 열릴 때 입력 UI가 가려지지 않도록 위치나 크기를 조정해주는 React Native 내장 컴포넌트다.

`behavior="position"`을 사용할 때는 콘텐츠가 화면 밖으로 밀릴 수 있으므로, `ScrollView`로 감싸 사용자가 필요한 UI에 스크롤로 접근할 수 있게 만드는 것이 좋다.

또한 `ScrollView`와 `KeyboardAvoidingView` 모두 `flex: 1`을 적용해 전체 화면 공간을 차지하도록 해야 기존 레이아웃이 깨지지 않는다.

이 패턴은 모바일 앱에서 입력 화면을 만들 때 매우 자주 쓰이는 기본기다.
