# 섹션 5. 프론트엔드 개발자 관점 정리

## 한 줄 요약

React Native에서 웹의 미디어 쿼리와 CSS 반응형 사고를 그대로 사용할 수는 없지만, `Dimensions`, `useWindowDimensions`, `KeyboardAvoidingView`, `ScrollView`, `Platform`, 플랫폼별 파일, `StatusBar`를 조합하면 다양한 화면 크기, 방향, 플랫폼에 대응하는 적응형 UI를 만들 수 있다.

## 이미 알고 있다고 보는 것

- React 함수 컴포넌트
- JSX 조건부 렌더링
- `useState`, `useEffect` 같은 React 훅
- Flexbox 기반 레이아웃
- CSS의 `width`, `max-width`, `min-width` 개념
- 반응형 웹에서 브레이크포인트를 잡는 감각
- 모바일에서 safe area, status bar, keyboard overlay가 UX에 영향을 준다는 사실

## 새로 잡아야 하는 관점

섹션 5는 “React Native의 반응형 웹 버전”을 배우는 구간에 가깝다.

다만 웹처럼 CSS 미디어 쿼리를 작성하는 방식이 아니다.

React Native에서는 화면 크기와 플랫폼 정보를 JavaScript에서 읽고, 그 값에 따라 스타일 객체나 렌더링 구조를 바꾼다.

```text
웹
-> CSS media query
-> viewport 단위
-> 브라우저가 CSS를 재평가

React Native
-> Dimensions / useWindowDimensions
-> Platform API
-> 조건부 스타일 객체
-> 조건부 JSX 구조
```

즉 반응형 책임이 CSS 파일보다 컴포넌트 코드와 스타일 계산 안으로 들어온다.

## 화면 크기 대응

처음에는 `width`, `maxWidth`, 퍼센트 값을 사용해 요소의 크기를 상대적으로 조정했다.

```jsx
title: {
  width: 300,
  maxWidth: '80%',
}
```

이 패턴은 웹의 다음 감각과 비슷하다.

```css
width: 300px;
max-width: 80%;
```

고정값만 쓰면 작은 기기에서 UI가 답답해질 수 있다.

반대로 퍼센트만 쓰면 큰 화면에서 필요 이상으로 넓어질 수 있다.

그래서 기본 크기와 최대 크기를 함께 두면 더 안전하다.

```text
큰 화면
-> 300px 사용

작은 화면
-> 부모 너비의 80%까지만 사용
```

## Dimensions API

`Dimensions.get('window')`는 현재 사용 가능한 화면 크기를 읽는 API다.

```jsx
const deviceWidth = Dimensions.get('window').width;
```

이를 이용해 작은 화면에서는 padding, margin, fontSize, 이미지 크기를 줄일 수 있다.

```jsx
padding: deviceWidth < 380 ? 12 : 24
```

웹의 브레이크포인트와 비슷하지만 작성 위치가 다르다.

```text
웹
-> CSS @media

React Native
-> JavaScript 조건식
```

다만 `Dimensions.get()`을 파일 최상단에서 한 번 호출하면 값이 고정된다.

앱 실행 중 화면 방향이 바뀌어도 자동으로 다시 계산되지 않는다.

이 점이 `useWindowDimensions`와의 중요한 차이다.

## useWindowDimensions

`useWindowDimensions`는 React 훅이다.

컴포넌트 내부에서 사용하며, 화면 크기나 방향이 바뀌면 새 값을 반영한다.

```jsx
const { width, height } = useWindowDimensions();
```

이 값으로 동적 스타일을 만들 수 있다.

```jsx
const marginTopDistance = height < 380 ? 30 : 100;

<View style={[styles.rootContainer, { marginTop: marginTopDistance }]} />
```

웹에서 브라우저가 viewport 변경에 따라 CSS를 다시 평가하는 것처럼, React Native에서는 훅이 최신 크기를 제공하고 컴포넌트가 다시 렌더링된다.

```text
Dimensions.get()
-> 초기값을 고정해도 되는 경우

useWindowDimensions()
-> 회전, 크기 변경에 동적으로 대응해야 하는 경우
```

## 화면 방향 대응

`app.json`의 orientation 설정을 `portrait`에서 `default`로 바꾸면 가로 모드를 허용할 수 있다.

하지만 허용만 한다고 UI가 자동으로 좋아지는 것은 아니다.

```text
세로 모드
-> 위에서 아래로 쌓는 레이아웃이 자연스러움

가로 모드
-> 높이가 부족해지고, 너비가 넓어짐
-> 같은 구조를 유지하면 화면이 답답하거나 잘림
```

섹션에서는 `width`가 충분히 넓을 때 JSX 구조 자체를 바꾸는 방식도 다뤘다.

```text
좁은 화면
-> NumberContainer
-> Card 안에 버튼
-> 로그 목록

넓은 화면
-> 버튼 / NumberContainer / 버튼을 가로 배치
-> 불필요한 안내 문구 제거
-> 로그 공간 확보
```

프론트엔드 관점에서는 단순 스타일 수정이 아니라 “레이아웃 구조 전환”이다.

웹에서도 모바일/데스크톱에서 markup 또는 component composition을 다르게 가져가는 경우와 비슷하다.

## 키보드 대응

모바일 앱에서 입력 필드는 키보드와 함께 생각해야 한다.

특히 iOS에서는 키보드가 화면을 덮어 입력 영역이나 버튼을 가릴 수 있다.

이를 해결하기 위해 `KeyboardAvoidingView`와 `ScrollView`를 함께 사용했다.

```jsx
<ScrollView style={styles.screen}>
  <KeyboardAvoidingView behavior="position" style={styles.screen}>
    ...
  </KeyboardAvoidingView>
</ScrollView>
```

```text
KeyboardAvoidingView
-> 키보드가 열릴 때 입력 UI를 밀어 올림

ScrollView
-> 공간이 부족할 때 사용자가 스크롤해서 접근 가능
```

웹에서는 브라우저와 OS가 어느 정도 처리해주는 영역이지만, React Native에서는 직접 신경 써야 하는 모바일 UX 포인트다.

## GameOverScreen의 이미지 크기

고정된 `width: 300`, `height: 300`은 작은 화면이나 가로 모드에서 문제가 된다.

그래서 `useWindowDimensions`로 화면 크기와 방향을 보고 이미지 크기를 계산했다.

```jsx
let imageSize = 300;

if (width < 380) {
  imageSize = 150;
}

if (height < 400) {
  imageSize = 80;
}
```

원형 이미지는 width, height, borderRadius가 함께 움직여야 한다.

```jsx
const imageStyle = {
  width: imageSize,
  height: imageSize,
  borderRadius: imageSize / 2,
};
```

웹에서 `aspect-ratio`나 `clamp()`로 비율을 유지하는 문제와 비슷하다.

React Native에서는 JavaScript 계산으로 해결하는 경우가 많다.

## Platform API

`Platform.OS`와 `Platform.select()`로 플랫폼별 스타일을 줄 수 있다.

```jsx
borderWidth: Platform.OS === 'android' ? 2 : 0
```

또는 다음처럼 쓸 수 있다.

```jsx
borderWidth: Platform.select({
  ios: 0,
  android: 2,
})
```

웹에서 브라우저별 feature detection을 하거나 OS별 스타일을 분기하는 것과 비슷하지만, React Native에서는 iOS/Android 차이가 더 직접적인 UI 품질에 영향을 준다.

```text
iOS
-> shadow* 스타일
-> status bar 관례
-> 키보드 동작

Android
-> elevation
-> ripple
-> auto-rotate 설정
```

## 플랫폼별 파일

React Native는 파일명으로 플랫폼별 구현을 나눌 수 있다.

```text
Title.android.js
Title.ios.js
```

import는 그대로 유지한다.

```jsx
import Title from '../components/ui/Title';
```

React Native가 실행 플랫폼에 맞는 파일을 선택한다.

이 방식은 단순 삼항 연산자로 해결하기 어려운 큰 차이가 있을 때 유용하다.

```text
작은 스타일 차이
-> Platform.OS / Platform.select

구조나 코드가 크게 다름
-> .ios.js / .android.js 파일 분리
```

## StatusBar

`expo-status-bar`의 `StatusBar`를 사용해 상태바 스타일을 조정했다.

```jsx
<StatusBar style="light" />
```

모바일 앱에서는 status bar도 화면의 일부다.

배경색이 어두운데 status bar 텍스트가 어둡다면 가독성이 떨어진다.

웹에서 document title, meta theme-color, safe-area inset 등을 챙기는 것처럼 React Native에서는 status bar를 명시적으로 관리하는 감각이 필요하다.

## 프론트엔드 개발자에게 중요한 전환

이번 섹션의 핵심은 “React Native 반응형은 CSS가 아니라 런타임 정보와 컴포넌트 조건부 렌더링으로 풀 때가 많다”는 점이다.

```text
CSS 중심 사고
-> 미디어 쿼리
-> cascade
-> 브라우저 레이아웃 엔진

React Native 중심 사고
-> 현재 window width/height
-> 현재 orientation
-> 현재 platform
-> style 배열 병합
-> JSX 구조 변경
```

모바일 앱은 화면 크기뿐 아니라 키보드, 회전, 상태바, 플랫폼 제스처까지 함께 고려해야 한다.

따라서 작은 화면 QA, 가로 모드 QA, iOS/Android QA가 UI 개발의 일부가 된다.

## 실무 체크리스트

- 고정 width/height가 작은 기기에서 깨지지 않는가?
- 큰 기기에서 요소가 필요 이상으로 넓어지지 않는가?
- `maxWidth`, `minWidth`, 퍼센트 값을 적절히 조합했는가?
- 방향 전환이 허용된다면 `useWindowDimensions`를 사용했는가?
- 가로 모드에서 세로 모드 레이아웃을 그대로 쓰는 것이 적절한가?
- 키보드가 입력창과 버튼을 가리지 않는가?
- 빈 공간이 부족할 때 `ScrollView`가 필요한가?
- iOS와 Android에서 그림자, 터치 피드백, status bar가 자연스러운가?
- 플랫폼별 차이가 작은 조건식으로 충분한가, 파일 분리가 필요한가?

## 정리

섹션 5는 React Native 앱을 다양한 기기와 플랫폼에서 “사용 가능하고 보기 좋게” 만드는 방법을 다룬다.

웹 프론트엔드의 반응형 사고를 그대로 가져오되, CSS 미디어 쿼리 대신 `Dimensions`, `useWindowDimensions`, `Platform`, `KeyboardAvoidingView`, `ScrollView`, 플랫폼별 파일 같은 React Native 도구로 다시 표현해야 한다.

결국 목표는 같다. 하나의 코드베이스가 여러 사용자 환경에서 깨지지 않고, 각 환경에 어울리는 UI를 제공하도록 만드는 것이다.
