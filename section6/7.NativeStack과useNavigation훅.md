# 7. Native Stack과 useNavigation 훅

## 핵심 요약

- React Navigation에는 `native-stack`과 일반 `stack` navigator가 있다.
- 둘 다 화면을 스택처럼 쌓고, 뒤로가기 가능한 Stack Navigation을 제공한다.
- `native-stack`은 네이티브 플랫폼 요소를 사용해 화면과 전환 애니메이션을 처리하므로 더 성능이 좋을 수 있다.
- 일반 `stack`은 네이티브 동작을 JavaScript 쪽에서 에뮬레이션하는 방식에 가깝다.
- 특별한 문제가 없다면 `native-stack`을 우선 사용하는 것이 좋다.
- `navigation` prop은 `Stack.Screen`에 직접 등록된 화면 컴포넌트에만 자동으로 전달된다.
- 일반 자식 컴포넌트에서 navigation 객체가 필요하다면 prop으로 내려주거나, React Navigation의 `useNavigation` 훅을 사용할 수 있다.

## 학습 목표

- `native-stack`과 일반 `stack` navigator의 차이를 이해한다.
- Stack Navigation이 화면을 쌓고 제거하는 방식이라는 점을 다시 정리한다.
- `navigation.navigate()`가 어디에서 사용 가능한지 이해한다.
- screen 컴포넌트와 일반 자식 컴포넌트의 차이를 구분한다.
- `useNavigation` 훅을 사용해 중첩 컴포넌트에서 navigation 객체를 얻는 방법을 익힌다.
- prop drilling과 hook 사용 방식의 장단점을 비교한다.

## 주요 내용

### Native Stack과 일반 Stack

React Navigation에는 여러 Navigator가 있다.

Stack Navigation 계열에도 두 가지 선택지가 있다.

```text
Stack 계열 Navigator
-> native-stack
-> stack
```

둘 다 기본 개념은 같다.

새 화면으로 이동하면 현재 화면 위에 새 화면이 쌓이고, 뒤로가기를 하면 위에 쌓인 화면이 제거된다.

```text
CategoriesScreen
-> MealsOverviewScreen
-> 뒤로가기
-> CategoriesScreen
```

이런 구조를 Stack Navigation이라고 한다.

### native-stack의 특징

강의에서는 `@react-navigation/native-stack`을 사용한다.

```jsx
import { createNativeStackNavigator } from '@react-navigation/native-stack';
```

`native-stack`은 화면 전환과 애니메이션에서 네이티브 플랫폼 요소를 활용한다.

```text
native-stack
-> 네이티브 플랫폼 요소 사용
-> iOS/Android 기본 전환 느낌에 가까움
-> 성능 면에서 유리할 수 있음
```

모바일 앱에서는 화면 전환이 자주 발생하므로, 가능한 네이티브 동작을 활용하는 것이 자연스럽고 성능에도 도움이 된다.

그래서 특별한 이유가 없다면 `native-stack`을 우선 선택하는 것이 좋다.

### 일반 stack의 특징

React Navigation에는 일반 `stack` navigator도 있다.

일반 `stack`은 native stack과 같은 스택 기반 내비게이션을 제공하지만, 네이티브 동작을 직접 사용하는 대신 JavaScript/React Native 레벨에서 네이티브 동작을 흉내 내는 방식에 가깝다.

```text
stack
-> native-stack과 같은 스택 내비게이션 제공
-> 네이티브 동작을 에뮬레이션
-> 특정 상황에서 native-stack 문제가 있을 때 대안이 될 수 있음
```

대부분의 경우 `native-stack`을 사용하면 되지만, 특정 기능이나 호환성 문제를 만난다면 일반 `stack`을 고려할 수 있다.

### navigate 메서드 복습

이전 강의에서는 카테고리 타일을 눌렀을 때 다음 화면으로 이동했다.

```jsx
navigation.navigate('MealsOverview');
```

`navigate`는 React Navigation이 제공하는 메서드다.

이 메서드는 `navigation` 객체 안에 들어 있다.

```text
navigation 객체
-> navigate
-> goBack
-> push
-> pop
-> 기타 내비게이션 관련 메서드
```

이번 단계에서는 `navigate`만 사용했지만, 앞으로 다른 메서드도 활용할 수 있다.

### navigation prop은 어디서 받는가?

`navigation` prop은 아무 컴포넌트에나 자동으로 들어오지 않는다.

React Navigation은 `Stack.Screen`의 `component`로 등록된 화면 컴포넌트에만 `navigation` prop을 자동으로 전달한다.

```jsx
<Stack.Screen
  name="MealsCategories"
  component={CategoriesScreen}
/>
```

위처럼 등록된 `CategoriesScreen`은 `navigation` prop을 받을 수 있다.

```jsx
function CategoriesScreen({ navigation }) {
  // navigation 사용 가능
}
```

하지만 `CategoryGridTile`은 `Stack.Screen`으로 등록된 화면이 아니다.

```jsx
function CategoryGridTile() {
  // navigation prop 자동 제공 안 됨
}
```

따라서 일반 자식 컴포넌트에서는 `navigation` prop을 바로 사용할 수 없다.

### 방법 1: 함수를 prop으로 내려주기

이전 강의에서는 `CategoriesScreen`에서 `pressHandler`를 만들고, 이를 `CategoryGridTile`에 `onPress` prop으로 내려줬다.

```jsx
function CategoriesScreen({ navigation }) {
  function renderCategoryItem(itemData) {
    function pressHandler() {
      navigation.navigate('MealsOverview');
    }

    return (
      <CategoryGridTile
        title={itemData.item.title}
        color={itemData.item.color}
        onPress={pressHandler}
      />
    );
  }
}
```

`CategoryGridTile`은 받은 `onPress`를 `Pressable`에 연결한다.

```jsx
function CategoryGridTile({ title, color, onPress }) {
  return (
    <Pressable onPress={onPress}>
      ...
    </Pressable>
  );
}
```

이 방식의 장점은 UI 컴포넌트가 내비게이션 라이브러리에 의존하지 않는다는 점이다.

```text
장점
-> CategoryGridTile이 재사용 가능해짐
-> 테스트하기 쉬움
-> 화면 이동 정책을 부모가 결정
```

작은 UI 컴포넌트는 이처럼 dumb component로 유지하는 편이 깔끔할 때가 많다.

### 방법 2: navigation prop을 그대로 전달하기

다른 방법으로는 `navigation` 객체 자체를 자식 컴포넌트에 전달할 수도 있다.

```jsx
<CategoryGridTile
  navigation={navigation}
/>
```

그리고 자식에서 사용한다.

```jsx
function CategoryGridTile({ navigation }) {
  function pressHandler() {
    navigation.navigate('MealsOverview');
  }
}
```

이 방식도 가능하다.

하지만 컴포넌트가 React Navigation에 직접 의존하게 된다.

또 중첩 단계가 깊어질수록 계속 `navigation` prop을 내려줘야 해서 번거로울 수 있다.

```text
navigation prop 전달 방식의 단점
-> prop drilling 발생 가능
-> UI 컴포넌트가 navigation에 직접 의존
-> 재사용성이 낮아질 수 있음
```

### 방법 3: useNavigation 훅

중첩 컴포넌트에서 직접 navigation 객체가 필요하다면 `useNavigation` 훅을 사용할 수 있다.

이 훅은 React Navigation의 core 패키지인 `@react-navigation/native`에서 가져온다.

```jsx
import { useNavigation } from '@react-navigation/native';
```

`useNavigation`은 화면 컴포넌트가 아닌 일반 컴포넌트에서도 사용할 수 있다.

```jsx
function CategoryGridTile({ title, color }) {
  const navigation = useNavigation();

  function pressHandler() {
    navigation.navigate('MealsOverview');
  }

  return (
    <Pressable onPress={pressHandler}>
      ...
    </Pressable>
  );
}
```

이렇게 하면 prop으로 `navigation`을 전달하지 않아도 된다.

```text
useNavigation
-> React Navigation이 제공하는 hook
-> 중첩 컴포넌트에서도 navigation 객체 접근 가능
-> screen 컴포넌트가 아니어도 사용 가능
```

단, 이 훅을 사용하려면 해당 컴포넌트가 `NavigationContainer`와 Navigator 하위에서 렌더링되고 있어야 한다.

### useNavigation을 꼭 써야 하는가?

이번 예제에서는 `useNavigation`이 꼭 필요하지 않다.

이미 부모인 `CategoriesScreen`에서 `navigation` prop을 받고 있고, `onPress` 함수를 내려주는 구조가 충분히 깔끔하다.

```text
이번 예제
-> parent에서 navigation 사용
-> child에는 onPress만 전달
-> UI 컴포넌트가 navigation에 덜 의존
```

하지만 실제 앱에서는 깊게 중첩된 컴포넌트에서 화면 이동이 필요할 수 있다.

그럴 때 `useNavigation`은 유용한 대안이 된다.

### 어떤 방식을 선택할까?

상황에 따라 선택하면 된다.

```text
부모가 이동 정책을 결정하는 것이 자연스러운 경우
-> onPress 같은 콜백 prop 전달

자식 컴포넌트가 명확히 특정 화면 이동 버튼인 경우
-> useNavigation 사용 가능

여러 단계로 navigation prop을 내려야 하는 경우
-> useNavigation이 더 간단할 수 있음
```

중요한 것은 모든 컴포넌트에서 무조건 `useNavigation`을 쓰는 것이 아니라, 컴포넌트의 책임과 재사용성을 고려하는 것이다.

## 예시 코드

### 화면 컴포넌트에서 navigation prop 사용

```jsx
function CategoriesScreen({ navigation }) {
  function renderCategoryItem(itemData) {
    function pressHandler() {
      navigation.navigate('MealsOverview');
    }

    return (
      <CategoryGridTile
        title={itemData.item.title}
        color={itemData.item.color}
        onPress={pressHandler}
      />
    );
  }
}
```

### 일반 컴포넌트에서 useNavigation 사용

```jsx
import { useNavigation } from '@react-navigation/native';
import { Pressable, Text, View } from 'react-native';

function CategoryGridTile({ title, color }) {
  const navigation = useNavigation();

  function pressHandler() {
    navigation.navigate('MealsOverview');
  }

  return (
    <View>
      <Pressable onPress={pressHandler}>
        <View style={{ backgroundColor: color }}>
          <Text>{title}</Text>
        </View>
      </Pressable>
    </View>
  );
}
```

## 프론트엔드 개발자 관점에서의 핵심 차이

### React Router의 useNavigate와 비슷하다

웹 React Router를 알고 있다면 `useNavigation`은 `useNavigate`와 비슷한 감각으로 이해할 수 있다.

```jsx
// React Router
const navigate = useNavigate();
navigate('/meals');
```

```jsx
// React Navigation
const navigation = useNavigation();
navigation.navigate('MealsOverview');
```

다만 React Native에서는 URL path가 아니라 screen name을 대상으로 이동한다.

```text
웹
-> navigate('/meals')

React Native
-> navigation.navigate('MealsOverview')
```

### prop으로 넘길지 hook을 쓸지의 기준

웹 React에서도 하위 컴포넌트에 이벤트 핸들러를 넘길지, 내부에서 라우터 훅을 쓸지 고민하는 경우가 있다.

React Native도 비슷하다.

```text
콜백 prop 방식
-> 컴포넌트가 라우터/내비게이션에 덜 의존
-> 재사용성이 높음
-> 부모가 행동을 결정

useNavigation 방식
-> 중첩 컴포넌트에서 바로 이동 가능
-> prop drilling 감소
-> navigation 의존성이 컴포넌트 안으로 들어옴
```

어느 쪽이 항상 정답은 아니다.

컴포넌트가 얼마나 재사용 가능한 UI인지, 또는 특정 화면 이동에 강하게 묶인 버튼인지에 따라 결정하면 된다.

### Native Stack은 모바일스럽게 가는 기본 선택지

웹 라우터는 보통 브라우저 히스토리와 URL이 중심이다.

React Native의 Native Stack은 네이티브 화면 전환과 뒤로가기 감각을 제공한다.

```text
Native Stack을 우선 고려하는 이유
-> 네이티브 전환 애니메이션
-> 플랫폼별 자연스러운 동작
-> 성능상 유리할 수 있음
-> 기본 뒤로가기 UX 제공
```

모바일 앱에서 화면 전환은 사용자 경험의 큰 부분이므로, 가능하면 네이티브 감각을 살리는 선택이 좋다.

## 정리

이번 강의에서는 React Navigation의 `native-stack`과 일반 `stack`의 차이를 살펴보고, `navigation` 객체에 접근하는 두 가지 방법을 정리했다.

화면 컴포넌트는 `navigation` prop을 자동으로 받을 수 있지만, 일반 자식 컴포넌트는 그렇지 않다.

자식 컴포넌트에서 이동이 필요할 때는 부모에서 콜백을 내려주거나, `useNavigation` 훅을 사용해 navigation 객체를 직접 가져올 수 있다.

이번 예제에서는 콜백 prop 방식이 충분하지만, 실제 앱에서는 `useNavigation`이 유용한 대안이 될 수 있다.
