# 23. 섹션 마무리와 React Navigation 핵심 정리

## 핵심 요약

- 이번 섹션에서는 React Native 앱에서 여러 화면을 자연스럽게 전환하는 방법을 배웠다.
- 핵심 도구는 React Navigation 패키지다.
- Stack Navigator를 중심으로 화면을 앞으로 쌓고 뒤로 돌아가는 기본 모바일 내비게이션 흐름을 구현했다.
- `navigate()`와 params를 사용해 화면 간 데이터를 전달하는 방법을 익혔다.
- `route.params`로 전달된 값을 꺼내고, 그 값을 기반으로 목록 필터링이나 상세 데이터 조회를 수행했다.
- screen options를 통해 헤더 제목, 색상, 스타일, 버튼, 아이콘을 설정했다.
- `navigation.setOptions()`와 `useLayoutEffect`를 사용해 화면 내부 데이터에 따라 헤더를 동적으로 설정했다.
- Drawer Navigator와 Bottom Tabs Navigator를 통해 Stack 외의 다른 네비게이션 패턴도 살펴봤다.
- 마지막에는 Stack과 Drawer를 중첩해 하나의 앱에서 여러 네비게이터를 조합하는 방법을 배웠다.
- React Navigation은 기능이 매우 많은 패키지이므로, 실제 프로젝트에서는 공식 문서를 함께 보며 필요한 옵션을 찾아 적용하는 습관이 중요하다.

## 학습 목표

- 이번 섹션에서 다룬 React Navigation의 핵심 개념을 정리한다.
- Stack, Drawer, Tabs Navigator의 역할 차이를 다시 확인한다.
- 화면 간 데이터 전달과 화면 옵션 설정 흐름을 복습한다.
- 정적 설정과 동적 설정의 차이를 이해한다.
- nested navigators가 왜 필요한지, 어떤 식으로 조합되는지 정리한다.
- 앞으로 공식 문서를 볼 때 어떤 기준으로 옵션과 네비게이터를 선택해야 하는지 감을 잡는다.

## 주요 내용

### 이번 섹션의 중심 주제

이전 섹션들에서는 React Native의 핵심 컴포넌트, 스타일링, 레이아웃, 상태 관리 기초를 많이 다뤘다.

이번 섹션에서는 그 위에 실제 앱다운 화면 전환을 얹었다.

```text
React Native 기본 UI
-> 화면 컴포넌트 구성
-> 스타일링과 레이아웃
-> React Navigation으로 화면 전환
```

단순히 조건문으로 화면을 바꾸는 방식이 아니라, 모바일 앱에서 기대하는 전환 애니메이션, 뒤로 가기, 헤더, 탭, 드로어 같은 네이티브 내비게이션 경험을 추가했다.

### Stack Navigator

가장 많이 다룬 네비게이터는 Stack Navigator다.

Stack Navigator는 화면을 스택처럼 쌓는다.

```text
Categories
-> MealsOverview
-> MealDetail
```

사용자가 더 자세한 정보를 보러 들어갈 때는 새 화면이 위에 쌓이고, 뒤로 가기를 누르면 이전 화면으로 돌아간다.

이 패턴은 모바일 앱에서 매우 흔하다.

```text
목록
-> 상세
-> 상세의 상세
```

이번 Meals 앱에서는 카테고리 목록, 음식 목록, 음식 상세 화면을 Stack Navigator로 연결했다.

### navigate로 화면 이동

화면 이동은 `navigation.navigate()`로 수행했다.

```jsx
navigation.navigate('MealsOverview');
```

특정 화면으로 이동할 때는 `Stack.Screen`에 등록한 `name`을 사용한다.

```jsx
<Stack.Screen
  name="MealsOverview"
  component={MealsOverviewScreen}
/>
```

이름은 화면 이동의 식별자이므로 중복되거나 모호하지 않게 관리하는 것이 중요하다.

### params로 데이터 전달

화면을 이동할 때 데이터가 필요한 경우가 많다.

예를 들어 카테고리를 누르면 어떤 카테고리를 눌렀는지 다음 화면이 알아야 한다.

이때 `navigate()`의 두 번째 인자로 params 객체를 전달한다.

```jsx
navigation.navigate('MealsOverview', {
  categoryId: itemData.item.id,
});
```

이후 대상 화면에서는 `route.params`로 값을 꺼낸다.

```jsx
function MealsOverviewScreen({ route }) {
  const catId = route.params.categoryId;
}
```

이 패턴은 React Navigation에서 매우 자주 사용된다.

```text
이전 화면
-> navigate(screenName, params)

다음 화면
-> route.params로 데이터 추출
```

### params를 기반으로 데이터 필터링

전달받은 `categoryId`는 단순히 출력하는 데 그치지 않고, 실제 화면 데이터 필터링에 사용했다.

```jsx
const displayedMeals = MEALS.filter((mealItem) => {
  return mealItem.categoryIds.indexOf(catId) >= 0;
});
```

즉 네비게이션 params는 화면 간 연결 정보이면서, 대상 화면의 데이터 렌더링 기준이 된다.

웹 프론트엔드에서 route param이나 query param으로 데이터를 조회하는 흐름과 비슷하다.

### screen options

React Navigation은 기본 헤더와 화면 셸을 제공한다.

하지만 실제 앱에서는 제목, 색상, 배경, 버튼 등을 커스터마이징해야 한다.

이를 위해 screen options를 사용했다.

```jsx
<Stack.Navigator
  screenOptions={{
    headerStyle: { backgroundColor: '#351401' },
    headerTintColor: 'white',
    contentStyle: { backgroundColor: '#3f2f25' },
  }}
>
  ...
</Stack.Navigator>
```

공통 설정은 `screenOptions`에 둔다.

특정 화면만의 설정은 해당 `Screen`의 `options`에 둔다.

```jsx
<Stack.Screen
  name="MealDetail"
  component={MealDetailScreen}
  options={{
    title: 'About the Meal',
  }}
/>
```

### 동적 screen options

고정된 제목이 아니라, 선택한 카테고리 이름처럼 동적으로 바뀌는 제목도 설정했다.

이때는 화면 컴포넌트 내부에서 `navigation.setOptions()`를 사용했다.

```jsx
useLayoutEffect(() => {
  navigation.setOptions({
    title: categoryTitle,
  });
}, [navigation, categoryTitle]);
```

`setOptions()`는 화면 외부 설정을 바꾸는 side effect이므로 컴포넌트 본문에서 바로 호출하지 않고 `useLayoutEffect` 안에서 처리했다.

`useLayoutEffect`를 사용하면 화면 전환 중 헤더가 늦게 바뀌는 어색함을 줄일 수 있다.

### 헤더 버튼과 아이콘

헤더 오른쪽에 버튼을 추가하는 방법도 배웠다.

처음에는 기본 `Button`을 사용했고, 이후 직접 `IconButton` 컴포넌트를 만들었다.

```jsx
navigation.setOptions({
  headerRight: () => (
    <IconButton
      icon="star"
      color="white"
      onPress={headerButtonPressHandler}
    />
  ),
});
```

`IconButton`은 `Pressable`과 `Ionicons`를 조합한 재사용 컴포넌트였다.

```text
Pressable
-> 터치 가능한 영역

Ionicons
-> 아이콘 표시

onPress
-> 헤더 액션 연결
```

현재는 로그만 출력하지만, 이후에는 즐겨찾기 기능과 연결될 수 있다.

### Drawer Navigator

Stack 외에도 Drawer Navigator를 살펴봤다.

Drawer Navigator는 사이드 메뉴 기반 네비게이션을 제공한다.

```text
드로어 열기
-> Categories
-> Favorites
```

Drawer는 앱의 큰 섹션을 전환할 때 적합하다.

이번 Meals 앱에서는 `Categories`와 `Favorites` 사이를 전환하는 역할을 맡겼다.

또한 다음 옵션들로 Drawer UI를 튜닝했다.

```text
drawerContentStyle
drawerInactiveTintColor
drawerActiveTintColor
drawerActiveBackgroundColor
drawerIcon
```

### Bottom Tabs Navigator

Tabs Navigator도 살펴봤다.

Bottom Tabs Navigator는 화면 하단 탭으로 주요 화면을 전환한다.

```text
Welcome tab
User tab
```

탭은 앱의 주요 섹션을 빠르게 오갈 때 적합하다.

Drawer와 마찬가지로 탭 아이콘은 `tabBarIcon`으로 설정했다.

```jsx
tabBarIcon: ({ color, size }) => (
  <Ionicons
    name="home"
    color={color}
    size={size}
  />
)
```

`color`, `size`는 React Navigation이 활성/비활성 상태에 맞게 제공해 준다.

### Nested Navigators

마지막에는 Stack Navigator와 Drawer Navigator를 조합했다.

핵심은 네비게이터 자체도 React 컴포넌트처럼 다른 네비게이터의 screen으로 등록할 수 있다는 점이다.

```jsx
function DrawerNavigator() {
  return (
    <Drawer.Navigator>
      <Drawer.Screen name="Categories" component={CategoriesScreen} />
      <Drawer.Screen name="Favorites" component={FavoritesScreen} />
    </Drawer.Navigator>
  );
}
```

```jsx
<Stack.Screen
  name="Drawer"
  component={DrawerNavigator}
  options={{ headerShown: false }}
/>
```

이렇게 하면 시작 영역은 Drawer가 담당하고, 상세 흐름은 Stack이 담당한다.

```text
Stack
-> DrawerNavigator
   -> Categories
   -> Favorites
-> MealsOverview
-> MealDetail
```

이 구조를 통해 서로 다른 네비게이션 패턴을 하나의 앱 안에서 조합할 수 있다.

### 공식 문서의 중요성

React Navigation은 매우 강력하고 옵션이 많은 패키지다.

이번 섹션에서는 핵심 패턴을 다뤘지만, 실제로는 더 많은 네비게이터와 설정이 있다.

```text
Stack
Native Stack
Drawer
Bottom Tabs
Material Tabs
Header options
Gesture options
Transition options
Custom drawer content
Custom tab bar
```

따라서 모든 옵션을 외우기보다, 이번 섹션에서 배운 구조를 바탕으로 공식 문서를 읽고 필요한 옵션을 찾아 적용하는 능력이 중요하다.

## 프론트엔드 개발자 관점에서의 핵심 차이

### 화면 전환은 UI 상태 이상의 문제다

React 웹 앱에서는 라우팅을 URL과 연결해서 생각하는 경우가 많다.

React Native에서는 사용자가 직접 URL을 입력하지 않는다.

대신 네비게이터가 화면 스택, 드로어, 탭 상태를 관리한다.

```text
웹
-> URL 중심 라우팅

React Native
-> 네이티브 내비게이션 패턴 중심
```

따라서 React Navigation은 단순한 화면 교체 도구가 아니라, 모바일 앱의 사용 흐름을 설계하는 핵심 도구다.

### 네비게이터 선택은 정보 구조 선택이다

어떤 네비게이터를 쓸지는 문법 문제가 아니라 앱 구조 문제다.

```text
상세로 깊게 들어간다
-> Stack

큰 섹션을 메뉴로 전환한다
-> Drawer

주요 섹션을 자주 오간다
-> Bottom Tabs
```

앱의 정보 구조를 먼저 생각하고, 그에 맞는 네비게이터를 선택해야 한다.

### 옵션 위치가 중요하다

`options`와 `screenOptions`의 차이는 유지보수에 직접적인 영향을 준다.

```text
여러 화면에 공통
-> Navigator의 screenOptions

특정 화면에만 적용
-> Screen의 options

화면 내부 데이터에 따라 변경
-> navigation.setOptions()
```

이 구분을 잘하면 네비게이션 설정이 지저분해지지 않는다.

### Nested Navigator는 복잡한 앱의 기본 기술이다

실제 앱에서는 하나의 네비게이터만으로 모든 흐름을 표현하기 어려운 경우가 많다.

예를 들어 다음과 같은 구조가 흔하다.

```text
Tabs
-> HomeStack
-> SearchStack
-> FavoritesStack
-> ProfileStack
```

이번 Meals 앱에서는 Stack 안에 Drawer를 넣는 구조를 배웠지만, 원리는 같다.

네비게이터를 조합하면 앱의 큰 섹션과 깊은 상세 흐름을 동시에 표현할 수 있다.

## 정리

이번 섹션에서는 React Native 앱의 내비게이션을 본격적으로 다뤘다.

Stack Navigator로 기본 화면 이동과 뒤로 가기 흐름을 만들고, params로 화면 간 데이터를 전달했으며, screen options와 `setOptions()`로 헤더를 정적/동적으로 설정했다.

또한 Drawer Navigator와 Bottom Tabs Navigator를 살펴보고, 마지막에는 Stack과 Drawer를 중첩해 복합적인 앱 구조를 만들었다.

이제 React Navigation의 핵심 감각을 갖췄으므로, 다음 단계에서는 더 깊은 React Native 앱 기능과 앱 전역 상태 관리 같은 주제로 확장할 수 있다.
