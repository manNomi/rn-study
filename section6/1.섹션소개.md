# 1. 섹션 소개

## 핵심 요약

- 이번 섹션의 주제는 React Native 앱에서 여러 화면 사이를 더 자연스럽게 이동하는 내비게이션이다.
- 이전 섹션의 숫자 맞히기 게임 앱에서도 여러 화면을 사용했지만, 화면 전환은 상태와 조건부 렌더링으로 직접 처리했다.
- 게임 앱에서는 단순한 화면 전환 방식이 충분했지만, 일반적인 앱에서는 앞으로 가기, 뒤로 가기, 애니메이션, 탭, 드로어 같은 더 체계적인 내비게이션이 필요하다.
- 이번 섹션에서는 내비게이션이 무엇인지 먼저 정리하고, Stack Navigation, Drawer Navigation, Tab Navigation 같은 대표적인 접근 방식을 배운다.
- 여러 개의 작은 더미 앱을 만들면서 각 내비게이션 패턴이 어떤 상황에 적합한지 확인한다.

## 학습 목표

- React Native 앱에서 내비게이션이 왜 중요한지 이해한다.
- 상태 기반 화면 전환과 내비게이션 시스템의 차이를 구분한다.
- Stack Navigation의 기본 개념을 이해한다.
- Drawer와 Tab을 이용한 화면 이동 패턴을 파악한다.
- 실제 앱에서 화면 이동 구조를 설계할 때 어떤 기준으로 내비게이션 방식을 선택할지 감을 잡는다.

## 주요 내용

### 이전 섹션의 화면 전환 방식

이전 섹션의 숫자 맞히기 게임 앱에는 여러 화면이 있었다.

```text
StartGameScreen
-> 사용자가 숫자를 입력하는 시작 화면

GameScreen
-> 앱이 숫자를 추측하고 사용자가 힌트를 주는 게임 화면

GameOverScreen
-> 게임 결과를 보여주는 종료 화면
```

하지만 이 화면들 사이의 이동은 내비게이션 라이브러리를 사용한 것이 아니었다.

`App` 컴포넌트에서 상태를 관리하고, 상태값에 따라 어떤 화면을 보여줄지 조건부로 결정했다.

```jsx
let screen = <StartGameScreen />;

if (userNumber) {
  screen = <GameScreen />;
}

if (gameIsOver && userNumber) {
  screen = <GameOverScreen />;
}
```

이 방식은 단순한 흐름에는 충분하다.

게임처럼 사용자가 뒤로 돌아갈 필요가 거의 없고, 화면 전환이 선형적인 경우에는 상태 기반 전환이 오히려 간단하고 명확할 수 있다.

### 상태 기반 전환의 한계

상태 기반 화면 전환은 직접적이고 이해하기 쉽지만, 앱 규모가 커지면 한계가 생긴다.

```text
상태 기반 전환의 한계
-> 화면 전환 애니메이션이 없음
-> 앞으로 가기 / 뒤로 가기 흐름을 직접 구현해야 함
-> 화면 히스토리를 관리하기 어려움
-> 헤더, 뒤로가기 버튼, 제스처 처리를 직접 붙여야 함
-> 탭, 드로어 같은 앱 구조를 직접 만들기 번거로움
```

웹 프론트엔드로 비유하면, 상태 하나로 페이지 컴포넌트를 바꿔 끼우는 것과 비슷하다.

```jsx
{page === 'home' && <HomePage />}
{page === 'detail' && <DetailPage />}
```

작은 예제에서는 가능하지만, 실제 서비스에서는 라우터가 필요하다.

React Native에서도 마찬가지다.

실제 앱에서는 화면 이동 자체를 관리하는 내비게이션 시스템이 필요하다.

### 내비게이션이란?

내비게이션은 단순히 “어떤 화면을 보여줄지”를 결정하는 것보다 넓은 개념이다.

```text
내비게이션이 담당하는 것
-> 현재 화면 표시
-> 다음 화면으로 이동
-> 이전 화면으로 돌아가기
-> 화면 전환 애니메이션
-> 화면 히스토리 관리
-> 헤더와 뒤로가기 버튼 처리
-> 탭, 드로어, 스택 같은 구조 관리
```

모바일 앱에서는 사용자가 화면을 이동할 때 기대하는 기본 UX가 있다.

예를 들어 상세 화면으로 들어가면 자연스럽게 뒤로 갈 수 있어야 하고, iOS에서는 스와이프 제스처로 뒤로 가는 흐름도 기대할 수 있다.

이런 흐름을 매번 직접 구현하는 대신, 내비게이션 도구를 사용해 앱 구조로 표현한다.

### Stack Navigation

이번 섹션에서 먼저 살펴볼 대표적인 개념은 Stack Navigation이다.

Stack은 말 그대로 화면을 스택처럼 쌓는 방식이다.

```text
HomeScreen
-> DetailScreen
-> MoreDetailScreen
```

새 화면으로 이동하면 이전 화면 위에 새 화면이 쌓인다.

뒤로 가면 가장 위에 있는 화면이 제거되고 이전 화면이 다시 보인다.

웹 프론트엔드 개발자 관점에서는 브라우저 히스토리와 비슷하게 생각할 수 있다.

```text
push
-> 새 화면을 스택 위에 추가

pop
-> 현재 화면을 제거하고 이전 화면으로 복귀
```

모바일 앱에서 가장 기본적인 화면 이동 패턴이다.

### Drawer Navigation

Drawer Navigation은 화면 옆에서 열리는 메뉴를 통해 주요 화면으로 이동하는 방식이다.

보통 왼쪽 또는 오른쪽에서 슬라이드되어 나오는 메뉴를 말한다.

```text
Drawer 예시
-> Home
-> Profile
-> Settings
-> Help
```

웹의 사이드바 메뉴와 비슷하지만, 모바일에서는 화면 폭이 좁기 때문에 메뉴를 항상 노출하기보다 필요할 때 열고 닫는 패턴을 자주 사용한다.

Drawer는 앱의 큰 섹션을 이동할 때 유용하다.

### Tab Navigation

Tab Navigation은 화면 하단 또는 상단의 탭을 통해 주요 화면을 전환하는 방식이다.

```text
Tab 예시
-> Home
-> Search
-> Favorites
-> Profile
```

모바일 앱에서 매우 흔한 구조다.

사용자가 앱의 주요 섹션을 빠르게 오갈 수 있고, 현재 어느 섹션에 있는지도 쉽게 알 수 있다.

웹 프론트엔드 관점에서는 하단 고정 네비게이션이나 앱 셸 구조와 비슷하다.

### 여러 개의 더미 앱으로 실습

이번 섹션에서는 하나의 큰 앱을 계속 확장하기보다, 여러 개의 작은 더미 앱을 만들면서 내비게이션 패턴을 익힌다.

이 접근의 장점은 각 내비게이션 개념을 분리해서 이해할 수 있다는 점이다.

```text
작은 실습 앱을 사용하는 이유
-> Stack Navigation만 따로 집중
-> Drawer Navigation만 따로 집중
-> Tab Navigation만 따로 집중
-> 복잡한 비즈니스 로직 없이 이동 구조에 집중
```

실제 앱에서는 이 패턴들이 조합될 수 있다.

예를 들어 하단 탭 안에 스택 내비게이션이 들어가거나, 드로어 안에 탭 구조가 들어갈 수도 있다.

## 프론트엔드 개발자 관점에서의 핵심 차이

### React Router와의 비교

웹 프론트엔드 개발자라면 React Router를 떠올리면 이해가 쉽다.

```text
웹 React
-> URL 기반 라우팅
-> 브라우저 히스토리
-> Link, Navigate, route params

React Native
-> 화면 스택 기반 내비게이션
-> 네이티브 화면 전환 UX
-> push, pop, navigate
-> 탭, 드로어, 헤더, 제스처
```

React Native 앱에는 브라우저 주소창이 없다.

따라서 “URL이 바뀌어서 페이지가 바뀐다”는 감각보다 “화면이 스택에 쌓이고 제거된다”는 감각이 더 중요하다.

### 조건부 렌더링과 내비게이션은 역할이 다르다

조건부 렌더링은 여전히 중요하다.

하지만 조건부 렌더링은 보통 화면 내부의 UI 상태를 바꿀 때 적합하다.

```text
조건부 렌더링에 적합한 경우
-> 로딩 상태
-> 에러 상태
-> 모달 표시 여부
-> 폼 단계 전환
-> 게임처럼 단순하고 선형적인 화면 흐름
```

반면 내비게이션은 앱의 화면 구조와 이동 히스토리를 다룬다.

```text
내비게이션에 적합한 경우
-> 홈에서 상세로 이동
-> 상세에서 뒤로가기
-> 탭으로 주요 섹션 전환
-> 드로어 메뉴로 앱 영역 전환
-> 화면 전환 애니메이션과 제스처 필요
```

즉 화면이 많아질수록 상태 하나로 전부 제어하기보다 내비게이션 구조로 모델링하는 편이 자연스럽다.

### 모바일 UX까지 포함해서 생각해야 한다

웹에서는 링크를 클릭하고 URL이 바뀌는 흐름이 익숙하다.

모바일 앱에서는 사용자가 기대하는 내비게이션 UX가 조금 다르다.

```text
모바일 내비게이션 UX
-> 화면 전환 애니메이션
-> 네이티브 뒤로가기 버튼
-> iOS 스와이프 백 제스처
-> Android 하드웨어/시스템 백 버튼
-> 하단 탭
-> 드로어 메뉴
```

따라서 React Native의 내비게이션은 단순한 컴포넌트 교체가 아니라, 플랫폼 사용성까지 포함하는 주제다.

## 정리

이번 섹션은 React Native 앱을 “여러 화면이 있는 실제 앱”처럼 만드는 핵심 단계다.

이전에는 상태와 조건부 렌더링으로 화면을 바꿨다면, 이제는 내비게이션 구조를 사용해 화면 이동을 더 자연스럽고 확장 가능하게 만든다.

앞으로 Stack, Drawer, Tab 같은 대표적인 내비게이션 패턴을 배우면서 모바일 앱의 화면 구조를 설계하는 감각을 익히게 된다.
