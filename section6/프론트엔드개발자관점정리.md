# 섹션 6. 프론트엔드 개발자 관점 정리

## 한 줄 요약

React Navigation을 사용해 React Native 앱의 화면 전환을 웹의 URL 라우팅이 아닌 모바일 앱의 네이티브 내비게이션 패턴으로 이해하고, Stack, Drawer, Tabs, params, screen options, header actions, nested navigators를 조합해 여러 화면을 가진 앱 구조를 설계하는 섹션이다.

## 이미 알고 있다고 보는 것

- React 함수 컴포넌트와 props
- 배열 데이터 렌더링과 `FlatList` 감각
- 컴포넌트 분리와 재사용
- 상태 기반 조건부 렌더링
- 웹 라우터에서 route param이나 query param으로 데이터를 전달하는 개념
- UI shell, header, sidebar, tab navigation 같은 레이아웃 패턴
- 이벤트 핸들러와 콜백 전달

## 새로 잡아야 하는 관점

웹 프론트엔드에서는 화면 이동을 보통 URL 중심으로 생각한다.

```text
/categories
/categories/:id
/meals/:id
```

React Native에서는 사용자가 주소창에 URL을 입력하지 않는다.

화면 이동은 앱 내부의 네비게이터가 관리한다.

```text
Stack Navigator
-> 앞으로 들어가고 뒤로 돌아가는 흐름

Drawer Navigator
-> 앱의 큰 섹션을 사이드 메뉴로 전환

Bottom Tabs Navigator
-> 주요 섹션을 하단 탭으로 전환
```

React Navigation은 라우터이면서 동시에 모바일 앱의 navigation shell을 만들어 주는 도구다.

## 기본 앱 구조

이번 섹션의 Meals 앱은 다음 흐름으로 시작했다.

```text
CategoriesScreen
-> MealsOverviewScreen
-> MealDetailScreen
```

이후 즐겨찾기 섹션을 위해 Drawer를 추가했다.

```text
Drawer
-> Categories
-> Favorites
```

최종적으로는 Stack 안에 Drawer가 들어가는 구조가 되었다.

```text
Stack
-> DrawerNavigator
   -> Categories
   -> Favorites
-> MealsOverview
-> MealDetail
```

웹에서 layout route 안에 nested route를 넣는 것과 비슷하지만, React Navigation에서는 각각의 네비게이터가 고유한 UI와 제스처, 헤더를 가진다.

## Stack Navigator

Stack Navigator는 가장 기본적인 모바일 화면 이동 패턴이다.

```text
목록
-> 상세
-> 더 깊은 상세
```

Meals 앱에서는 카테고리 목록에서 음식 목록으로, 음식 목록에서 음식 상세로 이동했다.

```jsx
navigation.navigate('MealsOverview', {
  categoryId: itemData.item.id,
});
```

`Stack.Screen`의 `name`은 화면 이동의 식별자다.

```jsx
<Stack.Screen
  name="MealsOverview"
  component={MealsOverviewScreen}
/>
```

프론트엔드 개발자 관점에서는 route name을 URL path 대신 사용하는 것으로 이해하면 된다.

```text
웹
-> navigate('/meals/m1')

React Native
-> navigation.navigate('MealDetail', { mealId: 'm1' })
```

## params와 route

화면 간 작은 식별자를 넘길 때 params를 사용했다.

```jsx
navigation.navigate('MealsOverview', {
  categoryId: itemData.item.id,
});
```

대상 화면에서는 `route.params`로 읽는다.

```jsx
const catId = route.params.categoryId;
```

이 값으로 원본 데이터를 필터링했다.

```jsx
const displayedMeals = MEALS.filter((mealItem) => {
  return mealItem.categoryIds.indexOf(catId) >= 0;
});
```

웹의 route params와 역할은 비슷하다.

다만 URL에 드러나는 값이 아니라 네비게이션 action과 route 객체를 통해 전달된다.

```text
params에 적합
-> categoryId
-> mealId
-> 현재 화면을 그리기 위한 식별자

params에 부적합
-> 앱 전체 favorite 목록
-> 로그인 사용자 상태
-> 장바구니 전체 상태
```

## screen options

React Navigation은 기본 헤더를 제공한다.

하지만 실제 앱에서는 제목, 색상, 배경, 버튼을 조정해야 한다.

공통 설정은 `screenOptions`에 둔다.

```jsx
<Stack.Navigator
  screenOptions={{
    headerStyle: { backgroundColor: '#351401' },
    headerTintColor: 'white',
    contentStyle: { backgroundColor: '#3f2f25' },
  }}
>
```

화면별 설정은 `options`에 둔다.

```jsx
<Stack.Screen
  name="MealDetail"
  component={MealDetailScreen}
  options={{
    title: 'About the Meal',
  }}
/>
```

웹 프론트엔드에서 layout component가 공통 header 스타일을 담당하고, page component가 title이나 action을 설정하는 구조와 비슷하다.

## 동적 options

선택된 카테고리 이름처럼 화면 데이터에 따라 헤더 제목이 달라져야 할 때는 `navigation.setOptions()`를 사용했다.

```jsx
useLayoutEffect(() => {
  navigation.setOptions({
    title: categoryTitle,
  });
}, [navigation, categoryTitle]);
```

이 로직은 렌더링 결과 외부의 navigation header를 바꾸는 side effect다.

그래서 컴포넌트 본문에서 바로 호출하지 않고 `useLayoutEffect` 안에서 처리했다.

```text
정적 options
-> screen 설정에서 작성

동적 options
-> 화면 컴포넌트 내부에서 navigation.setOptions()
```

## headerRight와 IconButton

헤더 오른쪽에 버튼을 넣을 수 있다.

```jsx
navigation.setOptions({
  headerRight: () => (
    <IconButton
      icon="star"
      color="white"
      onPress={headerButtonPressHandler}
    />
  ),
});
```

`headerRight`는 JSX를 반환하는 함수다.

여기에 직접 만든 React Native 컴포넌트도 넣을 수 있다.

```text
IconButton
-> Pressable
-> Ionicons
-> onPress
```

웹에서 header action button을 layout slot에 넣는 것과 비슷하다.

차이는 이 헤더가 React Navigation이 관리하는 네이티브 navigation shell의 일부라는 점이다.

## Drawer Navigator

Drawer Navigator는 사이드 메뉴 기반의 주요 섹션 전환이다.

```text
Drawer
-> Categories
-> Favorites
```

등록된 `Drawer.Screen`들은 기본적으로 메뉴 항목으로 표시된다.

```jsx
<Drawer.Screen
  name="Favorites"
  component={FavoritesScreen}
/>
```

Drawer는 단순한 화면 이동이라기보다 앱의 큰 영역을 나누는 navigation shell이다.

웹의 sidebar navigation과 비슷하지만, 모바일에서는 drawer open/close gesture와 header menu icon까지 함께 제공된다.

## Bottom Tabs Navigator

Bottom Tabs Navigator는 화면 하단 탭으로 주요 섹션을 전환한다.

```text
Home
Search
Favorites
Profile
```

강의에서는 별도 dummy 앱으로 tabs를 실습했다.

```jsx
<BottomTabs.Screen
  name="Welcome"
  component={WelcomeScreen}
  options={{
    tabBarIcon: ({ color, size }) => (
      <Ionicons name="home" color={color} size={size} />
    ),
  }}
/>
```

Drawer의 `drawerIcon`과 Tabs의 `tabBarIcon`은 유사한 콜백 패턴을 가진다.

```text
React Navigation이 제공
-> color
-> size
-> focused 여부

개발자가 연결
-> Ionicons에 color/size 전달
```

## Nested Navigators

이번 섹션의 큰 전환점은 네비게이터를 중첩한 것이다.

네비게이터도 하나의 React 컴포넌트처럼 만들 수 있다.

```jsx
function DrawerNavigator() {
  return (
    <Drawer.Navigator>
      <Drawer.Screen name="Categories" component={CategoriesScreen} />
      <Drawer.Screen name="Favorites" component={FavoritesScreen} />
    </Drawer.Navigator>
  );
}
```

그리고 다른 네비게이터의 screen으로 등록할 수 있다.

```jsx
<Stack.Screen
  name="Drawer"
  component={DrawerNavigator}
  options={{ headerShown: false }}
/>
```

이 구조는 실제 앱에서 자주 쓰인다.

```text
Tabs
-> HomeStack
-> SearchStack
-> FavoritesStack
-> ProfileStack
```

이번 앱에서는 Stack 안에 Drawer를 넣었다.

```text
Drawer
-> 큰 섹션 전환

Stack
-> 상세로 깊게 들어가는 흐름
```

## 헤더 중복 문제

네비게이터를 중첩하면 각 네비게이터가 자기 헤더를 렌더링할 수 있다.

그 결과 헤더가 두 개 보일 수 있다.

```text
Stack header
Drawer header
```

이번 앱에서는 Drawer 아이콘이 필요하므로 Drawer header를 남기고 Stack header를 숨겼다.

```jsx
options={{
  headerShown: false,
}}
```

Nested navigator에서는 “어떤 네비게이터의 헤더를 사용할 것인가”를 의식해야 한다.

## 컴포넌트 구조와 데이터 흐름

Meals 앱에서는 목록 UI를 `MealItem`, `MealsList`로 분리했다.

이 덕분에 카테고리 목록과 즐겨찾기 목록에서 같은 UI를 재사용할 수 있는 기반이 생겼다.

```text
MealsOverviewScreen
-> categoryId로 meals 선택
-> MealsList 렌더링

FavoritesScreen
-> favorite ids로 meals 선택
-> MealsList 렌더링
```

이 구조는 이후 섹션 7의 상태 관리와 자연스럽게 이어진다.

## 실무 체크리스트

- 화면 이동이 Stack인지, Drawer인지, Tabs인지 정보 구조에 맞게 선택했는가?
- `Screen name`이 중복되거나 모호하지 않은가?
- params에는 화면 식별자 수준의 작은 데이터만 담고 있는가?
- 앱 전역 상태를 params로 억지로 전달하고 있지는 않은가?
- 공통 header 스타일은 `screenOptions`로 올렸는가?
- 화면별 제목과 아이콘은 `options`에 두었는가?
- 화면 데이터에 따라 header가 바뀌면 `setOptions`를 적절히 사용했는가?
- nested navigator에서 헤더가 중복되지 않는가?
- Drawer/Tabs 아이콘이 active/inactive 색상을 자연스럽게 따르는가?
- 네비게이터 조합이 앱의 UX 구조를 설명하는가?

## 정리

섹션 6은 React Native 앱의 화면 전환을 제대로 앱답게 만드는 구간이다.

웹의 URL 라우팅 감각을 버리는 것이 아니라, 모바일 앱에서는 URL 대신 Stack, Drawer, Tabs 같은 네이티브 내비게이션 패턴으로 같은 문제를 푼다고 이해하면 된다.

React Navigation은 단순한 화면 교체 도구가 아니라 앱의 정보 구조, 헤더, 제스처, 주요 섹션 전환을 함께 설계하는 프레임워크에 가깝다.
