# 2. 내비게이션 개념과 Meals 앱 준비

## 핵심 요약

- React Native에서 내비게이션은 여러 화면 사이를 이동하는 흐름을 의미한다.
- 이전 게임 앱에서는 화면을 단순히 교체했지만, 실제 앱에서는 앞으로 이동, 뒤로가기, 애니메이션, 탭 전환 같은 더 세밀한 흐름이 필요하다.
- 웹에서는 URL과 링크를 중심으로 페이지를 이동하지만, 모바일 앱에서는 버튼이나 터치 동작을 통해 화면에서 화면으로 이동한다.
- 이번 섹션의 첫 실습 앱은 Meals 앱이다.
- Meals 앱에서는 사용자가 음식 카테고리를 선택하고, 해당 카테고리에 속한 음식 목록과 상세 정보를 탐색한다.
- 실습을 위해 카테고리와 음식 더미 데이터, 그리고 데이터 모델 파일을 프로젝트에 추가한다.

## 학습 목표

- React Native 앱에서 말하는 내비게이션의 의미를 명확히 이해한다.
- 웹 라우팅과 모바일 앱 내비게이션의 차이를 구분한다.
- 상태 기반 화면 교체와 실제 내비게이션 흐름의 차이를 복습한다.
- Meals 앱의 화면 흐름을 파악한다.
- 더미 데이터와 모델 파일을 왜 분리해서 준비하는지 이해한다.

## 주요 내용

### React Native에서 내비게이션이란?

React Native 앱에서 내비게이션은 여러 화면 사이를 이동하는 것을 말한다.

이전 섹션의 숫자 맞히기 게임 앱에서도 화면은 여러 개였다.

```text
StartGameScreen
-> GameScreen
-> GameOverScreen
```

하지만 그때는 진짜 내비게이션 시스템을 사용하지 않았다.

상태값에 따라 화면 하나를 다른 화면으로 교체했다.

```text
이전 게임 앱의 화면 전환
-> 현재 상태 확인
-> 조건에 맞는 화면 컴포넌트 렌더링
-> 기존 화면은 사라지고 새 화면이 나타남
```

이 방식은 단순하고 선형적인 게임 흐름에는 충분했다.

하지만 일반적인 앱에서는 더 많은 기능이 필요하다.

```text
일반 앱에서 필요한 내비게이션
-> 특정 화면으로 앞으로 이동
-> 이전 화면으로 돌아가기
-> 화면 전환 애니메이션
-> 탭으로 화면 전환
-> 여러 내비게이션 패턴 조합
```

즉 내비게이션은 단순한 컴포넌트 교체보다 더 큰 개념이다.

### 이전 화면으로 돌아가기

이전 게임 앱에서는 사용자가 게임 화면에서 시작 화면으로 자연스럽게 돌아가는 흐름이 없었다.

게임 오버 화면에서 “새 게임 시작” 버튼을 눌러 상태를 초기화하는 정도였다.

실제 앱에서는 이런 방식만으로는 부족하다.

예를 들어 음식 카테고리 목록에서 특정 카테고리로 들어갔다면, 사용자는 다시 카테고리 목록으로 돌아가고 싶을 수 있다.

```text
카테고리 화면
-> 이탈리안 음식 목록
-> 특정 음식 상세 화면
-> 뒤로가기
-> 이탈리안 음식 목록
-> 뒤로가기
-> 카테고리 화면
```

이런 흐름은 화면 히스토리가 있어야 자연스럽게 구현된다.

Stack Navigation이 필요한 대표적인 이유다.

### 웹 라우팅과의 차이

웹에서는 사용자가 URL을 입력하거나 링크를 클릭해서 페이지를 이동한다.

```text
웹 내비게이션
-> URL 입력
-> 링크 클릭
-> 브라우저 히스토리
-> 뒤로가기 / 앞으로가기
```

React 웹 앱에서는 React Router 같은 도구가 URL과 컴포넌트를 매핑한다.

```jsx
<Route path="/categories" element={<CategoriesPage />} />
<Route path="/categories/:id" element={<MealsPage />} />
```

반면 네이티브 모바일 앱에는 브라우저 주소창이 없다.

사용자는 URL을 보거나 직접 입력하지 않는다.

```text
모바일 앱 내비게이션
-> 버튼 탭
-> 리스트 아이템 탭
-> 탭 메뉴 선택
-> 드로어 메뉴 선택
-> 화면에서 화면으로 이동
```

따라서 React Native에서는 “URL 기반 페이지 이동”보다 “화면 간 이동 구조”를 중심으로 생각해야 한다.

### 모바일 앱의 화면 이동 감각

모바일 앱에서는 사용자가 화면을 탭해서 다음 화면으로 들어가고, 다시 이전 화면으로 돌아오는 흐름이 자연스럽다.

```text
모바일 내비게이션 흐름
-> 카테고리 선택
-> 해당 카테고리의 목록 표시
-> 목록 아이템 선택
-> 상세 화면 표시
-> 뒤로가기
```

이때 사용자는 단순히 화면이 갑자기 바뀌는 것보다, 네이티브 앱다운 애니메이션과 뒤로가기 동작을 기대한다.

이 섹션에서 배우는 내비게이션 패턴들은 이런 모바일 UX를 구현하기 위한 도구다.

### 이번 실습 앱: Meals 앱

첫 번째 데모 앱은 Meals 앱이다.

사용자는 음식 카테고리를 선택하고, 선택한 카테고리에 속한 음식들을 탐색한 뒤, 특정 음식의 상세 정보를 볼 수 있다.

```text
Meals 앱의 기본 흐름
-> 음식 카테고리 목록
-> 선택한 카테고리의 음식 목록
-> 음식 상세 화면
```

예를 들어 다음과 같은 흐름이 가능하다.

```text
Categories
-> Italian
-> Spaghetti with Tomato Sauce
-> Meal Details
```

이 앱은 내비게이션 개념을 배우기에 적합하다.

화면이 여러 단계로 이어지고, 뒤로가기 흐름도 자연스럽게 필요하기 때문이다.

### 제공되는 더미 데이터

강의에서는 카테고리와 음식 데이터를 직접 작성하지 않고 제공된 더미 데이터를 사용한다.

프로젝트에는 `data` 폴더를 만들고, 그 안에 `dummy-data.js` 파일을 추가한다.

```text
data/
  dummy-data.js
```

이 파일에는 음식 카테고리와 음식 목록 데이터가 들어 있다.

```text
dummy-data.js에 포함되는 데이터
-> 음식 카테고리 목록
-> 음식 목록
-> 각 음식이 속한 카테고리 정보
-> 상세 화면에서 사용할 음식 정보
```

강의의 핵심은 데이터 작성이 아니라 내비게이션 학습이므로, 데이터는 더미 형태로 제공된다.

### 모델 파일

더미 데이터와 함께 `models` 폴더도 추가한다.

```text
models/
  category.js
  meal.js
```

`category.js`와 `meal.js`는 데이터 객체의 형태를 정의하는 역할을 한다.

강의에서는 이를 데이터 객체를 만들기 위한 blueprint라고 설명한다.

프론트엔드 개발자 관점에서는 간단한 모델 클래스 또는 도메인 객체 생성자에 가깝게 이해하면 된다.

```text
Category 모델
-> 카테고리 id
-> 카테고리 제목
-> 카테고리 색상 등

Meal 모델
-> 음식 id
-> 음식 제목
-> 이미지
-> 조리 시간
-> 난이도
-> 재료
-> 조리 단계 등
```

이 모델 파일을 기준으로 `dummy-data.js`에서 카테고리와 음식 객체들이 생성된다.

### 폴더 구조

실습을 시작하기 위한 기본 구조는 다음과 같다.

```text
project-root/
  data/
    dummy-data.js
  models/
    category.js
    meal.js
```

이름과 위치가 중요하다.

강의 코드에서 해당 경로를 기준으로 데이터를 가져오기 때문에, 폴더명과 파일명을 맞춰야 한다.

```text
중요한 점
-> data 폴더에 dummy-data.js
-> models 폴더에 category.js, meal.js
-> 강의 코드와 같은 폴더 구조 유지
```

### 첫 번째 목표: 카테고리 출력

데이터 준비가 끝나면 첫 번째 구현 목표는 카테고리 목록을 화면에 출력하는 것이다.

```text
첫 구현 목표
-> 더미 데이터에서 카테고리 목록 가져오기
-> 화면에 카테고리 카드 또는 그리드로 출력하기
-> 이후 카테고리를 탭하면 해당 음식 목록으로 이동하도록 만들기
```

아직 내비게이션을 본격적으로 붙이기 전이라도, 화면에 출력할 데이터가 있어야 다음 단계로 넘어갈 수 있다.

따라서 이 강의는 내비게이션 구현 전 준비 단계라고 볼 수 있다.

## 프론트엔드 개발자 관점에서의 핵심 차이

### URL이 없는 라우팅 감각

웹에서는 URL이 라우팅의 중심이다.

```text
/categories
/categories/c1
/meals/m1
```

하지만 React Native에서는 사용자가 이런 URL을 직접 보지 않는다.

대신 화면 이름과 전달할 데이터를 중심으로 이동을 설계한다.

```text
React Native 내비게이션 사고
-> CategoriesScreen으로 이동
-> MealsOverviewScreen으로 이동하면서 categoryId 전달
-> MealDetailScreen으로 이동하면서 mealId 전달
```

즉 웹의 `params`와 유사한 개념은 있지만, 사용자 경험은 URL이 아니라 네이티브 화면 전환에 가깝다.

### 상태 전환과 라우터 전환의 차이

React 경험이 있으면 상태만으로 화면을 바꾸는 코드를 쉽게 작성할 수 있다.

하지만 앱이 커질수록 화면 전환을 상태 하나로 관리하는 방식은 복잡해진다.

```text
상태 기반 전환
-> 간단한 예제에 적합
-> 화면 수가 적을 때 명확함
-> 히스토리와 애니메이션은 직접 구현해야 함

내비게이션 기반 전환
-> 실제 앱 구조에 적합
-> 뒤로가기와 화면 스택 관리 가능
-> 탭, 드로어, 헤더 등을 통합 관리 가능
```

이번 섹션은 이 차이를 몸으로 익히는 단계다.

### 더미 데이터는 UI/내비게이션 학습을 가볍게 만든다

실제 앱이라면 서버 API, 캐싱, 로딩 상태, 에러 상태까지 고려해야 한다.

하지만 이번 섹션의 목적은 내비게이션이다.

따라서 외부 API 없이 더미 데이터를 사용한다.

```text
더미 데이터를 사용하는 이유
-> 네트워크 로직 없이 화면 이동에 집중
-> 카테고리와 상세 화면 흐름을 빠르게 구성
-> 데이터 구조와 화면 간 파라미터 전달을 연습
```

프론트엔드 실무에서도 새로운 UI 흐름을 설계할 때 더미 데이터로 먼저 화면 구조를 잡는 경우가 많다.

### 모델 파일의 의미

JavaScript 프로젝트에서 모델 클래스를 꼭 써야 하는 것은 아니다.

하지만 강의에서는 `Category`, `Meal` 같은 모델을 둬서 데이터 구조를 명확히 한다.

```text
모델 파일의 장점
-> 데이터 필드 구조를 한곳에서 확인 가능
-> 더미 데이터 생성 방식이 명확해짐
-> 화면에서 어떤 속성을 사용할지 예측 가능
```

TypeScript를 사용하는 프로젝트라면 이 모델은 인터페이스나 타입으로 대체될 수도 있다.

```ts
type Category = {
  id: string;
  title: string;
  color: string;
};
```

이번 강의는 JavaScript 기반이므로 모델 클래스를 통해 비슷한 역할을 수행한다.

## 정리

이번 강의는 본격적인 내비게이션 구현에 들어가기 전, 내비게이션이 무엇인지와 첫 실습 앱의 구조를 소개하는 단계다.

웹에서는 URL과 링크를 중심으로 이동하지만, React Native 앱에서는 사용자의 터치 동작을 통해 화면 스택, 탭, 드로어 같은 구조를 이동한다.

Meals 앱은 카테고리 목록, 음식 목록, 음식 상세 화면으로 이어지는 흐름을 통해 Stack Navigation을 배우기에 좋은 예제다.

다음 단계에서는 제공된 더미 데이터를 사용해 카테고리 목록 화면을 먼저 구축하게 된다.
